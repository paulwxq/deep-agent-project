# 技术设计文档审核检查清单

## 审核准备

审核前**必须**先读取需求原文 /drafts/requirement.md，再读取设计文档 /drafts/design.md，对照审核。

## 审核维度

### 1. 需求覆盖性检查（对照 /drafts/requirement.md）
- [ ] 逐条核对：需求文档中的每一条功能需求，在设计文档中是否有对应的技术方案？
- [ ] 非功能需求（日志、配置、部署、多环境支持等）是否都有设计覆盖？
- [ ] 对于长需求文档，使用 grep 工具在 /drafts/design.md 中搜索需求关键词，辅助确认覆盖情况（避免仅凭 read_file 后的记忆遗漏中间段落）
- [ ] 如发现需求遗漏，明确列出"需求第X条：[需求摘要] → 设计文档中未找到对应方案"

### 2. 可落地性检查
- [ ] Coder 能否仅凭此文档完成开发？
- [ ] 所有模块的入口函数/类是否已定义？
- [ ] 关键算法/流程是否有伪代码或步骤描述？
- [ ] 第三方依赖是否已明确版本？
- [ ] 配置项是否列出了完整的 key-value 结构？

### 3. 无歧义性检查
- [ ] 是否存在"方案A或方案B"式的未决策项？
- [ ] 接口参数类型和约束是否明确？
- [ ] 模块边界和职责是否清晰？
- [ ] 异常处理策略是否确定？

### 4. 完整性检查
- [ ] 接口设计是否包含入参、出参、异常？
- [ ] 数据模型是否包含字段、类型、约束？
- [ ] 模块间交互是否有完整的数据流描述？

### 5. 合理性检查
- [ ] 架构复杂度是否与需求匹配（不过度设计）？
- [ ] 技术选型是否有合理依据？
- [ ] 模块划分是否职责单一？

## 审核原则

1. **尊重 Writer 的技术判断**：如果 Writer 的方案合理可行，即使不是你的首选，也应放行
2. **区分"必须改"和"建议改"**：必须改的问题归入 REVISE 理由，建议改的放入非强制建议
3. **给出具体建议**：不要只说"不够详细"，要说清楚缺什么、怎么改
4. **标准是"能指导代码落地"**：不追求完美，达到标准就放行
