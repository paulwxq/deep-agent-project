# 技术设计文档撰写指南

## 标准文档结构

一份合格的技术设计文档应包含以下章节：

### 1. 概述
- 项目背景与目标
- 核心功能列表
- 术语定义

### 2. 技术方案
- 整体架构设计（附架构图描述）
- 技术选型及理由（给出明确建议，不留选择题）
- 核心流程设计（附流程图描述）

### 3. 模块设计
- 各模块职责与边界
- 模块间交互关系
- 关键类/函数设计

### 4. 数据设计
- 数据模型定义（字段、类型、约束）
- 数据流转描述

### 5. 接口设计
- 接口签名（函数名、入参、出参）
- 异常/错误处理
- 示例调用

### 6. 非功能性设计
- 日志与监控
- 性能考量
- 安全性

### 7. 部署与运维
- 环境要求
- 配置说明
- 启动/停止流程

## 写作原则

1. **明确性优先**：每个技术决策必须给出确定的建议，附带简要理由
2. **面向 Coder**：读者是开发工程师，文档应能直接指导编码
3. **可验证性**：每个功能点应能通过文档推导出对应的测试用例
4. **无歧义**：避免"可以考虑"、"建议选择A或B"等模糊表述

## 适用场景

本技能主要服务**初稿撰写**。根据 Reviewer 反馈执行局部修订时，应优先使用 `tech-doc-revision` 技能，而不是本技能。

## 修订场景边界

当任务是根据 Reviewer 反馈修订已有文档时，具体修订流程（需求基线读取、是否采纳判断、章节定位、`edit_file` 策略、自检、修订记录）统一由 `tech-doc-revision` skill 承载，本技能不定义独立修订流程。

本技能在修订场景下仅保留以下通用约束：

### 单文件规则

- 唯一正式草稿文件是 `/drafts/design.md`
- 不要创建 `design_v2.md`、`design_v3.md`、`design_final.md` 等变体文件
- 如果工作区中已经存在 `design_v*.md`，视为旁支文件，不要读取、不要继续写入

### 目录访问边界

- 允许读取：`/input/` 下与当前任务直接相关的文件、`/drafts/design.md`、`/drafts/qa-supplement.md`
- 禁止浏览：`/drafts/_backups`、`/output`、与当前任务无关的历史目录
