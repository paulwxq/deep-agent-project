# 技术设计文档局部修订指南

## 核心职责

1. **基线建立**：必须读取 Reviewer 反馈、`/drafts/design.md`、主需求文件及 `/drafts/qa-supplement.md`（若存在）。
2. **采纳决策**：核对 Reviewer 意见是否与基线一致；若不一致，应给出"不采纳"理由并保持基线权威。
3. **定位与修订**：将反馈映射到具体章节，优先使用 `edit_file`。
4. **一致性检查**：修订后必须检查是否破坏了与其他章节的关联。

## 修订执行流程

### 1. 利用 Reviewer 的"行动建议"
- 必须遵循 Reviewer 提供的 `## X -> ### Y` 锚点格式，将其作为 `edit_file` 定位的首选依据。
- 若 Reviewer 提供了代码/文本片段建议，应在符合设计逻辑的前提下优先采用。

### 2. 日期规范 (反幻觉)
- 修订记录中的日期**必须使用运行时的实际日期**。
- **严禁抄写示例中的静态日期**。如不确定当前日期，应从系统提示词、文件修改时间或 Orchestrator 传入的任务信息中推断。

### 3. `edit_file` (Replace) 稳定性增强策略
- **定位增强**：`old_string` 必须包含目标章节的**标题行**及下方 **1-3 行正文**。
- **匹配失败恢复**：
  - **初次失败**：重新 `read_file` 目标章节，检查是否存在不可见字符或换行符，调整 `old_string`。
  - **二次失败**：缩小 `old_string` 范围，仅包含最关键的几行。
  - **持续失败**：考虑使用 `write_file` 覆盖该章节所属的较大片段，或在极端情况下重写全文。

### 4. 跨章节一致性自检
- **术语同步**：修改了共享变量、模块名或算法名后，用 `grep` 搜索全文，同步更新引用。
- **数据一致性**：修改数据表字段后，必须检查对应的接口入参/出参章节是否需要同步修订。

## 修订记录规范

在 `/drafts/design.md` 末尾更新记录：
- **首次写入**：若文档末尾尚无 `## 修订记录` 章节，使用 `write_file` 重新写出文档并追加该章节及首条记录。
- **追加记录**：使用 `edit_file` 在末尾追加。
- **格式**：`日期 | 锚点 (## X -> ### Y) | 修改要点 | 状态 (已采纳/未采纳: 理由)`。

## 目录与边界

- **唯一草稿**：`/drafts/design.md`。
- **禁止访问**：`/drafts/_backups`、`/output`。
