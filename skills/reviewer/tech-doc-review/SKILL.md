# 技术设计文档审核检查清单

## 审核准备

1. **动态确认文件名**：先执行 `ls /input/` 确认实际的需求文件名（处理空格和中文字符），禁止盲目使用占位符 `<requirement_filename>`。
2. **基线对齐**：读取需求原文、`/drafts/design.md` 以及 `/drafts/qa-supplement.md`（若存在）。读取 `/drafts/design.md` 时，若返回行数等于请求行数（说明可能有后续内容），以 offset 递增继续读取，直到返回行数少于请求行数或返回为空为止。
3. **后续轮次高效策略**（当 Orchestrator 任务描述指明”后续轮次”或”第N轮审核”时，**必须**遵守）：
   - 若 Orchestrator 已列明上轮问题，**优先验证这些问题的修复情况**，聚焦于修订后的章节。
   - **禁止重读 /input/ 参考文件**：不要重新执行 `ls /input/` 或读取 `/input/` 下的源代码、数据文件等，除非某条修改项确实需要事实核对——此时仅读取与该项直接相关的单个文件。
   - **非回归检查**：核实文档末尾的”修订记录”，确保 Writer 没把之前对的部分改错。

## 审核原则

1. **标准是"能指导代码落地"**：不追求文档的文学美感，只要达到"需求全覆盖、技术路径清晰、无重大歧义"即可放行。
2. **区分"必须改" (REVISE) 与 "建议改" (OPTIONAL)**：
   - **REVISE**：遗漏需求、逻辑错误、API 不兼容、严重歧义、导致无法编码的缺失。
   - **OPTIONAL**：命名习惯、文档排版、次要重构建议。
3. **Actionable Feedback**：必须提供具体的路径式锚点（如 `## X -> ### Y`）和具体的修订动作建议。
4. **JSON 状态持久化**：
   - verdict 文件路径取决于你的身份：
     - **reviewer1**：写入 `/drafts/review-verdict.json`
     - **reviewer2**：写入 `/drafts/review-verdict-stage2.json`
   - verdict 文件是一个完整的 JSON 状态对象，优先使用 `write_file` **全量覆盖写入**。
   - reviewer1 与 reviewer2 已使用不同路径，不再需要通过 `edit_file` 规避路径冲突。

## 章节化反馈要求 (REVISE)

当结论为 `REVISE` 时，每条必须修改意见**必须**包含：

- **修订锚点**：精确到路径式标题，如 `## 5 模块设计 -> ### 5.2 数据解析流`（必须与 Revision Skill 格式严格统一）。
- **问题描述**：指出具体缺陷（缺什么、哪里错）。
- **修订动作**：明确告诉 Writer 应该：**补充**（Add）、**替换**（Replace）还是**删除**（Delete）。
- **建议内容（推荐）**：提供一段示例文本或伪代码片段，方便 Writer 直接采用。
- **参考依据**：对应需求第 X 条，或参考文件 Y。

## 审核边界

- 唯一正式审核对象是 `/drafts/design.md`。
- 禁止审核 `design_v*.md` 等临时文件。
- 禁止浏览 `/drafts/_backups` 或 `/output` 下的历史版本。
