# SAS数据血缘分析系统技术设计文档

## 1. 项目概述和目标

### 1.1 项目背景
SAS（Statistical Analysis System）在金融、保险等行业广泛用于数据分析和处理。随着数据治理和合规要求的提升，理解数据从源头到最终输出的完整血缘关系变得至关重要。当前SAS代码通常处理来自mainframe的数据集，并生成CSV格式的输出文件，中间可能涉及多个临时SAS数据集（.sas7bdat文件）。

### 1.2 项目目标
开发一个Python应用模块，能够：
- 自动解析SAS代码，识别数据输入源、输出目标和字段转换逻辑
- 分析数据血缘关系，建立从输出表字段到源表字段的完整映射
- 生成标准化的数据血缘对照表，包含：源表名/文件名、源表字段、目标表名/文件名、目标表字段
- 支持复杂的SAS代码模式，包括字段重命名、派生字段计算、多表合并等场景
- 处理大规模SAS代码库，通过智能分片和LLM集成实现高效分析

### 1.3 关键需求
- **输入支持**：mainframe定义文件（.sas）、CSV文件、Excel文件、SAS临时文件（.sas7bdat）
- **输出格式**：标准化的血缘对照表（CSV/JSON格式）
- **分析入口**：从输出表开始反向追踪到源表字段
- **可扩展性**：支持不同LLM Provider配置，适应不同规模的代码分析需求

## 3. 核心模块设计

### 3.1 SAS代码解析器模块

#### 3.1.1 功能职责
- 解析SAS代码文件，识别关键语法结构
- 提取数据集定义（DATA语句）
- 识别输入数据源（SET、MERGE、PROC IMPORT等）
- 提取字段操作（RENAME、KEEP、派生字段计算等）
- 识别输出目标（PROC EXPORT、DATA语句输出）

#### 3.1.2 关键解析规则

##### 派生字段识别正则表达式
- **基础赋值模式**：`r'^\s*(\w+)\s*=\s*(.+)\s*;'`
  - 匹配字段名（左侧）和计算表达式（右侧）
  - 支持空格和分号的灵活处理
- **复杂表达式处理**：
  - 函数调用模式：`r'\b([A-Z_]+)\s*\([^)]*\)'`（匹配SUM()、MEAN()等函数）
  - 条件表达式模式：`r'\b(IF|THEN|ELSE)\b'`（识别IF-THEN-ELSE结构）

##### 复杂派生字段处理策略
- **函数调用处理**：
  - 识别常见SAS函数：SUM、MEAN、MAX、MIN、SUBSTR等
  - 提取函数参数中的字段引用
  - 构建多源字段依赖关系
- **条件表达式处理**：
  - 解析IF-THEN-ELSE结构
  - 识别条件中的字段引用
  - 识别THEN/ELSE分支中的字段引用
  - 构建完整的字段依赖图谱

##### 解析示例
基于参考文件中的实际派生字段：

**示例1：简单算术运算（来自01_churn_transform.sas）**
```sas
bal_per_prod = Balance / num_prod;
```
- 正则匹配：`field_name="bal_per_prod"`, `expression="Balance / num_prod"`
- 字段依赖：`bal_per_prod` ← [`Balance`, `num_prod`]

**示例2：条件表达式（来自01_churn_transform.sas）**
```sas
senior_flag = (Age >= 60);
```
- 正则匹配：`field_name="senior_flag"`, `expression="(Age >= 60)"`
- 字段依赖：`senior_flag` ← [`Age`]

**示例3：跨源复杂派生（来自03_merge_transform.sas）**
```sas
risk_score = credit_scr - (dti * 10);
```
- 正则匹配：`field_name="risk_score"`, `expression="credit_scr - (dti * 10)"`
- 字段依赖：`risk_score` ← [`credit_scr`, `dti`]
- 源表追踪：`credit_scr`来自`churn_modeling_1.csv`，`dti`来自`train_1.csv`

**示例4：复杂算术运算（来自03_merge_transform.sas）**
```sas
age_loan_ratio = (Age / loan_amt) * 1000;
```
- 正则匹配：`field_name="age_loan_ratio"`, `expression="(Age / loan_amt) * 1000"`
- 字段依赖：`age_loan_ratio` ← [`Age`, `loan_amt`]

### 3.2 数据血缘分析引擎

#### 3.2.1 功能职责
- 构建完整的数据血缘图谱
- 支持从输出表反向追踪到源表
- 处理多级依赖关系（中间临时表）
- 合并来自不同SAS脚本的血缘信息
- 生成标准化的血缘对照表

#### 3.2.2 核心算法
- **图遍历算法**：使用深度优先搜索（DFS）进行反向血缘追踪
- **依赖解析**：按执行顺序解析SAS脚本间的依赖关系
- **字段传播**：跟踪字段在各个处理步骤中的变化

#### 3.2.3 血缘图谱数据结构
```python
class LineageGraph:
    nodes: Dict[str, DatasetNode]  # dataset_name -> node
    edges: List[LineageEdge]

class DatasetNode:
    name: str
    type: Literal["source", "intermediate", "output"]
    fields: Dict[str, FieldNode]

class FieldNode:
    name: str
    data_type: Optional[str]  # 存储字段类型信息（numeric/character等）
    source_fields: List[FieldReference]  # 反向引用

class LineageEdge:
    source_dataset: str
    source_field: str
    target_dataset: str
    target_field: str
    transformation: str  # "passthrough", "rename", "derived"
    formula: Optional[str]
    source_data_type: Optional[str]  # 源字段类型
    target_data_type: Optional[str]  # 目标字段类型

# 字段类型转换处理：
# - 当源字段和目标字段类型不同时，在LineageEdge中记录类型转换信息
# - 支持的类型转换：numeric ↔ character（通过PUT/INPUT函数）
# - 不支持的类型转换会被标记为警告
```

### 3.3 LLM集成模块

#### 3.3.1 功能职责
- 处理复杂SAS语法和业务逻辑的语义理解
- 辅助解析难以通过规则处理的派生字段逻辑
- 提供代码分片和上下文管理
- 支持多种LLM Provider配置

#### 3.3.2 Deep Agents架构

使用Deep Agents SDK构建多Agent协作系统，各Agent职责明确且协作机制清晰：

##### Agent通信机制
- **通信方式**：采用消息队列模式，使用Python的`queue.Queue`实现线程安全的消息传递
- **消息格式**：标准化JSON消息格式，包含`message_type`、`source_agent`、`target_agent`、`payload`字段
- **共享状态**：通过`SharedMemoryStore`类维护全局状态，所有Agent可读写

##### 任务分配策略
- **Coordinator Agent职责**：
  - 接收原始SAS代码输入
  - 根据代码复杂度和类型进行任务分解
  - 将任务分配给合适的Agent
  - 汇总各Agent的结果并生成最终输出
- **任务路由规则**：
  - 基础语法解析 → Parser Agent
  - 业务语义理解 → Semantic Agent  
  - 血缘关系构建 → Lineage Agent
  - 复杂派生字段分析 → Semantic Agent + Lineage Agent 协作

##### 冲突解决机制
- **冲突检测**：当多个Agent对同一字段或关系给出不同结论时，触发冲突检测
- **解决策略**：
  - 优先级机制：Semantic Agent > Parser Agent（语义理解优先于语法解析）
  - 投票机制：Lineage Agent和Semantic Agent对血缘关系有同等权重，采用多数决
  - 人工干预：冲突无法自动解决时，记录到冲突日志并标记需要人工审核

##### 完整工作流程示例
以处理`01_churn_transform.sas`为例：

1. **Coordinator Agent接收输入**：读取`01_churn_transform.sas`文件
2. **任务分解**：
   - 识别DATA步 → 分配给Parser Agent
   - 识别派生字段`bal_per_prod = Balance / num_prod` → 分配给Semantic Agent
   - 识别RENAME操作 → 分配给Parser Agent
3. **Agent并行处理**：
   - Parser Agent解析DATA步结构，识别输入`churn_raw`，输出`churn_out`
   - Semantic Agent分析派生字段语义，确认`bal_per_prod`依赖`Balance`和`num_prod`
   - Lineage Agent构建血缘关系：`churn_out.bal_per_prod` ← `churn_raw.Balance` + `churn_raw.num_prod`
4. **结果汇总**：Coordinator Agent合并各Agent结果，生成完整的血缘图谱
5. **冲突检查**：验证各Agent结果一致性，无冲突则输出最终结果

#### 3.3.3 上下文管理策略

针对需求第5条"实际场景中的SAS代码和文件会比较多，你不太可能把它们读取后一口气全部提交给LLM"，设计以下具体实现策略：

##### 分片策略
- **逻辑边界识别**：按SAS语法结构进行分片，主要边界包括：
  - DATA步（每个DATA步作为一个独立分片）
  - PROC步（每个PROC步作为一个独立分片）
  - 宏定义（%MACRO...%MEND作为一个分片）
- **分片算法**：
  - 基于行数限制：单个分片不超过200行代码
  - 基于语法完整性：确保分片内的代码语法完整（如完整的DATA步、完整的赋值语句）
  - 优先级：语法完整性 > 行数限制
- **分片大小限制**：每个分片的token数量控制在LLM上下文窗口的70%以内（例如GPT-4的8K token上下文，分片控制在5.6K token以内）

##### 上下文维护
- **跨分片上下文数据结构**：使用`GlobalContext`类维护全局状态
  ```python
  class GlobalContext:
      datasets: Dict[str, DatasetMetadata]  # 已识别的数据集元信息
      field_mappings: Dict[str, Dict[str, str]]  # 字段映射关系缓存
      processed_scripts: Set[str]  # 已处理的脚本路径
      lineage_fragments: List[LineageFragment]  # 血缘片段缓存
  ```
- **上下文传递**：每个分片处理时，将`GlobalContext`作为系统提示的一部分传递给LLM

##### 缓存机制
- **缓存内容**：
  - SAS代码解析结果（AST结构）
  - LLM响应结果（字段映射、血缘关系）
  - 字段类型推断结果
- **缓存策略**：
  - 使用LRU（最近最少使用）策略，缓存大小限制为1000个条目
  - TTL（生存时间）设置为1小时，避免内存泄漏
- **缓存失效条件**：
  - 相关SAS文件发生修改（通过文件mtime检测）
  - 手动清除缓存（通过CLI参数`--clear-cache`）
  - 内存使用超过阈值（自动触发LRU淘汰）

##### 分片示例
以包含多个DATA步的SAS脚本为例：
```sas
/* 脚本: multi_step.sas */
DATA step1;
    SET source1;
    new_field1 = old_field1 * 2;
RUN;

DATA step2;
    SET step1;
    new_field2 = new_field1 + 10;
RUN;
```

分片处理过程：
1. **分片1**：处理`DATA step1`部分，识别输入`source1`，输出`step1`，派生字段`new_field1`
2. **更新上下文**：将`step1`数据集信息存入`GlobalContext.datasets`
3. **分片2**：处理`DATA step2`部分，利用上下文知道`step1`存在，识别输入`step1`，输出`step2`，派生字段`new_field2`
4. **合并结果**：将两个分片的血缘关系合并为完整血缘图谱

### 3.4 文件处理器模块

#### 3.4.1 功能职责
- 读取和解析各种输入文件格式
- 提取源数据的字段元信息
- 处理mainframe定义文件的特殊格式
- 支持SAS临时文件（.sas7bdat）的读取

#### 3.4.2 文件格式支持

##### Mainframe定义文件解析
基于参考文件`def_Churn.sas`和`def_train.sas`的实际格式，设计以下解析策略：

###### Mainframe格式说明符解析
- **常见格式说明符及含义**：
  - `PKn.m`：Packed Decimal格式，n为总位数，m为小数位数（如`PK2.0`表示2位整数）
  - `S370FPDn.m`：IBM S/370 Floating Point Decimal，n为总位数，m为小数位数
  - `$EBCDICn.`：EBCDIC字符格式，n为字符长度（如`$EBCDIC3.`表示3个字符）
- **格式解析规则**：
  - 数值格式（PK、S370FPD）→ 字段类型为`numeric`
  - 字符格式（$EBCDIC）→ 字段类型为`character`

###### 位置指示符解析
- **位置指示符格式**：`@position`（如`@1`、`@3`、`@19`）
- **解析算法**：
  1. 按行扫描INPUT语句
  2. 提取位置指示符（正则表达式：`@(\d+)`）
  3. 提取字段名称（处理带引号的字段名，如`'Loan Amount'`）
  4. 提取格式说明符（最后一个非空白token）
  5. 构建字段元信息

###### 解析算法伪代码
```python
def parse_mainframe_definition(sas_code: str) -> List[FieldMetadata]:
    fields = []
    input_block = extract_input_block(sas_code)  # 提取INPUT...;之间的内容
    
    for line in input_block.split('\n'):
        line = line.strip()
        if not line or line.startswith('*'):
            continue
            
        # 匹配位置指示符、字段名、格式说明符
        pattern = r'@(\d+)\s+([\'"]?)([^\'"\s]+)\2?\s+([A-Z\$][A-Z0-9\.]+)'
        match = re.search(pattern, line)
        
        if match:
            position = int(match.group(1))
            field_name = match.group(3)
            format_spec = match.group(4)
            
            # 确定字段类型
            if format_spec.startswith('$'):
                data_type = 'character'
            else:
                data_type = 'numeric'
                
            fields.append(FieldMetadata(
                name=field_name,
                position=position,
                format=format_spec,
                data_type=data_type
            ))
    
    return fields
```

###### 解析示例
以`def_Churn.sas`为例：
```sas
input
    @1    CustomerId                       PK2.0
    @3    Surname                           PK2.0
    @5    CreditScore                             PK3.0
    @8    Geography                              S370FPD2.0
    @10   Gender                       $EBCDIC3.
```

解析结果：

- `Surname`: position=3, format=PK2.0, type=numeric  
- `CreditScore`: position=5, format=PK3.0, type=numeric

## 4. 数据流程设计

### 4.1 整体数据流
```
1. 输入发现 → 2. SAS代码解析 → 3. 源数据元信息提取 → 4. 血缘图谱构建 → 5. LLM辅助分析 → 6. 血缘对照表生成
```

### 4.2 详细处理流程

#### 阶段1：输入发现和预处理
- 扫描指定目录，发现所有SAS代码文件和相关数据文件
- 建立文件依赖关系图，确定处理顺序
- 预处理mainframe定义文件，提取字段元信息

#### 阶段2：SAS代码解析
- 逐个解析SAS代码文件，提取语法结构
- 识别输入/输出数据集和字段操作
- 生成初步的字段映射关系

#### 阶段3：血缘图谱构建
- 将所有SAS脚本的解析结果整合到统一的血缘图谱中
- 处理跨脚本的数据依赖关系
- 验证血缘关系的完整性和一致性

#### 阶段4：LLM辅助分析
- **复杂派生字段判断标准**：以下情况需要LLM辅助分析：
  - 包含嵌套函数调用的派生字段（如`SUM(IF(condition, value1, value2))`）
  - 涉及业务领域知识的字段转换（如金融风险评分公式）
  - 条件表达式中包含复杂逻辑的派生字段
  - 跨多个源表的复杂聚合计算
- 对复杂派生字段和业务逻辑进行语义分析
- 处理规则引擎无法解析的特殊情况
- 验证和修正血缘关系

#### 阶段5：输出生成
- 从指定的输出表开始，反向追踪血缘关系
- 生成标准化的血缘对照表
- 支持多种输出格式（CSV、JSON、数据库）

### 4.4 错误处理和异常情况

为确保系统在实际运行中的健壮性，设计以下错误处理机制：

##### SAS代码语法错误
- **检测机制**：在SAS代码解析阶段进行语法验证
- **处理策略**：
  - 记录错误文件路径、行号、错误类型到错误日志
  - 跳过错误文件，继续处理其他文件
  - 在最终报告中汇总所有语法错误
- **错误信息格式**：`[ERROR] File: {file_path}, Line: {line_number}, Error: {error_message}`

##### 源文件缺失
- **检测场景**：
  - Mainframe定义文件不存在
  - CSV/Excel源文件不存在
  - 引用的中间SAS数据集不存在
- **处理策略**：
  - 尝试从其他位置查找文件（如配置的备用路径）
  - 如果文件确实缺失，标记相关字段为"UNKNOWN_SOURCE"
  - 继续处理其他可解析的部分
  - 在血缘结果中标记缺失源的字段

##### LLM解析失败
- **失败场景**：
  - LLM API调用超时或失败
  - LLM返回无效或不完整的解析结果
  - LLM无法理解复杂业务逻辑
- **处理策略**：
  - **重试机制**：最多重试3次，每次增加提示词详细度
  - **降级策略**：如果LLM持续失败，回退到规则引擎处理
  - **人工标记**：将无法解析的代码片段标记为需要人工审核
  - **缓存失败结果**：避免重复调用LLM处理相同失败片段

##### 日志记录规范
- **日志级别**：
  - `INFO`：处理进度、成功解析的文件
  - `WARNING`：源文件缺失、字段类型推断不确定
  - `ERROR`：语法错误、LLM调用失败、系统异常
- **日志内容**：
  - 时间戳、模块名称、文件路径、处理状态
  - 性能指标（处理时间、内存使用）
  - 错误堆栈（仅ERROR级别）
- **日志输出**：
  - 控制台输出（支持--verbose参数控制详细程度）
  - 文件日志（默认保存到./logs/sas_lineage.log）

### 4.3 反向血缘追踪算法
```python
def trace_lineage_backward(output_table: str, output_field: str) -> List[LineageRecord]:
    """
    从输出表字段反向追踪到源表字段
    """
    lineage_records = []
    current_field = output_field
    current_table = output_table
    
    while current_table is not a source table:
        # 查找当前字段的来源
        field_node = lineage_graph.get_field_node(current_table, current_field)
        
        if field_node.transformation == "passthrough":
            # 直接传递，继续向上追踪
            source_field = field_node.source_fields[0]
            current_field = source_field.field_name
            current_table = source_field.dataset_name
            
        elif field_node.transformation == "rename":
            # 字段重命名，记录映射关系
            source_field = field_node.source_fields[0]
            lineage_records.append(LineageRecord(
                output_table=current_table,
                output_field=current_field,
                source_table=source_field.dataset_name,
                source_field=source_field.field_name,
                transformation="rename"
            ))
            current_field = source_field.field_name
            current_table = source_field.dataset_name
            
        elif field_node.transformation == "derived":
            # 派生字段，记录完整公式和所有源字段
            for source_field in field_node.source_fields:
                lineage_records.append(LineageRecord(
                    output_table=current_table,
                    output_field=current_field,
                    source_table=source_field.dataset_name,
                    source_field=source_field.field_name,
                    transformation="derived",
                    formula=field_node.formula
                ))
                # 递归追踪每个源字段
                sub_lineage = trace_lineage_backward(
                    source_field.dataset_name, 
                    source_field.field_name
                )
                lineage_records.extend(sub_lineage)
            break  # 派生字段处理完成，不需要继续单一路径追踪
    
    return lineage_records
```
