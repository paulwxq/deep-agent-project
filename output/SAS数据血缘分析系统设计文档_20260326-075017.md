# SAS数据血缘分析Python应用模块技术设计文档

## 1. 概述

### 1.1 项目背景
本项目旨在开发一个独立的命令行工具，用于分析SAS代码和相关数据集，生成数据血缘对照表。该工具将帮助数据工程师和分析师理解数据从源系统到最终输出的完整流转路径，特别是在复杂的SAS数据处理环境中。

### 1.2 核心需求
- 分析SAS代码（`.sas`文件）和数据集定义文件，识别数据处理逻辑
- 从输出表作为分析入口，反向追踪到源表字段
- 生成包含源表名/文件名、源表字段、目标表名/文件名、目标表字段、转换公式、脚本文件的数据血缘对照表
- 输出格式为CSV文件
- 支持处理几十个SAS文件，每个文件大小在几十MB级别
- 以定义文件（`def_*.sas`）中的字段名为准，即使代码中使用了不同的拼写

### 1.3 技术栈
- Python 3.12
- 可配置的LLM Provider（支持OpenAI、Anthropic等）
- langchain-ai / deep agents SDK
- pandas（用于CSV处理）
- sas7bdat（用于SAS数据集读取）

## 2. 整体架构

### 2.2 模块划分

#### 2.2.1 文件解析器模块（File Parser）
- **功能**：读取和解析各种SAS相关文件
- **输入**：SAS代码文件（`.sas`）、定义文件（`def_*.sas`）、CSV文件、SAS数据集（`.sas7bdat`）
- **输出**：结构化的文件元数据和内容信息

#### 2.2.2 LLM分析器模块（LLM Analyzer）
- **功能**：利用大语言模型智能分析SAS代码逻辑
- **输入**：分块后的SAS代码片段、上下文信息
- **输出**：代码语义分析结果、数据操作识别

#### 2.2.3 血缘构建器模块（Lineage Builder）
- **功能**：基于分析结果构建完整的数据血缘图谱
- **输入**：文件解析结果、LLM分析结果
- **输出**：数据血缘关系图、字段映射关系

#### 2.2.4 输出生成器模块（Output Generator）
- **功能**：将血缘关系转换为指定格式的CSV文件
- **输入**：血缘关系数据
- **输出**：CSV格式的数据血缘对照表

#### 2.2.5 主协调器模块（Main Orchestrator）
- **功能**：协调各模块执行、管理配置、处理错误
- **输入**：命令行参数、配置文件
- **输出**：最终结果文件、日志信息

## 3. 核心算法设计

### 3.1 SAS代码解析算法

#### 3.1.1 文件类型识别
- **定义文件识别**：通过文件名模式 `def_*.sas` 识别数据集定义文件
- **处理脚本识别**：通过文件内容分析识别数据处理脚本
- **输入/输出识别**：通过 `PROC IMPORT`、`PROC EXPORT`、`DATA` 语句识别数据流向

#### 3.1.2 字段定义提取
从定义文件中提取字段信息：
```python
# 伪代码示例
def extract_fields_from_def_file(file_content):
    fields = []
    lines = file_content.split('\n')
    in_input_section = False
    
    for line in lines:
        if 'input' in line.lower() and '@' in line:
            in_input_section = True
            continue
        if in_input_section and ';' in line:
            break
        if in_input_section and '@' in line:
            # 解析字段定义行，如: @1 CustomerId PK2.0
            field_info = parse_field_definition(line)
            fields.append(field_info)
    
    return fields
```

#### 3.1.3 数据操作识别
识别关键SAS语句：
- `DATA` 语句：创建新数据集
- `SET` 语句：读取现有数据集
- `MERGE` 语句：合并多个数据集
- `RENAME` 选项：字段重命名
- 赋值语句：字段计算和转换
- `KEEP`/`DROP` 语句：字段选择

### 3.2 数据血缘分析算法

#### 3.2.1 正向血缘构建
1. **初始化**：为每个数据集创建节点
2. **处理顺序**：按文件依赖关系排序处理
3. **字段映射**：
   - 直接映射：`new_field = old_field`
   - 重命名映射：通过 `RENAME` 选项建立映射
   - 计算字段：记录计算公式和依赖字段

#### 3.2.2 反向血缘追踪
1. **入口点**：从最终输出表（CSV文件）开始
2. **递归追踪**：对每个目标字段，递归查找其源字段
3. **路径记录**：记录完整的血缘路径，包括中间步骤

**中间文件识别逻辑**：
- **DATA语句识别**：`DATA dataset_name;` 创建新的中间数据集
- **SET语句识别**：`SET source_dataset;` 读取现有数据集作为输入
- **临时文件标记**：以`.sas7bdat`为后缀的文件或在SAS代码中创建的数据集视为中间文件

**中间文件处理流程**：
1. **依赖分析**：构建文件依赖图，识别所有中间数据集
2. **优先级排序**：按数据流方向从源到目标排序处理
3. **分阶段分析**：
   - 第一阶段：分析所有中间文件的输入输出关系
   - 第二阶段：基于中间文件分析结果，建立端到端的完整血缘链路

**跨文件血缘构建**：
- **全局数据集注册表**：维护所有数据集的输入输出映射
- **链路拼接**：当发现数据集A输出到数据集B，而B又作为数据集C的输入时，自动拼接A→B→C的完整链路
- **循环检测**：防止数据流循环导致无限递归

**示例**：以03_merge_transform.sas为例：
- churn_raw（中间文件）由01_churn_transform.sas生成，源为churn_modeling_1.csv
- churn_prep由03_merge_transform.sas生成，输入为churn_raw
- merged_out由03_merge_transform.sas生成，输入为churn_prep和loan_prep
- 完整链路：churn_modeling_1.csv → churn_raw → churn_prep → merged_out
- 系统会自动追踪Age字段从churn_modeling_1.csv经过churn_raw、churn_prep最终到merged_out的完整路径

#### 3.2.3 字段名标准化处理
根据需求澄清Q1，以定义文件（`def_*.sas`）中的字段名为准，即使代码中使用了不同的拼写。

**字段名冲突检测算法**：
1. 扫描所有SAS代码文件，提取所有字段引用（包括SET、MERGE、KEEP、DROP、RENAME等语句中的字段）
2. 从定义文件（`def_*.sas`）中提取标准字段定义列表
3. 对比代码中的字段引用与标准字段列表，识别不匹配的字段名
4. 使用编辑距离算法（Levenshtein distance）计算相似度，阈值设为2（允许最多2个字符差异）

**映射表建立方式**：
- **自动匹配规则**：基于编辑距离和模糊匹配（如Soundex算法）自动建立映射
- **配置文件覆盖**：支持通过`field_mapping.json`配置文件手动指定映射关系，格式如下：
```json
{
  "churn_raw": {
    "EstimatedSalary": "EstimagteSalary",
    "Credit_Score": "CreditScore"
  }
}
```

**映射表数据结构**：
- **存储格式**：Python字典，键为数据集名，值为字段映射字典
- **加载方式**：启动时从定义文件自动构建，如有配置文件则合并覆盖
- **查询接口**：`FieldMapper.get_standard_name(dataset_name, field_name) -> str`

**应用时机**：
- 在LLM分析前预处理阶段：将代码中的字段引用替换为标准字段名
- 在血缘关系构建阶段：确保源字段和目标字段都使用标准命名
- 在输出生成阶段：CSV中的字段名统一使用定义文件中的标准拼写

**示例**：def_Churn.sas中定义字段为"EstimagteSalary"，而代码中引用"EstimatedSalary"，系统会自动建立映射并在最终输出中使用"EstimagteSalary"作为标准字段名。

### 3.3 字段映射追踪算法

#### 3.3.1 映射关系建立
```python
class FieldMapping:
    def __init__(self):
        self.mappings = {}  # {target_field: source_field_info}
        self.transformations = {}  # {field: transformation_formula}
    
    def add_direct_mapping(self, source_dataset, source_field, target_dataset, target_field):
        self.mappings[target_field] = {
            'source_dataset': source_dataset,
            'source_field': source_field,
            'transformation': 'direct_copy'
        }
    
    def add_computed_mapping(self, dependencies, target_dataset, target_field, formula):
        self.mappings[target_field] = {
            'dependencies': dependencies,  # list of source fields
            'target_dataset': target_dataset,
            'formula': formula
        }
```

#### 3.3.2 多源合并处理
对于 `MERGE` 操作：
- 识别合并键（`BY` 语句）
- 分别追踪来自不同源数据集的字段
- 处理条件合并（`IN=a AND b`）

## 4. 分块处理策略

### 4.1 分块原则
考虑到LLM上下文限制和文件规模（几十个文件，每个几十MB），采用以下分块策略：

#### 4.1.1 文件级分块
- **独立处理**：每个SAS文件独立分析
- **依赖排序**：按数据流依赖关系确定处理顺序
- **并行处理**：无依赖关系的文件可并行处理

#### 4.1.2 代码级分块
对于大型SAS文件：
- **按逻辑块分割**：以 `DATA`、`PROC` 语句为边界
- **上下文保留**：每个块保留必要的上下文信息
- **增量分析**：逐步构建完整的血缘关系

**必要的上下文信息定义**：
- **宏变量定义**：所有`%LET`语句定义的宏变量，用于解析动态文件路径和参数
- **库引用**：`LIBNAME`语句定义的库引用，用于解析数据集的完整路径
- **当前DATA步的输入输出数据集**：当前处理的DATA步骤的源数据集（SET/MERGE）和目标数据集（DATA）
- **字段重命名映射**：当前DATA步中的RENAME选项定义的字段映射关系
- **全局变量作用域**：在当前代码块之前定义的全局变量和参数

**上下文提取算法**：
1. **预扫描阶段**：在分块前扫描整个文件，提取全局上下文（宏变量、库引用）
2. **局部上下文提取**：对每个代码块，提取该块特有的上下文信息
3. **依赖分析**：识别代码块之间的上下文依赖关系

**上下文传递方式**：
- **上下文对象**：每个代码块分析时传入一个Context对象，包含全局和局部上下文
- **增量更新**：处理完一个代码块后，更新全局上下文状态（如新创建的数据集）
- **作用域管理**：维护变量作用域栈，确保上下文的正确性

### 4.2 上下文管理

#### 4.2.1 全局上下文
- 宏变量定义（`%LET` 语句）
- 库引用（`LIBNAME` 语句）
- 公共字段定义

#### 4.2.2 局部上下文
- 当前 `DATA` 步骤的变量作用域
- 临时变量和计算字段
- 条件逻辑上下文

### 4.3 缓存机制
- **解析结果缓存**：避免重复解析相同文件
- **LLM响应缓存**：缓存已分析的代码片段结果
- **血缘关系缓存**：缓存已建立的映射关系

## 5. 错误处理与容错机制

### 5.1 错误类型分类

#### 5.1.1 解析错误
- **语法错误**：SAS代码语法不规范
- **文件格式错误**：无法识别的文件格式
- **编码错误**：文件编码问题

#### 5.1.2 分析错误
- **LLM分析失败**：无法理解代码逻辑
- **字段映射冲突**：同一字段有多个可能的源
- **循环依赖**：数据流存在循环引用

#### 5.1.3 运行时错误
- **内存不足**：处理大型文件时内存溢出
- **文件访问错误**：无法读取或写入文件
- **配置错误**：LLM配置或参数错误

### 5.2 容错策略

#### 5.2.1 渐进式处理
- **部分成功**：即使某些文件分析失败，仍继续处理其他文件
- **错误隔离**：单个文件的错误不影响整体处理
- **降级处理**：在LLM不可用时，使用规则引擎进行基础分析

#### 5.2.2 错误恢复
- **重试机制**：对临时性错误（如网络问题）进行重试
- **备用分析**：当LLM分析失败时，尝试基于规则的分析
- **人工干预点**：标记需要人工审核的复杂情况

**规则引擎定义**：
- **基于正则表达式的字段映射规则**：
  - 字段重命名规则：`RENAME\s*\(\s*(\w+)\s*=\s*(\w+)\s*\)` → 提取源字段和目标字段映射
  - 直接赋值规则：`(\w+)\s*=\s*(\w+);` → 识别直接字段复制
  - 计算字段规则：`(\w+)\s*=\s*([^;]+);` → 提取计算公式和依赖字段

- **基于语法树的简单分析规则**：
  - DATA语句解析：提取目标数据集名
  - SET/MERGE语句解析：提取源数据集名和合并条件
  - KEEP/DROP语句解析：提取字段选择列表

**规则引擎触发条件**：
- LLM服务不可用（网络错误、API错误）
- LLM分析超时（超过配置的timeout阈值）
- LLM返回结果不符合预期格式
- 用户明确配置使用规则引擎（性能优先模式）

**规则配置方式**：
- **规则文件格式**：YAML格式的规则配置文件`rules.yaml`
```yaml
field_mapping_rules:
  - pattern: "RENAME\\s*\\(\\s*(\\w+)\\s*=\\s*(\\w+)\\s*\\)"
    action: "extract_rename_mapping"
  - pattern: "(\\w+)\\s*=\\s*(\\w+);"
    action: "extract_direct_assignment"

data_flow_rules:
  - pattern: "DATA\\s+(\\w+);"
    action: "extract_target_dataset"
  - pattern: "SET\\s+(\\w+);"
    action: "extract_source_dataset"
```
- **加载机制**：启动时加载规则文件，支持热重载
- **优先级控制**：可配置规则执行顺序和优先级

## 7. 部署与使用说明

### 7.1 安装要求
- Python 3.12+
- 必需依赖包：
  - langchain-core
  - langchain-community  
  - pandas
  - sas7bdat
  - deep-agents-sdk (可选)

### 7.2 配置文件
创建 `config.yaml` 配置文件：

```yaml
llm:
  provider: "openai"  # or "anthropic", "ollama", etc.
  model: "gpt-4-turbo"
  api_key: "${LLM_API_KEY}"
  max_tokens: 4096
  temperature: 0.1

processing:
  max_file_size_mb: 100
  chunk_size_lines: 100
  parallel_workers: 4
  cache_enabled: true

output:
  format: "csv"
  output_path: "./lineage_output.csv"
```

### 7.3 命令行接口
```bash
# 基本用法
python sas_lineage_analyzer.py --input-dir /path/to/sas/files --output lineage.csv

# 指定配置文件
python sas_lineage_analyzer.py --config config.yaml --input-dir /path/to/sas/files

# 详细模式（显示处理进度）
python sas_lineage_analyzer.py --verbose --input-dir /path/to/sas/files

# 仅分析特定文件
python sas_lineage_analyzer.py --files file1.sas file2.sas --output lineage.csv
```

### 7.4 输出格式
CSV输出包含以下列：
- `source_table`: 源表名或文件名
- `source_field`: 源表字段名（以定义文件为准）
- `target_table`: 目标表名或文件名  
- `target_field`: 目标表字段名
- `transformation`: 转换公式或操作描述
- `script_file`: 相关的SAS脚本文件名
- `lineage_path`: 完整的血缘路径（可选）

### 7.5 使用示例
```bash
# 分析示例SAS文件
python sas_lineage_analyzer.py --input-dir ./input/sas --output ./output/lineage.csv

# 查看输出结果
cat ./output/lineage.csv
# source_table,source_field,target_table,target_field,transformation,script_file
# churn_raw,EstimagteSalary,churn_out,est_sal,rename,01_churn_transform.sas
# churn_raw,Balance,churn_out,bal_per_prod,Balance / num_prod,01_churn_transform.sas
```

## 10. 风险与缓解

### 10.1 技术风险
- **LLM准确性**：LLM可能误解复杂的SAS逻辑
  - *缓解*：结合规则引擎，提供人工审核机制
- **性能瓶颈**：大型文件处理可能较慢
  - *缓解*：优化分块策略，增加缓存机制

### 10.2 业务风险
- **字段名不一致**：代码和定义文件字段名差异
  - *缓解*：严格按照需求澄清，以定义文件为准
- **复杂逻辑处理**：嵌套条件、宏程序等复杂SAS特性
  - *缓解*：逐步支持，优先处理常见场景
