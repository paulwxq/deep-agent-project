# SAS数据血缘分析系统技术设计文档

## 1. 项目概述

### 1.1 项目背景
本项目旨在开发一个Python应用模块，用于分析SAS代码和相关数据集，自动生成数据血缘分析对照表。该系统将帮助数据治理团队理解数据流转过程，追踪字段来源，确保数据质量和合规性。

### 1.2 核心需求
- **输入源**：SAS代码文件（.sas）、CSV文件、Excel文件、Mainframe数据定义
- **输出格式**：四元组关系表（源表名/文件名、源表字段、目标表名/文件名、目标表字段）
- **分析方向**：从输出表作为入口，反向追踪每个结果字段的源字段
- **数据类型支持**：
  - 源数据：Mainframe（主要）、CSV、Excel（次要）
  - 中间文件：SAS文件（.sas7bdat）
  - 输出文件：CSV格式
- **处理规模**：支持大量SAS代码文件的批量处理，考虑LLM上下文限制

### 1.3 技术栈
- Python 3.12
- 可配置的LLM Provider（支持多种大语言模型）
- LangChain-AI / Deep Agents SDK
- SAS代码解析器（自研）

#### 关键方法实现步骤
1. **词法分析**：将SAS代码分割为tokens（关键字、标识符、操作符等）
2. **语法分析**：构建AST（抽象语法树），识别DATA steps、PROC statements等
3. **语义提取**：
   - 从DATA steps中提取input_datasets、output_dataset、assignments、renames、keep_fields、drop_fields
   - 从PROC IMPORT/EXPORT中提取文件路径和格式信息
   - 从MERGE/JOIN语句中提取关联表信息

#### 异常处理策略
- **语法错误**：记录错误位置和描述，返回ParseResult.success=False，errors包含详细信息
- **不支持的语法**：跳过无法解析的部分，记录警告信息，继续解析其他部分
- **文件读取失败**：抛出FileNotFoundError，由调用方处理
- **内存不足**：对大文件采用流式解析，避免一次性加载全部内容

### 3.2 数据源管理模块
**职责**：管理不同类型的数据源元数据

#### Mainframe数据定义解析
- **解析目标**：def_*.sas文件（如def_Churn.sas、def_train.sas）
- **解析格式**：识别DATA step中的infile语句和input语句
- **字段定义格式**：
  ```sas
  input
      @1    CustomerId                       PK2.0
      @3    Surname                           PK2.0
      @5    CreditScore                             PK3.0
      @8    Geography                              S370FPD2.0
      @10   Gender                       $EBCDIC3.
      @13   Age                  S370FPD6.0
  ```
- **数据类型处理**：
  - `PKn.m`：Packed Decimal格式，n为总位数，m为小数位数
  - `S370FPDn.m`：IBM Mainframe Floating Point Decimal格式
  - `$EBCDICn`：EBCDIC编码的字符串，n为长度
  - 其他SAS数据类型按标准SAS规范处理
- **解析策略**：使用正则表达式匹配`@位置 字段名 数据类型`模式，提取字段名、位置、数据类型

#### 中间文件（.sas7bdat）处理策略
- **不实际读取文件**：由于.sas7bdat是二进制格式且可能很大，系统不会实际读取这些文件
- **结构推断方式**：
  1. 从生成该文件的SAS代码中推断字段结构
  2. 分析DATA step的SET语句、assignments、renames、keep/drop等操作
  3. 构建虚拟的字段元数据，包含字段名、数据类型（基于源字段或赋值表达式推断）、描述
- **字段类型推断规则**：
  - 直接传递的字段：继承源字段的数据类型
  - 重命名字段：继承原字段的数据类型
  - 派生字段：基于表达式类型推断（数值运算→numeric，字符串操作→character，逻辑表达式→boolean）

#### CSV/Excel数据源处理
- **优先从SAS代码推断**：通过PROC IMPORT语句获取字段信息
- **备选方案**：如果SAS代码中没有明确字段定义，则尝试读取实际CSV/Excel文件的前几行来推断字段名和类型

#### 核心算法流程
1. **目标识别**：从输出CSV文件或PROC EXPORT语句识别目标表和字段
2. **反向追踪**：
   - 在相关SAS文件中查找生成目标字段的DATA step
   - 如果是直接赋值（passthrough），继续向上游追踪源字段
   - 如果是重命名，追踪原字段名
   - 如果是派生字段，解析表达式并追踪所有依赖的源字段
3. **跨文件关联**：当遇到中间.sas7bdat文件时，查找生成该文件的SAS代码，继续追踪
4. **Mainframe源定位**：最终定位到def_*.sas文件中的Mainframe字段定义

#### 异常处理策略
- **字段血缘无法追溯**：
  - 记录警告信息，包含目标字段和已追踪到的位置
  - 返回部分血缘路径（如果存在）
  - 在最终输出中标记为"UNRESOLVED"
- **循环依赖检测**：
  - 维护访问过的字段集合，检测重复访问
  - 发现循环依赖时抛出CircularDependencyError
  - 提供循环路径详情用于调试
- **多源字段处理**：
  - 当字段来自多个源（如MERGE操作），记录所有源字段
  - 在LineageRecord中用分号分隔多个源（参考Output_sas_20260318.csv格式）

### 3.4 LLM集成模块
**职责**：与大语言模型交互，处理复杂逻辑解析

#### LLM调用触发条件
LLM仅在以下情况下被调用：
1. **复杂表达式解析**：当字段派生逻辑包含复杂的数学运算、条件表达式或函数调用，且规则引擎无法准确解析时
2. **模糊语法处理**：当SAS代码包含非标准语法、宏变量展开或动态代码生成时
3. **上下文依赖分析**：当需要理解跨多个DATA steps的复杂数据流逻辑时
4. **异常情况处理**：当规则引擎返回不确定结果（置信度低于阈值）时

#### 规则引擎与LLM协作策略
- **优先级顺序**：规则引擎 → LLM（fallback）
- **处理流程**：
  1. 首先使用规则引擎尝试解析SAS代码
  2. 如果规则引擎成功解析且置信度高（>95%），直接使用结果
  3. 如果规则引擎失败或置信度低，将相关代码片段提交给LLM
  4. LLM返回结果后，进行格式验证和逻辑一致性检查
  5. 如果LLM结果通过验证，使用LLM结果；否则标记为需要人工审核
- **上下文优化**：
  - 只提交相关的代码片段（而非整个文件）给LLM
  - 提供必要的上下文信息（如相关变量定义、数据结构）
  - 使用专门设计的prompt模板，明确要求输出格式

## 4. 数据血缘分析算法

### 4.1 算法概述
采用**反向追踪算法**，从最终输出表开始，逐层向上追溯字段来源：

1. **初始化**：识别所有输出CSV文件及其字段
2. **反向映射**：对于每个输出字段，找到其在SAS代码中的定义
3. **源字段识别**：
   - 如果是直接传递（passthrough），记录源字段
   - 如果是重命名（rename），记录原始字段名
   - 如果是派生字段（derived），解析表达式中的依赖字段
4. **递归追踪**：对依赖字段继续向上追溯，直到到达原始数据源
5. **血缘路径构建**：构建完整的字段血缘路径

### 4.2 处理复杂场景

#### 4.2.1 字段重命名
```sas
SET source_data (RENAME=(CreditScore = credit_scr));
```
- **血缘关系**：`target_table.credit_scr ← source_table.CreditScore`

#### 4.2.2 派生字段
```sas
bal_per_prod = Balance / num_prod;
```
- **血缘关系**：`target_table.bal_per_prod ← [source_table.Balance, source_table.NumOfProducts]`
- **转换公式**：`Balance / num_prod`

#### 4.2.3 多源合并
```sas
MERGE churn_prep loan_prep;
BY CustomerId;
risk_score = credit_scr - (dti * 10);
```
- **血缘关系**：`merged_table.risk_score ← [churn_table.CreditScore, loan_table.Debit_to_Income]`
- **转换公式**：`credit_scr - (dti * 10)`

#### 4.2.4 条件字段
```sas
senior_flag = (Age >= 60);
```
- **血缘关系**：`target_table.senior_flag ← source_table.Age`
- **转换公式**：`(Age >= 60)`

### 4.3 算法伪代码
```python
def trace_lineage(output_table, output_field, processed_nodes=None):
    """反向追踪单个字段的血缘关系，支持循环检测"""
    if processed_nodes is None:
        processed_nodes = set()
    
    node_key = f"{output_table}.{output_field}"
    if node_key in processed_nodes:
        return []  # 避免循环引用
    
    processed_nodes.add(node_key)
    
    # 1. 找到生成该字段的SAS代码文件和DATA step
    sas_file, data_step = find_generating_code(output_table, output_field)
    if not data_step:
        return []  # 无法找到生成代码，可能是原始数据源
    
    source_tables = data_step.input_datasets
    lineage_results = []
    
    # 2. 分析字段类型
    if is_passthrough_field(output_field, data_step):
        source_field = get_source_field(output_field, data_step)
        for source_table in source_tables:
            lineage_results.append((output_table, output_field, source_table, source_field, "passthrough", sas_file))
    
    elif is_renamed_field(output_field, data_step):
        original_field = get_original_field_name(output_field, data_step)
        for source_table in source_tables:
            lineage_results.append((output_table, output_field, source_table, original_field, 
                                  f"rename {original_field} -> {output_field}", sas_file))
    
    elif is_derived_field(output_field, data_step):
        formula = data_step.assignments[output_field]
        dependencies = extract_dependencies(formula)
        combined_sources = []
        combined_fields = []
        
        # 递归追踪所有依赖字段
        for dep_field in dependencies:
            dep_lineage = trace_lineage(get_source_table_for_field(dep_field, data_step), 
                                      dep_field, processed_nodes.copy())
            lineage_results.extend(dep_lineage)
            if dep_lineage:
                combined_sources.extend([item[2] for item in dep_lineage])
                combined_fields.extend([item[3] for item in dep_lineage])
        
        # 添加当前派生字段的血缘关系
        unique_sources = list(set(combined_sources))
        unique_fields = list(set(combined_fields))
        lineage_results.append((output_table, output_field, "; ".join(unique_sources), 
                              "; ".join(unique_fields), formula, sas_file))
    
    else:
        # 使用LLM辅助解析复杂或未知情况
        llm_result = llm_assisted_lineage_analysis(sas_file, data_step, output_field)
        lineage_results.extend(llm_result)
    
    return lineage_results
```

```python
            if field['name'] == field_name:
                return field
        return None
```

##### ParseResult 类定义（补充）
```python
class ParseResult:
    """
    SAS解析结果
    """
    def __init__(self, 
                 data_steps: List[SASDataStep], 
                 import_statements: List[Dict], 
                 export_statements: List[Dict], 
                 errors: List[str] = None):
        self.data_steps = data_steps or []           # 必填：DATA steps列表，不能为空
        self.import_statements = import_statements or []  # 必填：导入语句列表
        self.export_statements = export_statements or []  # 必填：导出语句列表  
        self.errors = errors or []                   # 可选：错误信息列表
        self.success = len(errors) == 0              # 必填：解析是否成功
        
    def get_output_tables(self) -> Set[str]:
        """获取所有输出表名"""
        tables = set()
        for step in self.data_steps:
            tables.add(step.name)
        for export in self.export_statements:
            if 'outfile' in export:
                tables.add(export['outfile'])
        return tables
```

##### 字段约束说明
- **input_datasets**: 可以为空列表（表示没有明确的SET语句），但不能为None
- **assignments**: key-value格式为 `{target_field: expression}`，expression为字符串形式的SAS表达式
- **keep_fields vs drop_fields**: 两者互斥，如果同时存在，优先使用keep_fields
- **original_table/original_column**: 多个源时用分号分隔（如"churn_raw; loan_raw"）
- **formula**: 支持的操作类型包括：
  - "passthrough": 直接传递
  - "rename old_name -> new_name": 重命名操作  
  - "expression": 派生字段的SAS表达式
  - "merge": 多表合并操作

### 5.2 LLM集成策略

#### 5.2.1 上下文分块策略
由于SAS代码可能很长且LLM有上下文限制，采用以下分块策略：
- **按DATA step分块**：每个DATA step作为一个独立的上下文单元（主要策略）
- **按文件分块**：单个SAS文件过大时，按逻辑段落分块
- **优先级排序**：优先处理与输出表相关的代码段
- **依赖预加载**：在调用LLM前预加载相关变量和宏定义，减少上下文占用

### 5.4 模块交互流程
详细的模块调用流程如下：

#### 5.4.1 用户调用入口
1. **CLI入口**：用户通过命令行执行 `python lineage_analyzer.py --config config.yaml`
2. **API入口**：通过REST API调用 `/api/analyze-lineage` 端点
3. **参数解析**：解析配置文件和命令行参数，初始化系统配置

#### 5.4.2 主处理流程
1. **数据源发现**：
   - 扫描配置的`sas_paths`目录，收集所有.sas文件
   - 扫描`data_sources`配置，收集def_*.sas、CSV、Excel文件
   - 构建数据源元数据索引

2. **输出目标识别**：
   - 解析所有SAS文件，识别PROC EXPORT语句和最终输出表
   - 确定需要分析的输出表列表

3. **批量血缘分析**：
   - 调用`LineageAnalyzer.analyze_batch_lineage()`方法
   - 传入输出文件列表和SAS文件列表

4. **核心分析循环**：
   - 对每个输出表的每个字段，调用`analyze_lineage()`进行反向追踪
   - 在追踪过程中，按需调用各模块：
     - **SASCodeParser**：解析相关SAS文件
     - **DataSourceManager**：获取数据源元数据
     - **LLMIntegration**：处理复杂逻辑（仅在必要时）

5. **结果聚合与验证**：
   - 收集所有LineageRecord对象
   - 进行去重和合并处理
   - 验证血缘关系的完整性和一致性

6. **输出生成**：
   - 调用输出生成模块，将LineageRecord转换为CSV格式
   - 写入指定的输出路径

#### 5.4.3 模块间数据传递
- **SASCodeParser → LineageAnalyzer**：传递ParseResult对象
- **DataSourceManager → LineageAnalyzer**：传递DataSourceMetadata对象  
- **LineageAnalyzer → LLMIntegration**：传递代码片段和上下文信息
- **LLMIntegration → LineageAnalyzer**：返回解析后的结构化信息
- **LineageAnalyzer → OutputGenerator**：传递LineageRecord列表

#### 5.4.4 异常处理流程
- **解析错误**：SASCodeParser记录错误，LineageAnalyzer跳过相关字段
- **血缘断裂**：LineageAnalyzer标记为UNRESOLVED，继续处理其他字段
- **LLM失败**：回退到规则引擎，或标记需要人工审核
- **系统错误**：记录详细日志，提供错误恢复机制

## 6. 部署和配置

### 6.1 配置文件设计
```yaml
# config.yaml
sas_paths:
  - /path/to/sas/code/directory
  - /path/to/additional/sas/files

data_sources:
  mainframe_definitions:
    - /path/to/def_*.sas
  csv_files:
    - /path/to/csv/directory
  excel_files:
    - /path/to/excel/directory

llm_provider:
  type: openai  # or anthropic, gemini, etc.
  model: gpt-4-turbo
  api_key: ${LLM_API_KEY}
  max_tokens: 4096
  temperature: 0.1
  retry_attempts: 3        # LLM调用失败时的重试次数
  timeout_seconds: 60      # LLM调用超时时间（秒）

output:
  format: csv
  path: /path/to/output/lineage_results.csv

processing:
  batch_size: 10           # 同时处理的SAS文件数量，影响内存使用和并行度
  max_context_length: 3000 # LLM上下文长度限制，用于分块策略决策
  cache_enabled: true      # 是否启用解析结果缓存
  parallel_workers: 4      # 并行处理的工作线程数
```

#### 配置项详细说明

**sas_paths**
- **作用**：指定包含SAS代码文件的目录路径列表
- **类型**：字符串列表
- **默认值**：空列表
- **约束**：至少需要指定一个有效路径

**data_sources.mainframe_definitions**
- **作用**：指定Mainframe数据定义文件（def_*.sas）的路径模式
- **类型**：字符串列表（支持通配符）
- **示例**：`["/path/to/def_*.sas", "/another/path/definitions/*.sas"]`

**llm_provider.max_tokens**
- **作用**：控制LLM单次调用的最大token数量，影响上下文窗口大小
- **类型**：整数
- **默认值**：4096
- **影响**：值越大，单次可处理的代码片段越长，但成本越高

**llm_provider.retry_attempts**
- **作用**：LLM调用失败时的自动重试次数
- **类型**：整数
- **默认值**：3
- **策略**：指数退避重试，避免临时网络问题导致失败

**llm_provider.timeout_seconds**
- **作用**：LLM API调用的超时时间
- **类型**：整数（秒）
- **默认值**：60
- **影响**：防止长时间等待无响应的LLM调用

**processing.batch_size**
- **作用**：控制同时处理的SAS文件数量
- **类型**：整数
- **默认值**：10
- **影响**：值越大，并行度越高，但内存消耗也越大；需要根据系统资源调整

**processing.max_context_length**
- **作用**：用于决定SAS代码分块策略的上下文长度阈值
- **类型**：整数
- **默认值**：3000
- **影响**：当代码片段超过此长度时，会触发分块处理逻辑

**processing.cache_enabled**
- **作用**：是否启用SAS代码解析结果的缓存
- **类型**：布尔值
- **默认值**：true
- **优势**：避免重复解析相同文件，提升性能

**processing.parallel_workers**
- **作用**：并行处理的工作线程数量
- **类型**：整数
- **默认值**：4
- **建议**：设置为CPU核心数的1-2倍以获得最佳性能
