# SAS数据血缘分析系统技术设计文档

## 1. 项目概述

### 1.1 背景与目标
本项目旨在开发一个Python应用模块，用于分析SAS代码和相关数据集，自动生成数据血缘分析对照表。系统将从输出CSV文件作为入口点，反向追踪每个目标字段到其源字段的完整血缘路径，支持复杂的SAS数据处理场景包括字段重命名、派生计算和多源数据合并。

### 1.2 核心需求
- **输入**：SAS代码文件（.sas）、SAS数据集定义文件（def_*.sas）、输出CSV文件
- **输出**：数据血缘对照表（源表名/文件名、源表字段、目标表名/文件名、目标表字段、转换公式、脚本文件）
- **处理模式**：从输出表反向追踪到源表
- **扩展性**：支持大量文件的分批处理，避免LLM上下文限制

### 1.3 技术栈
- Python 3.12
- LangChain-AI + Deep Agents SDK
- 可配置的LLM Provider（支持OpenAI、Anthropic等）
- Pandas（数据处理）
- NetworkX（依赖图构建）

## 2. 系统架构设计

### 2.1 整体架构
系统采用分层架构设计，包含以下核心组件：

```
┌─────────────────┐    ┌──────────────────┐    ┌─────────────────┐
│   文件发现层     │───▶│   依赖分析层      │───▶│   血缘解析层     │
└─────────────────┘    └──────────────────┘    └─────────────────┘
        │                       │                       │
        ▼                       ▼                       ▼
┌─────────────────┐    ┌──────────────────┐    ┌─────────────────┐
│   配置管理层     │◀───│   LLM集成层       │───▶│   结果输出层     │
└─────────────────┘    └──────────────────┘    └─────────────────┘
```

### 2.2 组件职责
- **文件发现层**：扫描指定目录，识别SAS代码文件、数据集定义文件和CSV文件
- **依赖分析层**：构建文件依赖图，确定处理顺序（从输出文件到源文件）
- **血缘解析层**：逐个分析SAS脚本，提取字段映射关系
- **LLM集成层**：调用LLM进行复杂SAS语法解析和表达式理解
- **配置管理层**：管理LLM配置、文件路径配置和处理参数
- **结果输出层**：生成标准化的血缘对照表

### 2.3 数据流设计
1. **初始化阶段**：加载配置，扫描文件系统
2. **依赖图构建阶段**：分析所有SAS脚本，建立输出文件到输入文件的映射关系
3. **血缘分析阶段**：按依赖顺序处理每个输出文件
4. **结果聚合阶段**：合并所有血缘关系，生成最终对照表

## 4. 数据血缘分析算法

### 4.1 算法流程
1. **预处理阶段**：
   - 扫描所有SAS文件，提取PROC EXPORT语句识别输出文件
   - 提取DATA步骤中的SET/MERGE语句识别输入数据集
   - 构建完整的文件依赖图

2. **反向追踪阶段**：
   - 从每个输出CSV文件开始
   - 找到生成该文件的SAS脚本
   - 分析脚本中的字段映射关系

3. **字段映射分析**：
   - **直通字段**：出现在KEEP语句中且无RENAME操作
   - **重命名字段**：通过RENAME=(old_name=new_name)语法
   - **派生字段**：在DATA步骤中定义的新变量
   - **多源字段**：来自MERGE操作的多个输入数据集

### 4.1.1 具体解析规则

#### PROC EXPORT语句解析
使用正则表达式提取输出文件信息：
```python
# 匹配PROC EXPORT语句
proc_export_pattern = r'PROC EXPORT\s+DATA\s*=\s*(\w+)\s+OUTFILE\s*=\s*"([^"]+)"'
# 提取输出数据集名和文件路径
```

#### SET/MERGE语句解析
```python
# 匹配SET/MERGE语句
set_merge_pattern = r'(?:SET|MERGE)\s+(\w+(?:\s*\(.*?\))?(?:\s+\w+(?:\s*\(.*?\))?)*);'
# 提取输入数据集列表和选项（如RENAME、KEEP、DROP）
```

#### RENAME语法解析
```python
# 匹配RENAME=(old_name=new_name)语法
rename_pattern = r'RENAME\s*=\s*\(([^)]+)\)'
# 解析重命名对：old_name=new_name, old2=new2
def parse_rename_clause(rename_clause: str) -> Dict[str, str]:
    rename_pairs = {}
    for pair in rename_clause.split(','):
        if '=' in pair:
            old_name, new_name = pair.strip().split('=', 1)
            rename_pairs[old_name.strip()] = new_name.strip()
    return rename_pairs
```

#### KEEP/DROP语句解析
```python
# 匹配KEEP/DROP语句
keep_drop_pattern = r'(KEEP|DROP)\s*=\s*(.*?);'
# 提取字段列表
def parse_keep_drop_fields(fields_str: str) -> List[str]:
    # 处理字段列表：field1 field2 field3 或 field1-field3
    fields = []
    for part in fields_str.split():
        if '-' in part and not any(c in part for c in ['"', "'"]):
            # 处理范围：field1-field3
            start, end = part.split('-')
            fields.extend([start, end])  # 简化处理，实际需要更复杂的逻辑
        else:
            fields.append(part)
    return fields
```

#### source_tables字段格式规范
- **表名格式**：仅包含逻辑表名，不包含文件路径
- **扩展名处理**：不包含文件扩展名（如.csv、.sas7bdat）
- **mainframe数据集**：使用逻辑数据集名（如churn_raw），物理文件路径在notes字段中说明
- **CSV/Excel源文件**：使用逻辑表名（如churn_modeling_1），物理路径在notes字段中说明
- **中间结果**：使用SAS7BDAT逻辑名（如churn_out）

此设计确保血缘记录的一致性和可读性，同时通过notes字段保留必要的物理路径信息。

### 4.3 复杂场景处理

#### 4.3.1 多级依赖处理
对于中间结果（如churn_out.sas7bdat被03_merge_transform.sas使用），系统采用递归反向追踪：
1. 识别中间文件的生成脚本
2. 获取该脚本的血缘关系
3. 将中间字段映射到原始源字段
4. 合并多级血缘路径

**多级血缘路径合并规则**：
- **直通字段合并**：如果中间字段是直通字段，则直接使用其源字段
- **重命名字段合并**：保留最终重命名，源字段追溯到原始字段
- **派生字段合并**：组合公式表达式，用括号明确运算优先级
- **多源字段合并**：合并所有源表和源字段列表，去重处理

示例：
```
churn_out.credit_scr <- churn_raw.CreditScore (rename)
merged_out.credit_scr <- churn_out.credit_scr (passthrough)
最终结果：merged_out.credit_scr <- churn_raw.CreditScore (rename CreditScore -> credit_scr)
```

#### 4.3.2 条件逻辑处理
SAS中的条件赋值需要特殊处理：
```sas
/* 示例：条件字段赋值 */
IF Age >= 60 THEN senior_flag = 1;
ELSE senior_flag = 0;
```
处理策略：
- 使用LLM识别条件表达式中的源字段
- 记录所有可能的源字段路径
- 标记转换类型为"conditional"

#### 4.3.3 聚合操作处理
PROC SUMMARY/MEANS等聚合操作：
```sas
PROC SUMMARY DATA=input;
    VAR Balance;
    OUTPUT OUT=output MEAN=avg_balance;
RUN;
```
处理策略：
- 识别聚合函数类型（SUM、MEAN、COUNT等）
- 记录分组字段（BY语句）
- 源字段标记为聚合输入字段

#### 4.3.4 数组和循环处理
DO循环中的字段处理：
```sas
ARRAY new_vars[3] var1-var3;
DO i = 1 TO 3;
    new_vars[i] = old_vars[i] * 2;
END;
```
处理策略：
- 识别数组声明和循环结构
- 建立数组元素间的映射关系
- 使用通配符表示重复模式

#### 4.3.5 特殊SAS语法处理
- **FORMAT/INFORMAT**：字段格式转换不影响血缘
- **LABEL**：字段标签不影响血缘
- **DROP/KEEP**：影响字段可见性，需在血缘中体现
- **WHERE**：过滤条件中的字段也应记录为源字段

### 4.4 数据集名称映射机制

为解决逻辑数据集名与物理文件的映射问题，建立以下映射机制：

#### PROC IMPORT语句解析
从PROC IMPORT语句中提取输入文件路径和输出数据集名：
```python
# 匹配PROC IMPORT语句
proc_import_pattern = r'PROC IMPORT\s+DATAFILE\s*=\s*"([^"]+)"\s+OUT\s*=\s*(\w+)'

```

#### libname语句解析
从libname语句中提取库路径信息：


2. **间接映射**：通过libname + dataset引用建立映射（如lib.dataset）
3. **宏变量处理**：解析&macro_var等宏变量，还原真实路径

## 5. 文件处理策略

### 5.1 分批处理机制
为解决LLM上下文限制问题，采用以下策略：

1. **依赖图分割**：将大型依赖图分割为独立的子图（无交叉依赖的组件）
2. **增量处理**：按拓扑顺序逐个处理文件，确保依赖先于被依赖项处理
3. **上下文缓存**：使用LRU缓存存储已解析的SAS脚本和血缘关系，避免重复LLM调用
4. **批量提交**：对于相关性高的小文件组，合并提交给LLM以提高效率
5. **优先级队列**：优先处理关键路径上的文件（影响最多下游的文件）

### 5.2 上下文管理策略
- **Token预算管理**：动态计算每个批次的token使用量，确保不超过LLM限制
- **摘要压缩**：对大型SAS脚本生成摘要，保留关键DATA步骤信息
- **增量上下文**：在连续调用中传递必要的上下文信息，减少重复传输
- **错误恢复**：单个文件处理失败不影响整体流程，支持重试机制

### 5.2 文件分类策略
- **源文件**：mainframe SAS数据集定义文件（def_*.sas）
- **处理文件**：SAS转换脚本（*.sas）
- **输出文件**：CSV结果文件（*.csv）
- **中间文件**：SAS7BDAT临时文件

### 5.3 内存管理
- 使用生成器模式处理大文件
- 实现LRU缓存机制存储解析结果
- 支持断点续处理，保存中间状态

### 6.3 LLM调用完整流程

#### 代码分块策略
1. **按步骤分块**：将SAS代码按DATA步骤、PROC步骤分割
2. **大小控制**：每个块不超过2000 tokens，确保LLM处理效率
3. **上下文保留**：在每个块前添加必要的上下文信息（如宏变量定义、libname语句）

#### 结果解析与验证
```python
from pydantic import BaseModel, ValidationError

class LLMParseResult(BaseModel):
    data_steps: List[DataStep]
    
def parse_llm_response(response: str) -> LLMParseResult:
    """解析LLM返回的JSON结果，使用Pydantic模型验证"""
    try:
        json_data = json.loads(response)
        return LLMParseResult(**json_data)
    except (json.JSONDecodeError, ValidationError) as e:
        # 处理格式错误
        raise LLMResponseFormatError(f"LLM返回格式错误: {e}")
```

#### 错误处理与重试机制
1. **格式错误处理**：
   - 检测JSON格式错误
   - 提取有效部分，标记无效部分
   - 重新提交错误部分给LLM
   
2. **重试策略**：
   - 最多重试3次
   - 每次重试提供更详细的提示
   - 降级到规则引擎作为备选方案

3. **降级处理**：
   - 当LLM连续失败时，使用正则表达式规则引擎
   - 标记低置信度结果供人工审核
   - 记录失败案例用于后续模型优化

### 6.3 错误处理与降级策略

#### 6.3.1 多层降级机制
1. **Primary（LLM解析）**：使用LLM进行完整的语义理解
2. **Secondary（规则引擎）**：基于预定义规则的正则表达式匹配
3. **Tertiary（基础解析）**：简单的关键词和语法结构识别

#### 6.3.2 置信度评估
置信度评分计算方法：
- **基础分值（0.0-1.0）**：
  - 完整SAS语法：+0.3
  - 清晰字段映射：+0.3  
  - 无歧义表达式：+0.2
  - LLM解析成功：+0.2

- **高置信度（>0.8）**：LLM结果一致，语法结构清晰，所有字段映射明确
- **中置信度（0.5-0.8）**：部分模糊，需要验证，可能存在多个解释
- **低置信度（<0.5）**：复杂或不完整代码，标记为人工审核

置信度影响：
- 高置信度：直接接受结果
- 中置信度：标记为需要验证，可自动处理但需记录
- 低置信度：进入人工审核队列

#### 6.3.3 人工审核工作流
- 自动生成审核队列，包含低置信度结果
- 提供上下文信息和建议修正
- 支持人工修正后的结果反馈到系统
- 修正结果用于改进后续的LLM提示工程

#### 6.3.4 异常处理
- **语法错误**：跳过无法解析的代码段，记录警告
- **文件缺失**：标记断链血缘，继续处理其他路径
- **LLM超时**：自动重试或降级到规则引擎
- **内存溢出**：分批处理，释放中间结果

## 11. 异常处理完整方案

### 11.1 文件相关异常
- **文件不存在**：记录警告，跳过处理，继续其他文件
- **文件读取错误**：重试机制（最多3次），失败后标记为人工审核
- **文件格式错误**：使用规则引擎尝试解析，标记低置信度

### 11.2 解析相关异常
- **SAS语法错误**：提取可解析部分，标记错误位置
- **字段映射模糊**：生成多个可能的映射，标记中等置信度
- **表达式无法解析**：保留原始表达式，标记需要人工审核

### 11.3 LLM相关异常
- **API调用失败**：指数退避重试（最多3次）
- **响应格式错误**：使用正则表达式提取有效信息
- **超时异常**：降级到规则引擎，标记低置信度

### 11.4 系统异常
- **内存溢出**：分批处理，释放中间结果，保存断点状态
- **磁盘空间不足**：清理缓存，暂停处理，发出警告
- **网络中断**：保存当前进度，支持断点续处理

### 11.5 异常恢复机制
- **自动恢复**：对于临时性错误（网络、API限流），自动重试
- **手动恢复**：提供命令行工具，支持从断点继续处理
- **数据一致性**：确保部分失败不影响已成功处理的结果
