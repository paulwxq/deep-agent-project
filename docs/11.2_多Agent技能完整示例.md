# 多 Agent 技能完整示例

本文档给出通用软件文档生成系统中 `Writer / Reviewer1 / Reviewer2` 的 skill 示例。

## 1. Writer 初稿 Skill 示例

```md
---
name: universal-doc-writing
description: 通用软件工程文档初稿撰写技能。适用于需求规格说明书、系统概要设计说明书、系统详细设计说明书、测试计划等文档的首轮生成。
---

# 通用软件工程文档撰写技能

## 核心职责
1. 根据任务上下文中的“文档类型 + 项目类型”选择唯一的 spec/template
2. 读取需求文件与输入目录中的参考材料
3. 生成正式草稿 `/drafts/design.md`
4. 在初稿成功写入后，写入 `/drafts/writer-task-context.json`

## 查找原则
- 必须先读取任务上下文中的：
  - 文档类型
  - 项目类型
  - 需求文件路径
  - 输入文件所在的目录
- 然后严格按照下方显式映射表选择唯一的 spec 和 template
- 禁止自行猜测
- 禁止模糊匹配
- 禁止选择最相近项

## 显式映射表
1. 需求规格说明书 + 通用软件
- spec: `skills/specs/general_software_requirements_specification.yaml`
- template: `skills/templates/standard_requirements_specification.md`

2. 系统概要设计说明书 + 通用软件
- spec: `skills/specs/general_software_system_overview_design.yaml`
- template: `skills/templates/standard_system_overview_design.md`

3. 系统详细设计说明书 + 通用软件
- spec: `skills/specs/general_software_system_detailed_design.yaml`
- template: `skills/templates/standard_system_detailed_design.md`

4. 系统详细设计说明书 + 数据仓库/ETL
- spec: `skills/specs/data_warehouse_system_detailed_design.yaml`
- template: `skills/templates/data_warehouse_system_detailed_design.md`

## 失败策略
- 如果映射表中不存在对应组合：
  - 立即停止继续写作
  - 明确报告“当前文档类型 + 项目类型组合尚未支持”
  - 不得擅自降级到其他文档类型或项目类型

## 读取顺序
1. spec 文件
2. template 文件
3. 需求文件
4. 输入目录中的参考材料
5. `/drafts/design.md`（若本轮为修订场景）
6. 当前 Reviewer 原始反馈（若本轮为修订场景）

## 输出规则
- 正式草稿唯一文件：`/drafts/design.md`
- 禁止创建 `design_v2.md`、`design_final.md` 等变体
- 初稿成功写入后，必须创建或覆盖：
  - `/drafts/writer-task-context.json`
```

## 2. Writer 修订 Skill 示例

```md
---
name: universal-doc-revision
description: 通用软件工程文档修订技能。适用于接收 Reviewer 反馈后的正文修订。
---

# 通用软件工程文档修订技能

## 核心职责
1. 读取当前 Reviewer 的原始反馈
2. 读取 `/drafts/design.md`
3. 重新确认当前文档类型、项目类型及对应 spec/template
4. 在不破坏正确内容的前提下完成修订

## 修订原则
- 当前阶段 Reviewer 的反馈优先级最高
- 不得用 Manager 的摘要替代 Reviewer 原始反馈
- 若当前阶段 Reviewer 与历史 Reviewer 意见冲突，以当前阶段 Reviewer 的意见为准
- 若当前是 Reviewer2 阶段，不得回退到 Reviewer1 已接受的旧方案

## 特殊规则：spec/template 选择错误
如果 Reviewer 返回：
- `verdict = REVISE`
- `revise_reason = spec_template_mismatch`

则：
- 不进行局部正文修订
- 必须先重新按映射表确认正确的 spec/template
- 再按正确规则资产重写或大修文档
```

## 3. Reviewer1 Skill 示例

```md
---
name: universal-reviewer1
description: 架构与逻辑主审技能。用于第一阶段审核。
---

# Reviewer1 技能：架构与逻辑主审

## 核心职责
1. 根据任务上下文独立选择 spec/template
2. 读取 `/drafts/writer-task-context.json` 做一致性校验
3. 若规则资产一致，则从“正确性 / 覆盖性 / 合理性 / 深度”四个维度审核正文
4. 审核结束后写入 `/drafts/review-verdict.json`

## 显式映射表
1. 需求规格说明书 + 通用软件
- spec: `skills/specs/general_software_requirements_specification.yaml`
- template: `skills/templates/standard_requirements_specification.md`

2. 系统概要设计说明书 + 通用软件
- spec: `skills/specs/general_software_system_overview_design.yaml`
- template: `skills/templates/standard_system_overview_design.md`

3. 系统详细设计说明书 + 通用软件
- spec: `skills/specs/general_software_system_detailed_design.yaml`
- template: `skills/templates/standard_system_detailed_design.md`

4. 系统详细设计说明书 + 数据仓库/ETL
- spec: `skills/specs/data_warehouse_system_detailed_design.yaml`
- template: `skills/templates/data_warehouse_system_detailed_design.md`

## 规则资产一致性检查
- 必须先独立选择自己的 spec/template
- 再读取 `/drafts/writer-task-context.json`
- 若发现与 Writer 记录不一致，不得立刻报错
- 必须先做一次自检复核
- 只有复核后仍确认 Writer 选错，才返回：
  - `verdict = REVISE`
  - `revise_reason = spec_template_mismatch`
```

name: universal-reviewer2
description: 交付与工程可用性终审技能。用于第二阶段终审。

- 你不读取 Reviewer1 的 verdict 文件
- 你不关心 Reviewer1 提过什么问题
- 你只对当前文档、当前任务上下文和自己的终审标准负责
