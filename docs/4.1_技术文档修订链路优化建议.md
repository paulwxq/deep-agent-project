# 4.1 技术文档修订链路优化建议

## 文档目的

本文档用于沉淀 Writer-Reviewer 技术文档生成链路的修订优化建议，重点解决以下问题：

- 修订轮重复勘查 `/input/`，导致上下文冗余和执行变慢
- Writer 在修订阶段倾向于整稿重写，而不是针对问题章节做局部修订
- Reviewer 反馈不够章节化，难以驱动 Writer 稳定使用 `edit_file`
- `drafts/_backups`、`output` 等非目标目录缺少明确的访问约束
- 当前模型组合与 thinking 配置需要更明确地服务于稳定性和质量平衡

本文档不直接修改代码，而是作为后续 prompt、skill、配置和流程实现的设计依据。

---

## 4.1.1 问题背景与当前状态

本节不再将所有问题一概视为“当前未修复问题”，而是区分为两类：

- 已经在 prompt 层面基本落实或部分缓解的问题
- 当前仍然存在、需要继续修复的问题

这样可以避免把历史问题与现存问题混放在同一层级，造成实现状态判断混乱。

### 4.1.1.1 历史问题：目录勘查范围过宽且轮次重复（已部分缓解）

这一问题最初确实存在，并且是本轮优化讨论的重要起点。当前 `writer_prompt.py` 已经区分“初稿模式”和“修订模式”，明确要求修订模式默认不要重新全量勘查 `/input/`，因此 Writer 侧已经得到明显缓解。

但该问题不宜直接视为“完全消失”，原因包括：

- Reviewer 侧仍然保留审核前勘查 `/input/` 的流程
- Writer 代码中仍存在 `write_todos` 建议与修订模式规则不完全一致的残留问题，需后续按 4.1 文档继续收敛

因此，本项更适合作为“历史问题与部分缓解状态”记录，而不应继续笼统表述为“当前完全未修复问题”。

### 4.1.1.2 历史问题：修订阶段缺少“局部修改优先”约束（已基本落实）

这一问题在早期版本中成立，但当前 Writer prompt 已经补入修订模式、`edit_file` 优先、仅在例外条件下允许整份 `write_file` 等约束，因此其核心缺口已基本被覆盖。

本项仍保留在文档中，主要用于说明为什么后续方案要继续强化 `edit_file` 策略与 `tech-doc-revision` skill，而不是表示该问题在当前代码中仍原样存在。

### 4.1.1.3 历史问题：Reviewer 反馈缺少章节锚点（已基本落实）

这一问题同样属于优化背景。当前 Reviewer prompt 已经要求输出“必须修改项（章节化）”，并明确要求在反馈中标出影响章节、修改动作和参考依据，因此 Reviewer 缺少章节锚点的问题已基本得到修复。

本项继续保留，是为了说明为什么 4.1 后续章节会强调 Reviewer 直接提供局部修改建议，而不是表示 Reviewer 仍处于完全无章节锚点的状态。

### 4.1.1.4 历史问题：非目标目录缺少访问限制（已基本落实）

当前 Writer 和 Reviewer prompt 都已经补入目录访问边界，包括禁止浏览 `/drafts/_backups`、`/output` 以及与当前任务无关的历史目录。因此，这一项也更适合作为“历史问题及其修复背景”保留，而不是继续被表述为当前完全未修复问题。

保留本项的意义在于：说明这些访问边界为什么必要，以及后续 skill 与流程设计应继续与该边界保持一致。

### 4.1.1.5 当前仍待修复：`tech-doc-review` skill 中的需求路径与系统真实路径不一致

与前四项不同，这一问题目前仍然真实存在。`tech-doc-review` skill 仍将需求原文硬编码为 `/drafts/requirement.md`，但系统真实需求路径来自 `/input/<requirement_filename>`。这与 Reviewer prompt（`reviewer_prompt.py` 通过 `agent_factory.py` 传参生成的实际路径）已形成不一致，会带来以下问题：

- Reviewer skill 与 Reviewer prompt 对需求文件位置的约束相互矛盾
- Reviewer 可能尝试读取并不存在的 `/drafts/requirement.md`
- “需求覆盖性检查”的对照基线被写错，影响审核稳定性
- 如果只增强 skill 的反馈格式，而不修复该路径问题，会导致旧 bug 被原样保留

截至当前文档版本，这一问题仍处于“文档已识别、代码未修复”的状态。也就是说，`reviewer_prompt.py` 已使用正确路径，但 `skills/tech-doc-review/SKILL.md` 仍保留错误路径，两者仍在运行态中并存。

---

## 4.1.2 优化目标

本轮优化的目标如下：

1. 首轮保留必要的 `/input/` 勘查能力，修订轮减少无效重复读取
2. 明确规定 Writer 修订时优先使用 `edit_file`
3. 让 Reviewer 输出可直接驱动局部修订的章节级反馈
4. 保持 Orchestrator 的编排职责简单，避免其二次解释 Reviewer 意图
5. 对 `/drafts/_backups`、`/output` 等目录建立明确访问边界
6. 固化一套更稳健的模型搭配与 thinking 策略

---

## 4.1.3 目录访问与勘查策略

### 4.1.3.1 首轮目录勘查规则

首轮撰写技术设计文档时，允许并要求执行目录勘查，但范围应明确限定为：

- 必须读取主需求文件 `/input/<requirement_filename>`
- 必须勘查 `/input/` 顶层目录
- 如 `/input/` 存在与需求直接相关的子目录，可继续进入相关子目录
- 可按需读取 `/input/` 中的参考文件
- 可检查 `/drafts/qa-supplement.md` 是否存在

首轮默认不应扫描以下目录：

- `/drafts/_backups`
- `/output`
- 与本轮需求无关的历史目录

### 4.1.3.2 修订轮目录勘查规则

修订轮默认不重新执行全量目录勘查。这里需要明确区分两类动作：

- 应削减的是：`ls` / `glob` 式目录勘查，以及与当前反馈无直接关系的非核心参考资料回读
- 不应削减的是：主需求文件 `/input/<requirement_filename>` 的读取与对照，它仍然是修订轮判断是否接受 Reviewer 意见的基线之一

也就是说，修订轮虽然不再机械地全量扫描 `/input/`，但主需求文件不应被排除在外，而应作为 Writer 的“常驻参考”。如果存在 `/drafts/qa-supplement.md`，则二者共同构成需求基线，且以 `qa-supplement.md` 对冲突项的澄清为准。

修订轮的读取优先级应改为：

1. Reviewer 反馈
2. 当前工作稿 `/drafts/design.md`
3. 主需求文件 `/input/<requirement_filename>`
4. `/drafts/qa-supplement.md`（若存在，则作为需求澄清的高优先级补充）
5. 与反馈直接相关的单个或少量非核心参考文件

修订轮只有在以下情况才允许重新回读 `/input/` 中的参考文件：

- Reviewer 明确指出某个事实与参考文件不一致
- Writer 无法确认某段设计是否与输入样例相符
- 修订内容直接依赖某个未读或需复核的输入文件

换言之，修订轮要减少的是“参考资料勘查范围”，而不是放弃“需求原文对照”。如果 Writer 在修订阶段只看 Reviewer 反馈而不再对照主需求文件，就可能被 Reviewer 的局部偏见或表述偏差带偏，导致修订结果偏离原始需求。

### 4.1.3.3 明确禁止浏览的目录

建议在 Writer 和 Reviewer 的 prompt 中增加硬性约束：

- 不浏览 `/drafts/_backups`
- 不浏览 `/output`
- 不浏览与当前任务无关的临时目录

如需查看当前草稿，只允许读取：

- `/drafts/design.md`
- `/drafts/qa-supplement.md`
- `/drafts/review-verdict.json`

## 4.1.4 Writer 修订阶段的 `edit_file` 优先机制

### 4.1.4.1 总体原则

Writer 的工作应拆分为两个模式：

- 初稿模式：允许生成完整文档
- 修订模式：优先局部修改现有文档

修订模式的核心原则是：

- 首选 `edit_file`
- 只修改受影响章节
- 仅当需要从零生成新稿时，才允许 `write_file`

### 4.1.4.2 修订模式的标准执行流程

建议 Writer 在修订模式下严格遵循以下步骤：

1. 读取 Reviewer 反馈
2. 读取 `/drafts/design.md`
3. 读取主需求文件 `/input/<requirement_filename>`，必要时同时对照 `/drafts/qa-supplement.md`
4. 先判断 Reviewer 意见是否与需求基线一致，再将反馈映射到具体章节
5. 判断每条反馈属于：
   - 补充现有章节
   - 调整现有章节表述
   - 新增一个局部小节
6. 优先使用 `edit_file` 修改对应章节
7. 完成所有局部修订后，快速回读受影响章节及其上下文，执行一次轻量一致性自检
8. 仅在满足例外条件时，才允许整份 `write_file`

这里新增“自检一致性”步骤，是为了避免局部修订虽然命中了目标章节，却破坏了文档的整体闭环。例如：

- A 章节修改后，B 章节仍引用旧术语或旧参数名
- 某个新增小节与前文定义的概念不一致
- 章节编号、标题路径、文件名、接口名或约束描述前后失配

自检的范围应按**修改类型**区分，而不是对所有修改一律原地回读：

**孤立内容增补**（在现有章节补充说明、示例，不涉及共享元素）：

- 原章节局部回读即可
- 检查：新增内容的术语是否与本章节前文一致；是否引入悬空引用或过期描述

**共享元素修改**（接口参数、数据表字段、配置键、模块名、共享术语等）：

- 必须显式识别**依赖该元素的下游章节**，并用 `read_file` 回读 1-2 个依赖章节
- 原地回读受自洽性偏见影响，往往无法发现刚引入的跨章节矛盾；跨章节回读是发现逻辑断层的更可靠手段
- 检查：相关章节中对该元素的引用（参数名、表名、接口名、约束描述）是否仍与修改后一致

无论哪种类型，都应检查：

- 是否引入了新的悬空引用、过期描述或与需求基线冲突的表述
- 修改后的内容是否仍能与 Reviewer 的原始意图形成闭环

### 4.1.4.3 推荐写入 prompt 的强约束

建议在 Writer prompt 中增加如下约束语义：

- 当任务为“根据 Reviewer 反馈修订文档”时，进入修订模式
- 修订模式下，必须先读取 `/drafts/design.md`
- 修订模式下，必须保留对主需求文件 `/input/<requirement_filename>` 的对照；这属于需求基线读取，不属于应被削减的目录勘查
- 修订模式下，默认不得重新全量勘查 `/input/`
- 修订模式下，默认不得整份重写 `/drafts/design.md`
- 如果反馈只影响少量章节，必须使用 `edit_file`
- 新增内容时，应插入到最接近的现有章节
- 只有在例外条件成立时，才可使用 `write_file`

以上是属于 `writer_prompt.py` 的高层模式约束。细粒度编辑策略（`old_string` 定位容错、按修改类型执行自检等）应收敛到 `tech-doc-revision` skill，不在 prompt 中重复展开，避免两处维护漂移。

### 4.1.4.4 允许 `write_file` 的例外条件

建议仅在以下条件之一满足时，允许修订阶段整份 `write_file`：

1. `/drafts/design.md` 不存在
2. 文档结构需要整体重构，原有章节体系已不适合继续修订
3. 超过半数一级章节都需要重写
4. `edit_file` 已尝试多次失败，且失败原因是无法稳定定位旧文本

### 4.1.4.5 为何 `edit_file` 更稳

修订阶段优先使用 `edit_file` 的收益包括：

- 减少整篇文档重新生成时的上下文压力
- 降低模型因长文本输出而超时或停滞的概率
- 让修改动作与 Reviewer 反馈建立更清晰的对应关系
- 更容易保留已有的正确章节内容
- 更便于后续审计每轮修订影响范围

### 4.1.4.5.1 `edit_file` 的定位容错策略也应显式设计

仅仅要求“先定位章节、优先 `edit_file`”还不够。在工程实践中，大模型经常因为空格、缩进、换行或极小的文字差异，导致 `old_string` 无法命中，进而出现“字符串未找到”或“匹配不唯一”的失败。

因此，这些定位容错规则应主要沉入 `tech-doc-revision` skill，而不是在 `writer_prompt.py` 中展开——`writer_prompt.py` 只保留高层修订模式约束，细粒度编辑策略由 skill 承载，避免两处同时维护导致漂移。具体规则为：

- 当使用 `edit_file` 时，`old_string` 应尽量包含受影响章节的标题行
- `old_string` 不应只截取一句孤立正文，最好同时包含标题下的 1 到 3 行稳定上下文
- 如同一段文字在文件中可能重复出现，应扩大 `old_string` 范围，优先加入章节标题、前后相邻语句或列表项
- 发起 `edit_file` 前，应再次读取目标章节，确认 `old_string` 与当前文件内容完全一致
- 如连续多次因定位失败报错，不应盲目重复尝试，应回退为“重新读取目标章节 -> 扩大上下文 -> 再次编辑”这一流程

一个更稳妥的经验规则是：`old_string` 至少包含“章节标题 + 需要替换的正文片段”，而不是只包含正文片段本身。这样更容易保证定位唯一性，也更符合局部修订的章节化原则。

### 4.1.4.5.2 需要记录的不是“碎片化风险”，而是“非原子整文件重写风险”

对 `edit_file` 的底层风险，文档中更准确的说法不应是“文件碎片化”，而应是“非原子整文件重写风险”。

原因在于：当前工具实现并不是对文件做块级 patch，而是读取整个文件内容、执行字符串替换，再整文件写回。因此，真正需要关注的风险是：

- 如果定位失败，容易导致重复尝试和无效重写
- 如果底层写入不是原子替换，理论上可能出现截断后再写回的窗口
- 如果存在并发写入或多次连续覆盖，可能出现丢更新风险

因此，文档层面的建议应是：

- 优先减少不必要的 `edit_file` 次数，而不是担心“碎片化”
- 每次 `edit_file` 都尽量覆盖一个完整且稳定的局部上下文
- 多次局部修改同一章节时，应优先合并为一次更完整的编辑动作
- 若某一轮需要在大量位置同时修改，可评估是否已满足回退到整份 `write_file` 的例外条件

### 4.1.4.6 `write_todos` 也必须区分初稿模式与修订模式

除正文指令外，任务规划提示也必须与“初稿模式 / 修订模式”的规则保持一致，避免在同一份 prompt 中出现互相拉扯的行动建议。

当前风险点是：如果在 `write_todos` 指引中笼统建议“将勘查 `/input/` 目录结构作为第一个 todo 条目”，那么该建议更适用于初稿模式，却可能在修订模式下继续被模型沿用，从而与“默认不要重新全量勘查 `/input/`”的修订规则形成冲突。

截至当前文档版本，这一问题同样处于“文档已识别、实现未落地”的状态。也就是说，4.1 文档已经给出按模式拆分 `write_todos` 的建议，但 `writer_prompt.py` 中对应段落仍未完成改造，运行时仍保留统一写法。

因此，建议在 Writer prompt 中将任务规划要求改为按模式分别表述：

- 初稿模式：
  - 第一个 todo 可以是“读取主需求文件并勘查 `/input/` 目录结构”
  - 后续再进入参考文件阅读、设计撰写、落盘等条目
- 修订模式：
  - 第一个 todo 应是”读取 Reviewer 反馈、`/drafts/design.md`、主需求文件 `/input/<requirement_filename>`；若存在 `/drafts/qa-supplement.md`，也在同一步读取并作为冲突澄清基线”——主需求文件是修订轮的常驻基线，不属于应被削减的目录勘查
  - 第二个 todo 应是”识别受影响章节，先核对 Reviewer 意见是否与需求基线一致，再制定局部修改计划”
  - 只有当某条反馈需要进一步事实核对时，才追加”读取相关参考文件”的 todo（主需求文件已在第一步覆盖，不重复）

建议将此要求写成显式约束，而不是模糊建议。目标是让模型在修订模式下优先围绕“反馈定位 -> 局部修改”展开任务规划，而不是机械地把“勘查 `/input/`”当作每轮的默认开场动作。

## 4.1.5 `tech-doc-revision` Skill 设计建议

### 4.1.5.1 设计结论

本轮建议将“按 Reviewer 反馈执行局部修订”设计为一个单独的 skill，而不是继续完全依赖通用写作 skill 或让 Orchestrator 代为整理反馈。

推荐新增 skill：

- `skills/tech-doc-revision/SKILL.md`

同时保留并适度增强现有 skill：

- `skills/tech-doc-writing/SKILL.md`
- `skills/tech-doc-review/SKILL.md`

理由如下：

- “局部修订”是一个稳定且可复用的工作模式，值得独立建模
- 该模式的规则明显不同于初稿撰写规则
- 将其沉入 skill，比让 Orchestrator 临时整理反馈更清晰、更稳定
- Reviewer 负责指出改哪里，Revision skill 负责约束 Writer 如何改

### 4.1.5.2 Skill 的职责边界

`tech-doc-revision` 的职责应聚焦于：

1. 接收 Reviewer 原始反馈
2. 读取当前正式草稿 `/drafts/design.md`
3. 读取主需求文件 `/input/<requirement_filename>` 作为修订基线；若存在 `/drafts/qa-supplement.md`，必须同时读取，并以其为冲突澄清的高优先级来源
4. 先判断 Reviewer 意见是否与需求基线一致，再将反馈映射到具体章节
5. 优先通过 `edit_file` 完成局部修订
6. 仅在例外条件成立时，才允许整份 `write_file`
7. 更新 `/drafts/design.md` 末尾的修订记录

它不应负责：

- 替代 Reviewer 生成问题清单
- 替代 Orchestrator 做任务编排
- 自行决定广泛重扫 `/input/`

### 4.1.5.3 Skill 的核心规则建议

建议 `tech-doc-revision` skill 至少包含以下规则：

- 唯一正式草稿文件是 `/drafts/design.md`
- 禁止创建 `design_v2.md`、`design_v3.md`、`design_final.md` 等变体文件
- 必须先读取 Reviewer 原始反馈、`/drafts/design.md`、主需求文件 `/input/<requirement_filename>`；若存在 `/drafts/qa-supplement.md`，也必须读取，并以其为冲突澄清的高优先级来源
- 必须先定位受影响章节，再开始修改
- 默认不重新全量扫描 `/input/`，但主需求文件读取属于需求基线读取，不属于应被削减的目录勘查
- 只有在某条反馈需要事实核对时，才允许回读直接相关的参考文件（主需求文件除外，它在每轮修订中始终必读）
- 首选 `edit_file`
- 使用 `edit_file` 时，`old_string` 应尽量包含受影响章节的标题行
- `old_string` 应包含足以唯一定位的周边上下文，而不是只截取孤立正文
- 发起 `edit_file` 前，应重新核对目标章节文本，确保 `old_string` 与当前文件完全一致
- 只修改 Reviewer 指出的章节或与其直接相关的局部小节
- 如需新增内容，应插入到最接近的现有章节
- 对同一章节的多处小改动，按"逻辑临近优先"原则决定是否合并：
  - 改动点在同一个二级/三级标题下的相邻逻辑段落内（通常不超过 15-20 行文本）：合并为一次 `edit_file`，减少重复写回
  - 改动点跨段落：即便在同一章节内，也应拆为独立的 `edit_file` 调用；强行合并会拉长 `old_string` 跨度，反而推高匹配失败概率
- 完成局部修订后，按修改类型执行自检：孤立内容增补只需原章节回读；共享元素修改（接口参数、字段、配置键、模块名、共享术语）必须用 `read_file` 回读 1-2 个依赖该元素的下游章节，不得仅做原地检查
- 修订记录必须保持极简，只记录本轮必要变更摘要，不得写成长篇解释
- 仅当满足例外条件时，才允许整份 `write_file`

### 4.1.5.3.1 修订记录格式应极简，避免反向污染上下文

虽然 `tech-doc-revision` 需要更新文档末尾的修订记录，但修订记录本身不应在多轮迭代后膨胀成新的上下文负担。否则经过 5 到 6 轮修订后，模型可能把大量注意力消耗在历史修订备注上，反而削弱对正文和最新 Reviewer 反馈的关注。

因此，建议将修订记录的格式固定为极简条目，例如：

```md
## 修订记录
- 2026-03-25 | `## 5 模块设计 -> ### 5.2 数据库模式` | 补充表结构与字段约束 | 已采纳
- 2026-03-25 | `## 8 CLI 与运行参数 -> ### 8.1 参数定义` | 增加日志级别与递归扫描参数 | 已采纳
```

推荐遵循以下约束：

- 每条记录只保留：日期 | 修订锚点 | 修改要点 | 状态
- 修改要点应是极短摘要，避免写成长段解释
- 对未采纳建议，只写一行简短理由，不展开辩论式说明
- 同一轮对同一章节的多处改动，尽量合并成一条记录
- 修订记录应保持极简，本轮不引入自动裁剪机制；如果后续长轮次任务中确实观察到修订记录造成明显上下文负担，再单独设计归档方案

修订记录的位置维持在文档末尾，理由是：末尾位置对模型注意力干扰最小，模型在处理正文和 Reviewer 反馈时不会被历史修订记录分散。`<details>` 等折叠标签对模型无效（`read_file` 仍会全量读取），且会增加 `edit_file` 定位复杂度，不推荐引入。

修订记录的目标是”保留最小必要审计线索”，而不是替代正文，也不是累积完整讨论历史。极简格式本身就是控制增长的主要手段，无需额外裁剪机制。

> **设计边界说明**：不将”修订记录归档”交给 Orchestrator 执行，原因是 Orchestrator 的职责边界已明确限定为纯编排，`orchestrator_prompt.py` 中明确写有”不要自己编写或修改设计文档内容，始终通过委派 Writer 完成”；引入归档职责会打破这一边界，并引发”谁执行、何时执行、如何避免冲突”等当前未解答的工程问题。

### 4.1.5.4 推荐写入 skill 的例外条件

建议仅在以下条件之一成立时，允许整份 `write_file`：

1. `/drafts/design.md` 不存在
2. 文档结构需要整体重构，原有章节体系已不适合继续修订
3. 超过半数一级章节都需要重写
4. 已尝试多次 `edit_file`，但因无法稳定定位旧文本而失败

同时建议在 skill 中补一条实现说明：当连续出现定位失败时，优先扩大 `old_string` 的上下文重新尝试；若仍失败，再进入整份 `write_file` 的例外判断。这样既能提高局部修订成功率，也能避免无意义的重复编辑。

### 4.1.5.5 与现有 Writing / Review Skill 的关系

新增 `tech-doc-revision` skill 后，建议角色分工如下：

- `tech-doc-writing`：服务于初稿撰写
- `tech-doc-review`：服务于审核与反馈输出
- `tech-doc-revision`：服务于按反馈进行局部修订

这三者互补，不互相替代。

### 4.1.5.6 现有 Skill 的配套增强

即使新增 `tech-doc-revision`，也仍建议增强现有 skill。

对 `tech-doc-writing`：

`tech-doc-writing` 的修订模式基础规则、单文件规则和目录边界已基本落地（见 `skills/tech-doc-writing/SKILL.md`），本轮增强重点不是重复补这些基础约束，而是与新增的 `tech-doc-revision` skill 拉清职责边界：

- `tech-doc-writing` 主要服务初稿撰写
- 细粒度局部修订规则（`old_string` 容错策略、按修改类型自检、逻辑临近合并等）应收敛到 `tech-doc-revision`，避免两个 skill 在修订阶段给出重叠或冲突的指令
- 若 `tech-doc-writing` 的修订模式条目与 `tech-doc-revision` 的规则产生重复，以 `tech-doc-revision` 为准，`tech-doc-writing` 可相应简化或移除冗余内容

对 `tech-doc-review` skill，当前状态是”章节化反馈已部分落地，但仍有缺口”，而不是空白待补：

已落地的部分（无需重复实现）：

- `skills/tech-doc-review/SKILL.md` 的”审核原则”中已要求区分”必须改”和”建议改”
- `skills/tech-doc-review/SKILL.md` 中已有”章节化反馈要求”一节，明确要求：影响章节、问题、修改动作、参考依据

因此，这里的改造目标是**增强**，而不是从零补充。尚待落地的三项缺口：

1. **路径式锚点格式**：当前只要求”标出影响章节”，未规定格式。应增加要求：使用路径式锚点，例如 `## X 模块设计 -> ### X.2 子节`，确保定位唯一、Writer 可直接执行 `edit_file`
2. **存在性预检**：当前 skill 无此流程。应增加：读取需求前先检查 `/input/<requirement_filename>` 是否存在；若不存在，立即向 Orchestrator 返回错误，不得继续审核

这里需要特别强调：本节描述的是“待实施修复项”，而不是“已完成修复项”。截至当前文档版本，相关 skill 文件中的错误路径仍然存在，因此必须在后续代码或 skill 修改阶段显式落地。

## 4.1.6 Reviewer 提示词优化建议

### 4.1.6.1 Reviewer 反馈必须章节化

Reviewer 的输出应从“泛化问题描述”升级为“章节级修订清单”，并且局部修改建议应主要由 Reviewer 提出，而不是由 Orchestrator 代为推断。建议每条必须修改意见至少包含：

- 问题级别：必须修改 / 建议修改
- 影响章节：尽量使用路径式锚点，而不是只写单个章节名；例如 `## 5 模块设计 -> ### 5.2 中间文件关联`
- 问题描述：当前缺什么
- 修改动作：应补充什么
- 参考依据：来自需求、参考文件或当前文档中的哪一处

这里建议进一步标准化“修订锚点”的表达方式。原因是 Markdown 文档中可能存在重名或近似重名章节，例如多个“详细设计”“实现方案”“配置说明”等。如果 Reviewer 只写一个局部标题，Writer 在执行局部修订时仍可能出现定位歧义。

因此，更推荐 Reviewer 使用“路径式锚点”：

- 一级标题 -> 二级标题
- 必要时继续细化到三级标题
- 若当前文档缺少该小节，也应写成“建议新增到 `## x ... -> ### y ...` 之下”

路径式锚点的目标不是把目录写死，而是让修订意见在当前文档结构中有更稳定的落点，从而提升 Writer 执行 `edit_file` 时的准确性。

### 4.1.6.2 推荐反馈模板

建议 Reviewer 在 `REVISE` 时使用类似格式：

```md
### 必须修改 1
- 影响章节：`## 4 解析流程设计 -> ### 4.3 SAS 代码解析器`
- 问题：未说明 `def_Churn.sas` 和 `def_train.sas` 的解析流程
- 修改动作：补充定义文件识别规则、字段提取方式、输出结构
- 参考依据：`/input/sas/def_Churn.sas`、需求第 X 条
```

```md
### 建议修改 1
- 影响章节：`## 8 CLI 与运行参数 -> ### 8.1 参数定义`
- 问题：CLI 参数说明可进一步补全
- 修改动作：补充递归扫描、文件过滤、日志级别等参数定义
- 参考依据：Reviewer 完整性检查
```

如果当前文档尚不存在目标小节，也建议 Reviewer 显式写成类似格式：

```md
### 必须修改 2
- 影响章节：建议新增到 `## 5 模块设计 -> ### 5.4 缓存策略`
- 问题：当前文档未定义缓存命中与失效规则
- 修改动作：新增缓存键、过期策略、失效触发条件
- 参考依据：需求第 X 条、现有配置约束
```

### 4.1.6.3 Reviewer 的判定逻辑也要更稳定

建议 Reviewer 在 REVISE 判定时，只把真正阻碍落地实现的问题纳入“必须修改”。其余内容归入“建议修改”，避免 Writer 每轮被迫大面积修改。

### 4.1.6.4 Reviewer 应直接支持局部修订

Reviewer 的反馈应当直接服务于 Writer 的局部修订动作。为此，Reviewer 在提出“必须修改项”时，应尽量让每一项都具备以下特征：

- Writer 看到后可以直接定位到某个章节
- Writer 能判断这是补充、澄清、替换还是新增局部小节
- Writer 能基于该意见直接执行 `edit_file`

换言之，Reviewer 不应只指出“问题存在”，还应尽量指出“应改到哪里、怎样改”。

## 4.1.7 Orchestrator 委派策略优化建议

### 4.1.7.1 初稿委派

初稿委派给 Writer 时，允许使用相对宽泛的任务描述，例如：

- 根据需求撰写技术设计文档
- 先勘查 `/input/`
- 按需读取参考文件
- 输出到 `/drafts/design.md`

### 4.1.7.2 修订委派

修订委派给 Writer 时，不应让 Orchestrator 以“结构化概括”替代 Reviewer 原始反馈。这里必须尊重已有设计原则：向 Writer 传递反馈时，应完整保留 Reviewer 的具体意见，避免转述损失信息。

截至当前文档版本，这一条处于”已部分落地但存在冲突实现”的状态。具体而言：

- `orchestrator_prompt.py` 中已写入”向 Writer 传递反馈时，完整保留 Reviewer 的具体意见，不要过度概括”，这是新规则，且已落地。
- `orchestrator_prompt.py` 中仍保留”将审核反馈整理为结构化修订任务后传递给 Writer”，这是旧写法，尚未清理。
- `orchestrator_prompt.py` 中又要求”修订轮应尽量将 Reviewer 反馈整理成受影响章节 + 修改动作的结构化清单”，等于回到了旧方向。

三处指令并存且互相矛盾，模型在运行时面对的是冲突信号，而不是清晰的单一规则。因此，问题不是”未落地”，而是”旧新规则并存、尚未清理冲突”。

**修复策略：删除旧指令，不尝试合并改写。**

对于这类存量冲突，应采取”先删除旧指令、再保留唯一新规则”的策略，而不是尝试将新旧指令合并改写成一段折中表述。原因在于：合并产生的模糊语境依然会在 corner case 下触发旧行为——模型在收到冲突信号时，倾向于退化为”倒向更具体的旧指令”，而不是执行抽象的新原则。因此，唯一可靠的修复方式是彻底清除旧指令，只保留单一明确的规则。

因此，本节建议明确改为：

- Orchestrator 必须完整保留 Reviewer 原始反馈
- Orchestrator 不应删减、改写、压缩或重新解释 Reviewer 的核心意见
- Writer 的局部修订范围应主要由 Reviewer 的章节化反馈和 `tech-doc-revision` skill 共同决定

如果需要补充修订导航，Orchestrator 只能做**附加式包装**，不能做**替代式概括**。

### 4.1.7.3 可接受的附加式包装

在不破坏原始反馈的前提下，Orchestrator 可以附加以下辅助信息：

- 修订目标文件：`/drafts/design.md`
- 工作模式提醒：按 Reviewer 反馈修订
- 修改方式提醒：优先 `edit_file`
- 范围提醒：默认不要重新全量扫描 `/input/`
- 需求基线提醒：修订时必须继续对照主需求文件 `/input/<requirement_filename>`；若存在 `/drafts/qa-supplement.md`，也必须读取，并以其为冲突澄清基线；主需求文件读取不属于应被削减的目录勘查
- 单文件提醒：不要创建 `design_v*.md`

这些附加信息的作用是”提醒执行方式”，不是”重写 Reviewer 意图”。

### 4.1.7.4 推荐的修订任务表达方式

建议 Orchestrator 在修订轮使用类似措辞：

```md
请根据以下 Reviewer 原始反馈修订 `/drafts/design.md`。

要求：
- 完整阅读并保留 Reviewer 的原始意见
- 唯一正式草稿文件是 `/drafts/design.md`
- 优先使用 `edit_file`
- 只修改 Reviewer 指出的受影响章节及其直接相关的小节
- 默认不要重新全量扫描 `/input/`
- 修订时必须继续对照主需求文件 `/input/<requirement_filename>`；若存在 `/drafts/qa-supplement.md`，也必须读取，并以其为冲突澄清基线；主需求文件读取不属于应被削减的目录勘查
- 禁止创建 `design_v2.md`、`design_v3.md`、`design_final.md` 等变体文件

以下为 Reviewer 原始反馈全文：
<原文照贴>
```

### 4.1.7.5 与旧设计原则的兼容说明

本节建议并不是推翻“完整保留 Reviewer 具体意见，不要过度概括”的原有原则，而是进一步明确：

- 原始反馈是权威依据
- Orchestrator 只做编排，不做二次解释
- 局部修订建议主要由 Reviewer 提出
- Writer 的修订动作主要由 `tech-doc-revision` skill 规范

这样既能保住原始信息完整性，也能避免 Orchestrator 职责膨胀。

## 4.1.8 文档结构稳定性建议

为了提高 `edit_file` 的成功率，建议保持文档结构在多轮修订中的相对稳定，但不应将目录完全写死，更不应把某个具体项目的章节结构上升为通用模板。

本节的核心建议不是“固定章节名称”，而是“保持章节锚点稳定”。也就是说：

- 在同一个项目内，已经形成的一级、二级标题应尽量延续使用，避免每轮随意改名或重排
- 当 Reviewer 指出问题时，应优先定位到现有章节并在原位置补充、澄清或替换内容
- 只有在现有结构明显不适合承载新内容时，才新增局部小节或调整章节结构
- 通用 writing skill 仍应保留通用技术设计文档骨架，不应被某个项目的特定目录替代

保持结构稳定的价值在于：

- Reviewer 更容易点名具体章节
- Writer 更容易执行局部 `edit_file`
- 后续多轮修订时，章节引用和修订记录更稳定

- Orchestrator：`qwen3-max`，`thinking disabled`
- Writer：`kimi-k2.5`，`thinking enabled`
- Reviewer：`glm-5`，`thinking disabled`

### 4.1.9.1 配置意图

- Orchestrator 的核心职责是编排，不需要高强度思考链
- Writer 负责生成和修订长文档，保留 thinking 有助于复杂内容组织
- Reviewer 的核心职责是判定和指出缺口，先关闭 thinking 换取更稳的工具调用和更短延迟

### 4.1.9.2 thinking 与 tools 的关系说明

需要明确的是：

- thinking 并不等于不能调用 tools
- 但 thinking 会增加请求链路复杂度、输出时延和状态管理压力
- 在长文档、多轮修订、工具密集调用场景下，这种复杂度会放大

因此本轮策略不是“一刀切关闭 thinking”，而是将 thinking 保留给最需要它的 Writer，同时尽量通过 prompt 和流程约束降低其副作用。

### 4.1.9.3 模型组合应补充“降级 / 切换逻辑”说明

当前文档给出的 `qwen3-max / kimi-k2.5 / glm-5` 组合，可以作为默认推荐配置，但不应被理解为“所有场景下永远固定不变”。对于长文档、多轮修订、`edit_file` 密集调用的链路，更稳妥的设计是同时定义“默认组合”和“异常情况下的降级策略”。

建议至少区分两级处理：

1. 一级降级：不切模型，只切修订策略
   - 当 Writer 在局部修订中连续出现 `edit_file` 定位失败时，优先不要立即切模型
   - 先执行“重新读取目标章节 -> 扩大 `old_string` 上下文 -> 合并多处局部修改 -> 再次尝试 `edit_file`”
   - 如果仍失败，再按既定例外条件允许回退到整份 `write_file`

2. 二级降级：切换 Writer 执行模式或备用模型
   - 如果某个 Writer 模型在长文档局部修订上持续表现出不稳定，例如连续多轮出现定位失败、过度整稿重写、或工具调用链路异常
   - 则可以考虑让 Orchestrator 进入“降级委派模式”，明确要求本轮改用更保守的写法，或者在未来实现中切换到备用 Writer 模型

这里需要特别说明：截至当前文档版本，项目实现层仍按 `config/agents.yaml` 在启动时固定加载模型，尚未支持运行中的热切换。因此，文档中的“模型切换”目前应被视为未来扩展设计，而不是现有能力。

基于当前实现能力，文档更推荐先落地以下可执行策略：

- Orchestrator 在修订轮观察到连续定位失败时，优先要求 Writer 回到更保守的局部修订流程
- 满足例外条件时，允许本轮明确回退到 `write_file`
- 将“连续定位失败次数”“是否出现异常整稿重写”“是否出现 tool 调用异常”作为后续是否引入备用 Writer 模型的判断信号

如果未来实现运行时模型切换机制，则建议补充一个简单而明确的策略，例如：

- 默认 Writer：`kimi-k2.5`
- 降级触发条件：连续 N 次 `edit_file` 定位失败，且扩大上下文后仍失败
- 降级动作：切换到更偏格式稳定性的备用 Writer 模型，或关闭 thinking 后重试一轮

文档层面的目标不是立即把模型切换做复杂，而是先把“固定推荐组合”升级为“默认组合 + 异常降级策略”的设计思路。

## 4.1.11 结论

本次优化的核心不是单纯换模型，而是将“文档修订”从大模型自由发挥，收束为一套更稳定的工程化流程：

- 首轮勘查，修订轮收敛
- Reviewer 章节化反馈
- Reviewer 直接给出局部修改建议
- Writer 局部修订优先
- `tech-doc-revision` skill 约束修订动作
- Orchestrator 完整转发 Reviewer 原文并负责编排
- 明确目录边界
- 模型组合服务于稳定性和质量平衡

只有当 `prompt + skill + orchestration + 模型配置` 同时配套时，`edit_file` 优先策略才会真正落地并带来稳定收益。
