# 需求澄清与人机协作（HIL）扩展设计

> 本文档描述在现有 Writer-Reviewer 多 Agent 系统上叠加两类人机协作能力的扩展方案：
>
> - **第一类 HIL（需求澄清）**：Orchestrator 读取需求后识别出关键歧义，主动向用户提问，
>   待补充说明后继续执行，产出更贴合真实意图的设计文档。
> - **第二类 HIL（超限确认）**：Writer-Reviewer 迭代轮次达到上限时，系统暂停并询问用户
>   是否继续，由用户决策而非程序强制截断。
>
> 两类 HIL 共用同一个 `--interactive` 开关和底层 interrupt/resume 机制，
> 均通过 Orchestrator 主动调用专用工具触发中断。

---

## 1. 功能概述

### 1.1 第一类 HIL：需求澄清

当前系统完全依赖用户在需求文件中写清所有上下文。如果需求存在歧义（如边界不清、术语含义不明、缺少关键约束），Agent 只能自行假设，可能导致设计文档与用户预期偏差较大，需要多轮人工修订才能收敛。

引入需求澄清流程后：

1. Orchestrator 读取需求后，主动识别关键歧义
2. 向用户提出不超过 3 个最重要的问题
3. 用户在终端补充回答
4. 回答被持久化为补充文档，Writer 和 Reviewer 均可读取
5. 后续流程与现有逻辑完全一致

### 1.2 第二类 HIL：超限确认

当前系统对 Writer-Reviewer 最大迭代轮次的处理是：达到上限后 Orchestrator 输出"当前版本 + 最后一轮审核意见"并退出。这是一种**程序强制截断**，用户无从干预。

引入超限确认后：

1. Orchestrator 在达到最大轮次时，暂停并向用户展示当前迭代情况
2. 用户决策：继续（尽量重置计数、再给约一轮完整配额，Reviewer 提前 ACCEPT 可提前退出）或结束（以当前版本作为最终输出）
3. 选择继续时，Orchestrator 重新进入迭代循环
4. 选择结束时，Orchestrator 按正常结束流程输出文件名并收尾

这让"最大轮次"从硬截断变为**用户可感知的软边界**。

### 1.3 共同设计原则

- **非侵入**：通过 `--interactive` 参数开启，默认关闭，不影响现有批处理和测试流程
- **Orchestrator 主导**：两类中断均由 Orchestrator 调用专用工具发起，main.py 不做外部监控
- **工具对称**：`ask_user` 用于需求澄清，`confirm_continue` 用于超限确认，结构完全对称
- **第一类：落地为文件**：Q&A 写入 `/drafts/qa-supplement.md`，确保 Agent 间信息可靠传递
- **不修改原始需求**：`/input/` 目录保持只读，补充信息放入 `/drafts/`

## 2. 架构设计

### 2.1 中断发起方：Orchestrator，不是 Writer

HIL 断点设置在 **Orchestrator** 层，而不是 Writer 或 Reviewer。

原因：

| 维度 | Orchestrator 发起（推荐） | Writer 发起（不推荐） |
|------|--------------------------|----------------------|
| **时机** | 读需求后、委派 Writer 前，一次澄清覆盖全流程 | 写作中途发现问题，但上下文已部分展开 |
| **可靠性** | interrupt 直接冒泡到 `main.py`，路径确定 | 子代理 interrupt 需穿透 `task` 工具，SDK 未明确文档化此行为 |
| **职责边界** | Orchestrator = 项目经理，澄清需求是其本职 | Writer = 撰写者，不应承担需求澄清责任 |
| **Agent 感知** | Writer/Reviewer 透明，只需读补充文档 | 需要修改 Writer/Reviewer 的提示词和工具配置 |

### 2.2 Q&A 落地方式：写入 `/drafts/qa-supplement.md`

当 Orchestrator 通过 `task` 工具委派 Writer 时，传递给 Writer 的只有 `task(description=...)` 里的文字——Orchestrator 与用户的对话历史**不会自动注入**给 Writer。因此 Q&A 内容必须落地为文件。

选择 `/drafts/qa-supplement.md` 而非修改原始需求文件，理由：

- `/input/` 设计为用户只读目录，程序运行期间不应写入
- `/drafts/` 是 Agent 工作区，所有中间产物的统一存放位置
- Writer 和 Reviewer 已被配置为可读取 `/drafts/` 下的任意文件

`qa-supplement.md` 文件格式：

```markdown
## 需求澄清记录

<!-- 生成时间：{timestamp}，需求文件：{requirement_filename} -->

> 以下问答由 Orchestrator 在分析需求后向用户提问，用户回答已确认。
> 本文档为用户对需求的最新澄清，权威性高于原始需求文件；
> 若与原始需求存在冲突，必须以本文档为准。设计文档必须覆盖此处所有约束。

---

**Q1: 数据血缘分析的粒度要到字段级别还是表级别？**

A1: 需要到字段级别，尤其关注派生字段的来源追踪。

---

**Q2: 是否需要支持增量分析？**

A2: 本期不需要，全量分析即可。
```

## 4. 实现方案

### 4.1 新增工具：`ask_user` 与 `confirm_continue`

在 `src/tools/hil.py`（新建）中定义两个工具，均只传给 **Orchestrator**，不传给 Writer/Reviewer：

```python
from langchain_core.tools import tool
from langgraph.types import interrupt


@tool
def ask_user(questions: str) -> str:
    """当需求存在关键歧义、需要用户补充说明时调用。
    questions: 问题列表，必须按 'Q1: 问题一\nQ2: 问题二' 格式编写（分隔符请用英文冒号），
    最多 3 个问题。_run_with_hil() 用正则 r'^Q(\d+)[:.：、]\s*(.+)$' 兼容英文冒号、
    全角冒号、句点、顿号等常见变体，并校验三个条件：编号从 1 开始、连续递增（不允许
    Q1/Q3 跳号）、数量不超过 3。三个条件全部满足时逐条收集 A1/A2/A3（Qn 与 An
    严格对应）；任一不满足则降级为自由文本输入并打印 WARNING。
    调用后程序将暂停并等待用户在终端输入回答。
    """
    user_answer = interrupt({"questions": questions})
    return (
        f"需求澄清完成。\n\n"
        f"【问题】\n{questions}\n\n"
        f"【用户回答】\n{user_answer}"
    )


@tool
def confirm_continue(status: str) -> str:
    """当 Writer-Reviewer 迭代达到最大轮次时调用，询问用户是否继续。
    status: 当前迭代情况的简要说明，例如"已完成 3 轮迭代，Reviewer 仍返回 REVISE"。
    用户回答 yes 则继续迭代，回答 no 则以当前版本作为最终输出。
    """
    decision = interrupt({"status": status})
    normalized = str(decision).strip().lower()
    if normalized in ("yes", "y", "继续", "是"):
        return "用户选择继续迭代。请从第 1 轮重新开始计数（进入续跑阶段），再给约一轮完整配额，Reviewer 提前 ACCEPT 则提前退出。"
    return "用户选择结束迭代，以当前版本作为最终输出，请进入步骤 6。"
```

> **为什么是两个独立工具而不是一个通用工具？**
> 语义明确：`ask_user` 的中断界面显示问答输入框，`confirm_continue` 的中断界面显示
> yes/no 选项。`_run_with_hil()` 通过 `interrupt_value` 中的 key（`"questions"` vs `"status"`）
> 区分中断类型，呈现不同的交互界面。

### 4.2 Orchestrator `create_deep_agent()` 参数变更

```python
from langgraph.checkpoint.memory import MemorySaver
from src.tools.hil import ask_user, confirm_continue

# 每个工具独立按对应开关注入，两者互不影响
hil_tools = []
if config.hil_clarify:
    hil_tools.append(ask_user)
if config.hil_confirm:
    hil_tools.append(confirm_continue)

interactive = bool(hil_tools)  # 至少一类 HIL 开启时才需要 checkpointer

orchestrator = create_deep_agent(
    model=...,
    tools=hil_tools,
    system_prompt=build_orchestrator_prompt(...),
    subagents=[writer_subagent, reviewer_subagent],
    backend=FilesystemBackend(root_dir=".", virtual_mode=True),
    **({"checkpointer": MemorySaver()} if interactive else {}),  # 非交互模式零额外开销
    middleware=[LoggingMiddleware("orchestrator")],

```

> 避免 `MemorySaver` 在每个图节点序列化完整状态带来的不必要开销。

### 4.3 Orchestrator 提示词补充（`orchestrator_prompt.py`）

在"工作流程"中新增两段规则：

**步骤 0（需求澄清，插入现有步骤 1 之前）**：

```
需求澄清（ask_user 工具可用时执行）：
0. 在委派 Writer 之前，先读取需求文件和 /input/ 下的参考文件（ls 浏览，按需阅读）。
   在判断是否存在冲突之前，必须对需求中提及的关键字段名、表名、逻辑名称在 /input/
   文件中进行至少一次 grep 检索验证，不得仅凭阅读印象断言"无冲突"。
   完成检索后，判断是否存在以下任一情况，若存在则必须调用 ask_user 工具提问：

   必须提问的情况（高召回率触发条件）：
   a) 参考文件与需求文件存在明确冲突——例如：需求文件说"A 表只有 3 个字段"
      但实际代码显示 A 表有 7 个字段；此时必须问用户"以哪个为准"，不得自行假设
   b) 功能边界不清：无法判断该做 A 还是 B，且两者的设计差异显著
   c) 关键约束缺失：缺少性能要求、数据量级、集成目标等直接影响架构决策的信息

   可跳过的情况：
   - 参考文件与需求一致，或差异属于实现细节不影响设计方向
   - 需求已足够清晰，没有上述任何一种情况

   提问规则：最多 3 个问题，合并为一次 ask_user 调用。只问真正影响设计方向的问题。
   问题必须按以下格式编写，每个问题独占一行，分隔符必须使用英文冒号（:）：
   Q1: 问题一
   Q2: 问题二
   Q3: 问题三（如有）

   收到用户回答后：
   - 将问题和回答整理为 Markdown 格式，写入 /drafts/qa-supplement.md
   - 委派 Writer 和 Reviewer 时，告知其阅读 /drafts/qa-supplement.md（其内容与需求等效）
```

**步骤 5 修改（超限处理，替换现有步骤 5）**：

```
5. 达到最大轮次（{max_iterations} 轮）时：
   - 如果工具列表中存在 confirm_continue，使用该工具告知用户当前迭代情况
     （例如："已完成 {max_iterations} 轮迭代，Reviewer 最后一轮返回 REVISE，
     主要问题：XXX。是否重置计数、再给约一轮完整配额继续迭代？"）
     - 用户回答 yes：尽量从第 1 轮重新开始计数，再给约一轮完整配额，Reviewer 提前 ACCEPT 则提前退出
     - 用户回答 no：直接进入步骤 6（输出文件名）和步骤 7（返回摘要）
   - 如果工具列表中不存在 confirm_continue（非交互模式），输出当前版本 + 最后一轮审核意见后退出
```

> **设计说明**：轮次计数由 Orchestrator LLM 在 context 中维护，"重置计数、再给约一轮配额"
> 是对模型的自然语言引导，而非程序变量归零。这是有意为之的设计选择——保留 LLM 的弹性，
> 允许模型根据上下文判断实际还需要几轮，而不强制归零到固定值。
> 实测中模型的实际行为可能与预期略有偏差，属于可接受的正常范围。

### 4.5 `main.py` 执行流变更

新增 `--interactive` / `-i` 参数。执行路径根据 `config` 的最终值（已合并 `-i` 和 YAML 配置）决定走 `invoke()` 还是支持中断的流式循环：

```python
# 新增 CLI 参数
parser.add_argument("-i", "--interactive", action="store_true",
                    help="开启交互模式：Agent 可在分析需求后向用户提问")

# main() 中的执行逻辑
thread_config = {"configurable": {"thread_id": str(uuid.uuid4())}}

# -i 已在前面将 config.hil_clarify/hil_confirm 置为 True，此处统一用 config 判断
interactive = config.hil_clarify or config.hil_confirm

if interactive:
    # 流式执行，处理可能出现的中断
    result = _run_with_hil(agent, initial_messages, thread_config)
else:
    # 原有同步调用，无中断
    result = agent.invoke({"messages": initial_messages}, config=thread_config)
```

`_run_with_hil()` 辅助函数（处理两类中断）：

```python
def _run_with_hil(agent, initial_messages: list, thread_config: dict) -> dict:
    """执行 Agent，处理 ask_user（需求澄清）和 confirm_continue（超限确认）两类中断。"""
    from langgraph.types import Command

    payload = {"messages": initial_messages}
    result = None

    while True:
        interrupted = False
        gen = agent.stream(payload, config=thread_config)
        try:
            for event in gen:
                interrupts = event.get("__interrupt__")
                if not interrupts:
                    continue

                interrupted = True
                interrupt_value = interrupts[0].value  # dict，由 interrupt() 传入的参数构成

                # ── 第一类：需求澄清 ──────────────────────────────────────────
                if "questions" in interrupt_value:
                    import re
                    questions = interrupt_value.get("questions", "（Agent 未提供具体问题）")

                    # 提取 Qn: 行，同时捕获编号用于连续性校验
                    # 分隔符兼容英文冒号、全角冒号、句点、顿号，应对 LLM 输出变体
                    q_pattern = re.compile(r'^Q(\d+)[:.：、]\s*(.+)$')
                    q_matches = [
                        (int(m.group(1)), l.strip())
                        for l in questions.splitlines()
                        if (m := q_pattern.match(l.strip()))
                    ]
                    # 校验三个条件：编号从 1 开始、连续递增、不超过 3 个
                    actual_nums = [num for num, _ in q_matches]
                    expected_nums = list(range(1, len(q_matches) + 1))
                    protocol_valid = (
                        2 <= len(q_matches) <= 3 and
                        actual_nums == expected_nums
                    )

                    _EXIT_CMDS = {"quit", "exit"}   # 用户可随时输入退出

                    logger.info("Agent 有需求澄清问题，请逐条回答（输入 quit 可终止程序）：")
                    print(f"\n{'─' * 60}")
                    print(questions)
                    print('─' * 60)

                    if protocol_valid:
                        # 多问题且协议合法：逐条收集，Qn 与 An 严格对应
                        print("（请逐条回答，每条输入完成后按回车；输入 quit 终止）")
                        answers = []
                        for num, _ in q_matches:
                            print(f"A{num}：", end="", flush=True)
                            ans = input().strip()
                            if ans.lower() in _EXIT_CMDS:
                                logger.info("用户主动退出，程序终止")
                                raise SystemExit(0)
                            while not ans:          # 不允许空答案
                                ans = input().strip()
                                if ans.lower() in _EXIT_CMDS:
                                    logger.info("用户主动退出，程序终止")
                                    raise SystemExit(0)
                            answers.append(f"A{num}：{ans}")
                        user_answer = "\n".join(answers)
                    else:
                        # 单问题、编号不连续、超过 3 个：降级为自由文本
                        if len(q_matches) > 3:
                            logger.warning("问题数量超过上限 3 个，降级为自由文本回答")
                        elif actual_nums != expected_nums:
                            logger.warning(
                                "问题编号不连续或不从 1 开始（实际: %s），降级为自由文本回答",
                                actual_nums,
                            )
                        lines = []
                        while True:
                            line = input()
                            if line.lower() in _EXIT_CMDS:
                                logger.info("用户主动退出，程序终止")
                                raise SystemExit(0)
                            if line == "" and lines:
                                break
                            if line:
                                lines.append(line)
                        user_answer = "\n".join(lines)

                    payload = Command(resume=user_answer)

                # ── 第二类：超限确认 ──────────────────────────────────────────
                elif "status" in interrupt_value:
                    status = interrupt_value.get("status", "迭代已达上限")
                    logger.info("迭代轮次已达上限，等待用户决策：")
                    print(f"\n{'─' * 60}")
                    print(f"[迭代超限] {status}")
                    print("是否重置计数、再给约一轮完整配额继续迭代？[yes/no/quit]")
                    print('─' * 60)
                    while True:
                        choice = input().strip().lower()
                        if choice in _EXIT_CMDS:
                            logger.info("用户主动退出，程序终止")
                            raise SystemExit(0)
                        if choice:                  # 不接受空输入
                            break
                        print("请输入 yes 或 no：", end="", flush=True)
                    user_answer = "yes" if choice in ("yes", "y", "继续", "是") else "no"
                    if user_answer == "yes":
                        logger.info("[HIL] 用户授权继续，尽量重置计数，再给约一轮完整配额")
                    else:
                        logger.info("[HIL] 用户选择结束迭代，以当前版本作为最终输出")
                    payload = Command(resume=user_answer)

                # ── 未知中断类型：安全兜底 ────────────────────────────────────
                else:
                    logger.warning("收到未知类型的 HIL 中断，自动以空字符串恢复")
                    payload = Command(resume="")

                break  # 退出本轮 stream，用新 payload 重新进入 while
        finally:
            gen.close()  # 确保 generator 在 break 或异常时均被正确关闭

        if not interrupted:
            # 正常结束：流已耗尽，从 checkpointer 读取最终状态
            # get_state().values 返回与 invoke() 相同结构的 state dict（含 "messages" 列表），
            # 与 main.py 第 13 步从 result["messages"] 反向提取 AIMessage 的逻辑完全兼容
            result = agent.get_state(config=thread_config).values
            break

    return result
```

### 5.1 配置文件开关（`config/agents.yaml`）

两类 HIL 可在 `global` 节点下独立控制，默认均为关闭：

```yaml
global:
  max_iterations: 3
  log_level: "INFO"
  hil_clarify: false     # 需求澄清 HIL（ask_user）
  hil_confirm: false     # 超限确认 HIL（confirm_continue）
```

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `hil_clarify` | bool | `false` | 开启后 Orchestrator 读完需求后可向用户提问 |
| `hil_confirm` | bool | `false` | 开启后迭代达到 `max_iterations` 时询问用户是否继续 |

可按需组合：

```yaml
# 只开启需求澄清，不开启超限确认
hil_clarify: true
hil_confirm: false

# 两类全开
hil_clarify: true
hil_confirm: true
```

## 7. 关键设计决策

### 决策 1：使用 `interrupt()` 而非 `interrupt_on`

LangGraph/deepagents 存在两套 HIL 机制，语义完全不同：

- **`interrupt_on`（工具审批协议）**：由 `HumanInTheLoopMiddleware` 驱动，在工具执行*前*拦截，恢复时需传结构化决策 `{"decisions": [{"type": "approve"}]}`。设计用于"是否允许执行此工具调用"的审批场景（如 delete_file、send_email）。
- **`interrupt()`（通用暂停/恢复）**：LangGraph 原生底层机制，在工具或节点*内部*调用，暂停后通过 `Command(resume=任意值)` 恢复，`interrupt()` 的返回值即为用户传入的内容，工具可基于此构造返回值。

我们的场景是**自由文本问答**（用户回答"需要字段级别分析"），不是对工具调用的审批决策，因此必须使用 `interrupt()`。

关键优势：工具通过正常返回值把"问题 + 用户回答"一并交还给 Agent，Agent 将其作为一等输入纳入推理，进而正确写入 `qa-supplement.md`。若使用 `interrupt_on`，用户回答被审批协议消费，Agent 看不到答案内容，后续逻辑完全失效。

### 决策 2：两个独立工具，而非一个通用 `hil_pause` 工具

若合并为一个工具，`_run_with_hil()` 需要靠解析文本内容来区分"提问"和"确认"两种场景，脆弱且易出错。两个独立工具的语义更清晰，`_run_with_hil()` 直接用 `"questions" in value` 和 `"status" in value` 分支，也便于将来独立扩展（如 `confirm_continue` 加入"跳过并接受当前版本"第三选项）。

### 决策 3：超限确认由 Orchestrator 主动调用，而非 main.py 外部监控

另一种方案是 `main.py` 统计 `task` 工具调用次数，超限时从外部中断流。这种方案的问题：没有优雅的 resume 路径，且 main.py 需要理解迭代业务逻辑，破坏关注点分离。让 Orchestrator 主动调用 `confirm_continue` 则完全一致于现有 interrupt/resume 模式，main.py 不需要了解业务状态。

### 决策 4：非交互模式不引入 checkpointer

`MemorySaver()` 会在每个图节点执行后序列化完整状态，对于无需恢复的单次调用是纯开销。因此 `checkpointer` 仅在 `config.hil_clarify or config.hil_confirm` 为真时传入 `create_deep_agent()`，普通模式保持零额外开销。中断能力完全由工具内部的 `interrupt()` 调用提供，不依赖 `interrupt_on`。

### 决策 5：冲突检测作为"必须提问"的高优先级触发条件

需求澄清的触发逻辑分为两类：
- **软触发**（依赖模型主观判断）：功能边界不清、术语含义不明、约束缺失——这些靠提示词引导，召回率受模型能力影响
- **硬触发**（基于客观比对）：参考文件与需求文件存在明确冲突——这是可以被规则化的，无需主观判断

在代码迁移类任务（如 SAS→Python）中，代码与文档的不一致是最高频的歧义来源。如果 Orchestrator 检测到冲突却自行假设"以代码为准"或"以文档为准"，Writer 和 Reviewer 都会基于错误前提工作，整个迭代循环的产出都将偏离预期，且问题极难在最终 Review 阶段被识别。

因此将"参考文件与需求冲突"列为**必须触发 ask_user 的硬条件**，而非"建议提问"的软条件。

### 决策 6：超限续跑记录在日志而非 `task_counts`

用户授权继续后，`task_counts` 会继续累计后续的 task 委派次数——这是正确行为，统计的是"实际发生了多少次委派"，不应归零或打折。

可观测性的补充通过日志行实现：

```
[INFO] [system] [HIL] 用户授权继续，迭代计数归零，进入下一轮
```

`[HIL]` 前缀在日志文件中可被 `grep "[HIL]"` 精确过滤，既不干扰正常统计，又保留了完整的审计轨迹——事后可以清楚地看到"在第几次委派时发生了 HIL 续跑授权"。

### 决策 7：`qa-supplement.md` 不纳入 `input/` 保护

补充文档放在 `/drafts/` 而不是 `/input/`，因为：
- `/input/` 是用户预置的只读目录，语义上不应由程序写入
- `/drafts/` 每次运行前会备份并清空，Q&A 记录会随 drafts 备份一同保留，可回溯

## 9. 后续优化方向

1. **多轮提问**：支持 Orchestrator 在 Writer 修订过程中再次发起澄清（需要 Reviewer 返回 REVISE + 澄清建议）
2. **Web UI**：将终端 `input()` 替换为 HTTP 回调，支持网页表单填写答案（利用 AG-UI / CopilotKit 协议）
3. **问题质量过滤**：在提示词中加入"如果问题可以通过参考 /input/ 下的附件回答，不要提问"，减少不必要的打扰
4. **审计日志**：将 `qa-supplement.md` 的生成时间戳记录到日志，便于追溯
