# 双阶段审核架构设计

> **版本**: v1.1  
> **日期**: 2026-03-24  
> **状态**: 待审核  
> **基于**: deep-agent-project 现有架构 / deepagents SDK v0.4.12  

---

## 1. 需求整理

### 1.1 背景

当前系统为 **Orchestrator + Writer + Reviewer** 的单循环架构：

```
Orchestrator → [Writer 写初稿] → [Reviewer 审核]
                                  ↓
                        REVISE → [Writer 修订] → [Reviewer 再审]（最多3轮）
                                  ↓
                        ACCEPT 或 达上限 → 输出
```

本次扩展后，审核者命名统一为：

- `reviewer1`：第一阶段审核者
- `reviewer2`：第二阶段审核者

### 1.2 新需求

用户希望将审核阶段扩展为 **两个独立审核者（reviewer1 + reviewer2）**，形成双阶段串联审核。

核心目标不是让 `reviewer2` 接手一个尚未被 `reviewer1` 认可的半成品，而是让 `reviewer2` 作为 **升阶终审**：

1. `reviewer1` 负责首轮完整审核与打磨
2. 只有当 `reviewer1` 明确 `ACCEPT` 且 `reviewer2.enabled=true` 时，才自动进入 `reviewer2`
3. `reviewer2` 负责验证 `reviewer1` 提过的问题是否真的解决，并从更高标准补充审视

### 1.3 核心规则

| # | 规则 | 说明 |
|---|------|------|
| 1 | `reviewer1` 必须启用，`reviewer2` 可配置是否启用 | `reviewer1` 是基础审核阶段，`reviewer2` 是可选升阶终审 |
| 2 | 当 `reviewer2.enabled=true` 时，`reviewer1` 和 `reviewer2` 必须使用**不同模型** | 否则双阶段审核价值有限 |
| 3 | 两个审核阶段的最大循环次数**独立配置** | 默认建议 `reviewer1=3`，`reviewer2=2` |
| 4 | **阶段切换只能由当前 Reviewer 的 `ACCEPT` 触发** | 不允许因为“达到上限”而自动切换到下一个 Reviewer |
| 5 | `reviewer1` 达上限后，HIL 只决定**是否继续 reviewer1** | 不决定是否切换到 `reviewer2` |
| 6 | `reviewer2` 达上限后，HIL 只决定**是否继续 reviewer2** | 不回退到 `reviewer1` |
| 7 | 一旦进入 `reviewer2`，**不得回退**到 `reviewer1` | 避免两个 Reviewer 来回拉扯 |
| 8 | `reviewer2` 的 verdict 文件必须与 `reviewer1` 分开存放 | 避免覆盖 |

### 1.4 新流程总览

```
第1阶段：reviewer1 审核循环
  Writer 写 → reviewer1 审 → REVISE → Writer 改 → reviewer1 再审 → ...（最多 reviewer1_max 轮）
                                                        ↓
                                              ACCEPT → 若 reviewer2.enabled=true，则进入第2阶段；否则直接输出
                                              达上限 → HIL：是否继续 reviewer1？
                                                        ├─ yes：继续 reviewer1
                                                        └─ no：结束任务，输出当前版本

第2阶段：reviewer2 审核循环
  Writer 改 → reviewer2 审 → REVISE → Writer 改 → reviewer2 再审 → ...（最多 reviewer2_max 轮）
                                                        ↓
                                              ACCEPT → 最终输出
                                              达上限 → HIL：是否继续 reviewer2？
                                                        ├─ yes：继续 reviewer2
                                                        └─ no：结束任务，输出当前版本
```

## 2. 解决方案

### 2.1 整体架构

```
Orchestrator（编排器）
    │
    ├── Writer（子代理）──────→ /drafts/design.md
    │
    ├── reviewer1（子代理）──→ /drafts/review-verdict.json
    │
    └── reviewer2（子代理，可选）──→ /drafts/review-verdict-stage2.json
```

### 2.2 与现有架构的差异

| 维度 | 现有架构 | 目标架构 |
|------|----------|----------|
| Reviewer 数量 | 1 个 | 2 个（`reviewer1`、`reviewer2`） |
| 审核阶段 | 单阶段循环 | 双阶段串联 |
| reviewer2 启用方式 | 不涉及 | `reviewer2.enabled` 控制是否启用第二阶段 |
| 阶段切换条件 | 不涉及 | **仅当前阶段 `ACCEPT` 才切换** |
| 循环次数控制 | 全局 `max_iterations` | 阶段级 `max_reviewer_iterations` 为主，全局 `max_iterations` 仅作兜底 |
| HIL 触发时机 | 总循环达上限 | `reviewer1` / `reviewer2` / 全局兜底上限 |
| verdict 文件 | 1 个 | 2 个（分开存储） |

### 2.3 双阶段工作流详解

#### 第1阶段：reviewer1 审核循环

1. Orchestrator 委派 **Writer** 撰写初稿（或修订稿）
2. Orchestrator 委派 **reviewer1** 审核 `/drafts/design.md`
3. Orchestrator 解析 `reviewer1` 的 `VERDICT`
4. 分支规则：
   - **ACCEPT**：
     - 若 `reviewer2.enabled=true`：进入第2阶段
     - 若 `reviewer2.enabled=false`：直接进入最终输出阶段
   - **REVISE** 且当前轮次 < `reviewer1_max`：将反馈发给 Writer 修订，继续 `reviewer1` 循环
   - **REVISE** 且当前轮次 = `reviewer1_max`：调用 `confirm_continue`
     - 用户回答 `yes`：将 `reviewer1` 当前阶段计数清零，并重新获得完整的 `reviewer1_max` 配额，然后继续 `reviewer1`
     - 用户回答 `no`：结束任务，输出当前版本，不进入 `reviewer2`

补充说明：

- 若 `reviewer1` `ACCEPT` 且进入第2阶段，则 `reviewer2` 的 `max_reviewer_iterations` **从 1 重新开始计数**
- `reviewer2` 的阶段计数与 `reviewer1` 已消耗轮次完全独立，不做继承、不做扣减

#### 第2阶段：reviewer2 审核循环

5. 仅当 `reviewer2.enabled=true` 且 `reviewer1` 返回 **ACCEPT** 后，Orchestrator 才委派 **reviewer2** 审核当前草稿
6. Orchestrator 解析 `reviewer2` 的 `VERDICT`
7. 分支规则：
   - **ACCEPT**：进入最终输出阶段
   - **REVISE** 且当前轮次 < `reviewer2_max`：将反馈发给 Writer 修订，继续 `reviewer2` 循环
   - **REVISE** 且当前轮次 = `reviewer2_max`：调用 `confirm_continue`
     - 用户回答 `yes`：将 `reviewer2` 当前阶段计数清零，并重新获得完整的 `reviewer2_max` 配额，然后继续 `reviewer2`
     - 用户回答 `no`：结束任务，输出当前版本

#### 关键约束

- **阶段切换只能由 `ACCEPT` 触发**
- `reviewer2.enabled=false` 时，系统退化为单 Reviewer 模式：`reviewer1 ACCEPT` 后直接输出
- `reviewer1` 达上限时，不能通过 HIL 直接“跳过”到 `reviewer2`
- 一旦进入 `reviewer2`，不得回退到 `reviewer1`
- Writer 每次修订时，都应收到当前阶段 Reviewer 的完整反馈，不得过度概括

#### 最终输出

8. Orchestrator 建议输出文件名，写入 `/drafts/output-filename.txt`
9. `main.py` 将 `/drafts/design.md` 复制到 `output/` 目录

### 2.4 全局 `max_iterations` 的定位

为兼容现有实现，保留全局 `max_iterations`，但其角色调整为 **系统级安全兜底**，不是主循环控制器。

约定如下：

1. 阶段内循环控制以已启用 Reviewer 的 `max_reviewer_iterations` 为主
2. 全局 `max_iterations` 仅用于防止 Orchestrator 因异常提示词行为进入失控循环
3. 若触发全局 `max_iterations`：
   - 调用 `confirm_continue`
   - `yes`：继续当前阶段，并尽量重置全局计数
   - `no`：结束任务，输出当前版本
4. 配置建议：`max_iterations >= reviewer1_max + reviewer2_max`

补充说明：

- `reviewer1_max` 与 `reviewer2_max` 是两个独立的阶段计数器
- 当流程从第1阶段进入第2阶段时，`reviewer2` 的阶段计数从 1 开始重新计算
- 阶段计数重置不等于全局 `max_iterations` 自动重置；全局计数是否重置，仅由全局超限后的 HIL 续跑决定

### 2.5 HIL 确认语义统一

`confirm_continue` 的 `yes/no` 语义在所有场景下保持一致：

| 用户输入 | 统一语义 |
|----------|----------|
| `yes` | 继续**当前阶段**，并重新给一轮完整配额 |
| `no` | 不再继续当前阶段，直接结束任务并输出当前版本 |

因此：

- 在 `reviewer1` 超限时，`yes` 表示继续 `reviewer1`，**不是**切换到 `reviewer2`
- 在 `reviewer2` 超限时，`yes` 表示继续 `reviewer2`
- 只有当前 Reviewer 明确 `ACCEPT`，才会进入下一阶段或最终输出

这里的“重新给一轮完整配额”含义明确为：

- 将当前阶段计数器清零
- 重新获得该阶段配置值对应的全部轮次
- **不是**仅额外追加 1 轮

例如：

- 若 `reviewer1_max=3`，则用户在 `reviewer1` 超限后选择 `yes`，表示重新获得完整 3 轮 `reviewer1` 配额
- 若 `reviewer2_max=2`，则用户在 `reviewer2` 超限后选择 `yes`，表示重新获得完整 2 轮 `reviewer2` 配额

### 2.6 verdict 文件分离与结构

```
/drafts/review-verdict.json          → reviewer1 的结构化审核结论
/drafts/review-verdict-stage2.json   → reviewer2 的结构化审核结论
```

建议统一为以下结构：

```json
{
  "verdict": "REVISE",
  "summary": "存在 3 项必须修改的问题",
  "issues": [
    {
      "id": "R1-1",
      "severity": "major",
      "description": "缺少数据回滚策略"
    },
    {
      "id": "R1-2",
      "severity": "major",
      "description": "接口异常处理未定义"
    }
  ]
}
```

说明：

- `reviewer1` 和 `reviewer2` 都使用相同 schema，仅输出到不同路径
- `reviewer2` 必须读取 `/drafts/review-verdict.json`，逐项验证 `reviewer1` 提出的问题是否真正被解决
- 若 `reviewer1` 的问题仍未解决，`reviewer2` 必须返回 `REVISE`
- 当 `reviewer2.enabled=false` 时，不产生 `/drafts/review-verdict-stage2.json`
- `issues.id` 应体现问题来源：
  - `R1-*`：来源于 `reviewer1`
  - `R2-*`：来源于 `reviewer2` 在第二阶段新增发现的问题

字段约束建议：

- `verdict`：必填，取值为 `ACCEPT` 或 `REVISE`
- `summary`：必填，用一句话概括本轮审核结论
- `issues`：
  - 当 `verdict=REVISE` 时必填，且至少包含 1 条问题
  - 当 `verdict=ACCEPT` 时可为空数组 `[]`

`issues.id` 编号与继承规则：

- `reviewer1` 首次提出的问题，编号为 `R1-1`、`R1-2`、`R1-3` ...
- `reviewer2` 若确认某个 `reviewer1` 历史问题仍未解决，应沿用原 `R1-*` 编号
- `reviewer2` 新发现的问题，编号为 `R2-1`、`R2-2`、`R2-3` ...
- `reviewer2` 已确认解决的 `R1-*` 问题，不应继续出现在当前 verdict 的 `issues` 列表中

兼容性说明：

- 旧版本 verdict 结构仅包含 `verdict + summary`
- 本次设计将 verdict schema 正式升级为 `verdict + summary + issues`
- 因此需要同步修改 Reviewer 输出要求、Orchestrator 结构化读取逻辑，以及依赖旧 schema 的测试用例

## 5. reviewer2 提示词设计

### 5.1 `stage=2` 与 `stage=1` 的差异

| 维度 | `stage=1` (`reviewer1`) | `stage=2` (`reviewer2`) |
|------|-------------------------|-------------------------|
| 审核目标 | 从零开始全面审核 | 验证 `reviewer1` 问题是否解决 + 更严格审视 |
| 反馈来源 | 基于需求原文与参考文件 | 额外关注 `/drafts/review-verdict.json` |
| 通过条件 | 需求全覆盖、可落地、无歧义、完整 | 在满足上述条件基础上，确认 `reviewer1` 问题已解决 |
| 阶段位置 | 第一阶段 | 第二阶段终审 |

### 5.2 `stage=2` 额外规则

```
【第二阶段审核特别说明】

这是 reviewer2 的审核阶段。只有在 `reviewer2.enabled=true` 且 reviewer1 已经 ACCEPT 后，文档才会进入本阶段。

你的职责是：
1. 读取 /drafts/review-verdict.json，逐项检查 reviewer1 提出的问题是否真正解决
2. 如果 reviewer1 的问题仍未解决，必须返回 REVISE
3. 从更严格的视角审视文档，补充 reviewer1 未覆盖到的问题
4. 你的 REVISE 反馈将继续交给 Writer 修改，但流程仍停留在 reviewer2 阶段
5. 你不能要求流程回退到 reviewer1
```

## 6. Orchestrator 提示词扩展

### 6.1 新增工作流规则

Orchestrator 提示词中必须明确写入以下规则：

```text
多阶段审核流程：

【第1阶段：reviewer1 审核循环】
- 每轮：Writer 写 → reviewer1 审
- reviewer1 ACCEPT：若 reviewer2.enabled=true 则进入第2阶段，否则直接输出
- reviewer1 REVISE：继续停留在第1阶段
- reviewer1 达上限：调用 confirm_continue，yes=继续 reviewer1，no=结束任务输出当前版本

【第2阶段：reviewer2 审核循环】
- 仅在 reviewer2.enabled=true 且 reviewer1 ACCEPT 后进入
- 每轮：Writer 改 → reviewer2 审
- reviewer2 ACCEPT：进入最终输出
- reviewer2 REVISE：继续停留在第2阶段
- reviewer2 达上限：调用 confirm_continue，yes=继续 reviewer2，no=结束任务输出当前版本

【重要约束】
- 阶段切换只能由 ACCEPT 触发
- 不允许因为达到上限而自动切换到下一个 Reviewer
- 一旦进入 reviewer2，不允许回退到 reviewer1
```

### 6.2 HIL 触发场景

`confirm_continue` 的触发场景扩展为三种：

| 场景 | status 文案示例 | yes 的含义 | no 的含义 |
|------|-----------------|------------|-----------|
| `reviewer1` 达上限 | "`reviewer1` 已完成 N 轮审核，仍返回 REVISE。是否继续 `reviewer1` 审核？" | 继续 `reviewer1` | 结束任务并输出当前版本 |
| `reviewer2` 达上限 | "`reviewer2` 已完成 N 轮审核，仍返回 REVISE。是否继续 `reviewer2` 审核？" | 继续 `reviewer2` | 结束任务并输出当前版本 |
| 全局兜底上限 | "已达到最大总迭代轮次，是否继续当前审核阶段？" | 继续当前阶段 | 结束任务并输出当前版本 |

### 6.3 verdict 解析逻辑

Orchestrator 根据当前阶段读取不同 verdict 文件：

```text
第1阶段：读取 /drafts/review-verdict.json
第2阶段：读取 /drafts/review-verdict-stage2.json
```

解析顺序：

1. 优先从 Reviewer 返回文本中查找 `VERDICT: ACCEPT` 或 `VERDICT: REVISE`
2. 若文本解析失败，读取对应阶段 verdict 文件
3. 若仍失败，按 `REVISE` 处理

结构化字段使用规则：

1. 判定当前阶段是否通过时，优先使用：
   - Reviewer 文本中的 `VERDICT`
   - 或 verdict 文件中的 `verdict` 字段
2. 当 `verdict=REVISE` 时：
   - Orchestrator 应尽量读取 `issues`
   - 向 Writer 透传反馈时，优先保留 `issues[].description`
   - 生成 HIL 提示文案中的“主要问题：XXX”时，优先从 `issues` 提取
3. 若 `issues` 缺失但存在 `summary`：
   - 允许以 `summary` 作为降级兜底
   - 但应视为 Reviewer 未完全遵守新 schema
4. 若 `verdict`、`summary`、`issues` 三者均不可用：
   - 按 `REVISE` 处理
   - 并以“结构化审核结论缺失”作为兜底原因

这意味着：

- Orchestrator 的 verdict 解析主逻辑仍以 `verdict` 为准
- 但其结构化反馈消费逻辑需要从“只读 `summary`”扩展为“优先读 `issues`，必要时回退 `summary`”

### 6.4 Writer 在两个阶段的反馈消费规则

Writer 修订时，应始终只处理“**当前阶段、当前轮次 verdict 中仍然开放的问题**”，而不是机械累计历史所有问题。

规则如下：

1. 在 `reviewer1` 阶段：
   - Writer 处理 `reviewer1` 当前 verdict 的全部 `issues`
2. 在 `reviewer2` 阶段：
   - Writer 处理 `reviewer2` 当前 verdict 的全部 `issues`
   - 这些 `issues` 可能同时包含：
     - `R1-*`：`reviewer2` 认定尚未真正解决的历史问题
     - `R2-*`：`reviewer2` 新发现的问题
3. Writer **不能**简单认为“第2阶段只处理 `R2-*`”
4. Writer 也**不需要**机械回头处理所有历史 `R1-*`
   - 只有那些仍出现在 `reviewer2` 当前 `issues` 列表中的 `R1-*`，才代表仍未关闭

换句话说：

- `issues.id` 前缀用于标识**问题来源**
- 当前 verdict 的 `issues` 列表用于标识**当前待处理问题集合**

示例 1：

1. `reviewer1` 提出 `R1-1`、`R1-2`、`R1-3`、`R1-4`、`R1-5`
2. Writer 修订后，`reviewer1` `ACCEPT`
3. `reviewer2` 检查后认为：
   - `R1-1` ~ `R1-5` 均已解决
   - 新发现 `R2-1`、`R2-2`、`R2-3`
4. 此时 `reviewer2` 当前 verdict 的 `issues` 应只包含 `R2-1`、`R2-2`、`R2-3`
5. Writer 在第2阶段只需要处理这 3 个 `R2-*`

示例 2：

1. `reviewer2` 认为 `R1-3` 实际上仍未解决
2. 同时新发现 `R2-1`、`R2-2`
3. 则当前 verdict 的 `issues` 应包含：
   - `R1-3`
   - `R2-1`
   - `R2-2`
4. Writer 在第2阶段需要处理的是这 3 个当前仍开放的问题，而不是所有历史问题，也不是只处理 `R2-*`

## 7. 风险评估

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| Orchestrator 提示词逻辑复杂，LLM 误把“超限”理解为“切换阶段” | 中 | 明确写死“阶段切换只能由 ACCEPT 触发” |
| `reviewer2` 开关关闭时，Orchestrator 仍错误委派 `reviewer2` | 中 | 在 Prompt 与 AgentFactory 两处同时受 `reviewer2.enabled` 控制 |
| `reviewer2` 过早接手未成熟草稿 | 中 | 只允许 `reviewer1 ACCEPT` 后进入 `reviewer2` |
| 两个 Reviewer 来回拉扯，流程震荡 | 中 | 明确“进入 `reviewer2` 后不得回退” |
| 双阶段循环与全局 `max_iterations` 概念混淆 | 中 | 将全局 `max_iterations` 定位为系统兜底，而非主流程控制 |
| verdict schema 已升级，但 Prompt / Orchestrator / 测试仍沿用旧结构 | 中 | 在 Reviewer Prompt、Orchestrator Prompt 和测试中同步升级为新 schema |
| 配置中误将两个 Reviewer 设为相同模型 | 低 | 启动时校验 `(provider, model)` 组合不能完全相同 |

## 8. 测试建议

| 测试用例 | 验证点 |
|----------|--------|
| `reviewer2.enabled=false` | 系统退化为单 Reviewer 模式，`reviewer1 ACCEPT` 后直接输出 |
| `reviewer1` 3 轮内 `ACCEPT` 且 `reviewer2.enabled=true` | 正常进入第2阶段 |
| `reviewer1` 达上限且用户选 `yes` | 继续 `reviewer1`，不进入 `reviewer2` |
| `reviewer1` 达上限且用户选 `no` | 结束任务，输出当前版本，不进入 `reviewer2` |
| `reviewer2` 2 轮内 `ACCEPT` | 进入最终输出 |
| `reviewer2` 达上限且用户选 `yes` | 继续 `reviewer2` |
| `reviewer2` 达上限且用户选 `no` | 结束任务，输出当前版本 |
| `reviewer2` 发现 `reviewer1` 历史问题未解决 | 必须返回 `REVISE` |
| Reviewer 输出新 schema | `REVISE` 时 verdict 文件必须包含 `issues` 数组 |
| reviewer2 问题编号继承 | 未解决的历史问题沿用 `R1-*`，新问题使用 `R2-*` |
| Writer 第2阶段反馈消费 | 处理 reviewer2 当前 `issues` 的全部开放问题，而不是只处理 `R2-*` |
| Orchestrator 读取新 schema | HIL / Writer 反馈优先使用 `issues[].description` |
| 旧 schema 兼容兜底 | 缺失 `issues` 时可降级使用 `summary`，但流程应记录为 schema 不完整 |
| 阶段不可回退 | 进入 `reviewer2` 后不会再次委派 `reviewer1` |
| verdict 文件正确分离 | `reviewer1` 写 `review-verdict.json`，`reviewer2` 写 `review-verdict-stage2.json` |
| 测试更新 | `tests/test_run_with_hil.py` 等依赖旧 schema 的测试夹具和断言需要同步更新 |
| 配置校验 | `reviewer1` 未启用应报错；`reviewer2` 启用但漏配或与 `reviewer1` 模型相同应报错 |
