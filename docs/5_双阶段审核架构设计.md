# 双阶段审核架构设计

> **版本**: v1.2
> **日期**: 2026-03-27
> **状态**: 待审核
> **基于**: deep-agent-project 现有架构 / deepagents SDK v0.4.12

---

## 1. 需求整理

### 1.1 背景

当前系统为 **Orchestrator + Writer + Reviewer** 的单循环架构：

```
Orchestrator → [Writer 写初稿] → [Reviewer 审核]
                                  ↓
                        REVISE → [Writer 修订] → [Reviewer 再审]（最多3轮）
                                  ↓
                        ACCEPT 或 达上限 → 输出
```

本次扩展后，审核者命名统一为：

- `reviewer1`：第一阶段审核者（原 `reviewer`，重命名）
- `reviewer2`：第二阶段审核者（新增，可选）

### 1.2 新需求

用户希望将审核阶段扩展为 **两个独立审核者（reviewer1 + reviewer2）**，形成双阶段串联审核。

核心设计理念：**reviewer1 和 reviewer2 使用不同的 LLM，从不同角度独立审核，互不负责对方的问题**。

1. `reviewer1` 负责首轮完整审核与打磨
2. 只有当 `reviewer1` 明确 `ACCEPT` 且 `reviewer2.enabled=true` 时，才自动进入 `reviewer2`
3. `reviewer2` 基于自己的视角对文档进行独立的终审，不跟踪 `reviewer1` 提过的问题

### 1.3 核心规则

| # | 规则 | 说明 |
|---|------|------|
| 1 | `reviewer1` 必须启用，`reviewer2` 可配置是否启用 | `reviewer1` 是基础审核阶段，`reviewer2` 是可选终审 |
| 2 | 当 `reviewer2.enabled=true` 时，`reviewer1` 和 `reviewer2` 必须使用**不同模型** | 否则双阶段审核价值有限 |
| 3 | 两个审核阶段的最大循环次数**独立配置** | 各自有独立的 `max_reviewer_iterations` |
| 4 | **阶段切换只能由当前 Reviewer 的 `ACCEPT` 触发** | 不允许因为"达到上限"而自动切换到下一个 Reviewer |
| 5 | `reviewer1` 达上限后，HIL 只决定**是否继续 reviewer1** | 不决定是否切换到 `reviewer2` |
| 6 | `reviewer2` 达上限后，HIL 只决定**是否继续 reviewer2** | 不回退到 `reviewer1` |
| 7 | 一旦进入 `reviewer2`，**不得回退**到 `reviewer1` | 避免两个 Reviewer 来回拉扯 |
| 8 | `reviewer2` 的 verdict 文件必须与 `reviewer1` 分开存放 | 避免覆盖 |
| 9 | `reviewer2` 独立审核，不读取 `reviewer1` 的 verdict 文件 | 两者从不同角度评审，互不负责 |

### 1.4 新流程总览

```
第1阶段：reviewer1 审核循环
  Writer 写 → reviewer1 审 → REVISE → Writer 改 → reviewer1 再审 → ...（最多 reviewer1.max_reviewer_iterations 轮）
                                                        ↓
                                              ACCEPT → 若 reviewer2.enabled=true，则进入第2阶段；否则直接输出
                                              达上限 → HIL：是否继续 reviewer1？
                                                        ├─ yes：继续 reviewer1
                                                        └─ no：结束任务，输出当前版本

第2阶段：reviewer2 审核循环
  reviewer2 审 → REVISE → Writer 改 → reviewer2 再审 → ...（最多 reviewer2.max_reviewer_iterations 轮）
                                                        ↓
                                              ACCEPT → 最终输出
                                              达上限 → HIL：是否继续 reviewer2？
                                                        ├─ yes：继续 reviewer2
                                                        └─ no：结束任务，输出当前版本
```

## 2. 解决方案

### 2.1 整体架构

```
Orchestrator（编排器）
    │
    ├── Writer（子代理）──────→ /drafts/design.md
    │
    ├── reviewer1（子代理）──→ /drafts/review-verdict.json
    │
    └── reviewer2（子代理，可选）──→ /drafts/review-verdict-stage2.json
```

### 2.2 与现有架构的差异

| 维度 | 现有架构 | 目标架构 |
|------|----------|----------|
| Reviewer 数量 | 1 个（`reviewer`） | 2 个（`reviewer1`、`reviewer2`） |
| 审核阶段 | 单阶段循环 | 双阶段串联 |
| reviewer2 启用方式 | 不涉及 | `reviewer2.enabled` 控制是否启用第二阶段 |
| 阶段切换条件 | 不涉及 | **仅当前阶段 `ACCEPT` 才切换** |
| 循环次数控制 | 全局 `max_iterations` | 各阶段独立的 `max_reviewer_iterations`，全局 `max_iterations` 仅作兜底 |
| HIL 触发时机 | 总循环达上限 | `reviewer1` / `reviewer2` 各自达上限，以及全局兜底上限 |
| verdict 文件 | 1 个 | 2 个（分开存储） |
| 审核关系 | 不涉及 | reviewer1 和 reviewer2 完全独立，不互相跟踪问题 |
| reviewer1 ACCEPT 的含义 | 不涉及 | 仅表示"当前版本达到进入第二阶段的标准"，不构成后续阶段必须保持的内容基线 |
| 最终验收标准 | 不涉及 | 当前版本通过**最终阶段** Reviewer 的独立终审，满足需求与 skill 标准；不要求同时保持 reviewer1 曾放行的具体方案 |

### 2.3 双阶段工作流详解

#### 第1阶段：reviewer1 审核循环

1. Orchestrator 委派 **Writer** 撰写初稿（或修订稿）
2. Orchestrator 委派 **reviewer1** 审核 `/drafts/design.md`
3. `StageStateMiddleware` 在 reviewer1 返回后更新 `review-state.json`（递增轮次、推进阶段）
4. Orchestrator 读取 `review-state.json` 判断下一步：
   - `current_stage == "reviewer2"`：进入第2阶段（reviewer1 已 ACCEPT，middleware 已推进状态）
   - `current_stage == "done"`：直接进入最终输出阶段
   - `awaiting_confirm_for == "reviewer1"`：调用 `confirm_continue`（middleware 已检测到超限）
   - 其余（`current_stage == "reviewer1"`）：将 reviewer1 的**原始任务返回文本**完整传给 Writer，委派修订，继续循环

#### 第2阶段：reviewer2 审核循环

5. 仅当 `current_stage == "reviewer2"` 时，Orchestrator 才委派 **reviewer2** 审核当前草稿
6. `reviewer2` 从自己的视角独立审核 `/drafts/design.md`，不读取 `reviewer1` 的 verdict 文件
7. `StageStateMiddleware` 在 reviewer2 返回后更新 `review-state.json`
8. Orchestrator 读取 `review-state.json` 判断下一步：
   - `current_stage == "done"`：进入最终输出阶段
   - `awaiting_confirm_for == "reviewer2"`：调用 `confirm_continue`
   - 其余（`current_stage == "reviewer2"`）：将 reviewer2 的**原始任务返回文本**完整传给 Writer，委派修订，继续循环

#### 关键约束

- **阶段切换只能由 `ACCEPT` 触发**
- `reviewer2.enabled=false` 时，系统退化为单 Reviewer 模式：`reviewer1 ACCEPT` 后直接输出
- `reviewer1` 达上限时，不能通过 HIL 直接"跳过"到 `reviewer2`
- 一旦进入 `reviewer2`，不得回退到 `reviewer1`
- Writer 每次修订时，都应收到当前阶段 Reviewer 的完整反馈，不得过度概括
- **`[STAGE_LIMIT_REACHED]` 是系统控制信号，不是审核意见**：收到该前缀时，Orchestrator 唯一允许的下一步是调用 `confirm_continue`，不得解读为 REVISE 或继续派发任务
- **reviewer1 的 ACCEPT 仅是进入第二阶段的前置条件，不构成后续阶段必须保持的冻结基线**：进入 reviewer2 后，Writer 可以修改任何章节（包括 reviewer1 已放行的内容），reviewer2 也可以否定这些内容；最终验收语义为"当前版本通过 reviewer2 的独立终审，满足需求与 skill 标准"，不要求同时讨好两个 reviewer
- **Writer 修订时以当前阶段 Reviewer 的最新反馈为唯一优先依据**：若当前阶段 Reviewer 的意见与历史 Reviewer 的意见存在冲突，Writer 必须遵循当前阶段 Reviewer 的最新指令，不得为了兼顾历史已放行方案而折中修改
- **当 `current_stage == "reviewer2"` 且 reviewer2 返回 REVISE 时，Orchestrator 在委派 Writer 修订时必须附加显式提醒**：你正在接受第二阶段终审；请以 reviewer2 的本轮反馈为唯一优先依据。若其与 reviewer1 在第一阶段提出或接受的方案存在冲突，请忽略这些历史冲突意见，不要折中，不要回退到 reviewer1 已接受的旧方案，直接按 reviewer2 的最新要求修订。

> **实施说明**：以上约束不能仅靠提示词约定来保证。阶段切换、禁止回退、计数追踪均由 §2.7 描述的"最小可验证状态 + 代码守卫"机制强制执行。

#### 最终输出

9. Orchestrator 建议输出文件名，写入 `/drafts/output-filename.txt`
10. `main.py` 将 `/drafts/design.md` 复制到 `output/` 目录

### 2.4 全局 `max_iterations` 的定位

保留全局 `max_iterations`，其角色为 **系统级安全兜底**，不是主循环控制器。

约定如下：

1. 阶段内循环控制以各 Reviewer 的 `max_reviewer_iterations` 为主；`StageStateMiddleware` 负责阶段计数与超限拦截
2. 全局 `max_iterations` 仅用于防止 Orchestrator 因异常提示词行为进入失控循环，是框架层安全网，**不由 `StageStateMiddleware` 计数或触发**；仅在 `confirm_continue=no` 时由其负责把 `current_stage` 落为 `"done"`
3. 配置建议：`max_iterations >= reviewer1.max_reviewer_iterations + reviewer2.max_reviewer_iterations`
4. 若触发全局 `max_iterations`：Orchestrator **直接调用** `confirm_continue`，不写入 `awaiting_confirm_for`
   - `yes`：仅允许继续执行，**不修改** `reviewer1_round` / `reviewer2_round`；阶段计数仍由 `StageStateMiddleware` 按 reviewer 成功返回后累积
   - `no`：`StageStateMiddleware._update_after_hil` 写入 `current_stage="done"`，保持状态文件自洽；Orchestrator 随即结束任务

   > 全局 `max_iterations` 是系统保险丝，不是业务轮次，不应影响阶段计数。两种超限的唯一差异在 `yes` 分支：阶段超限清零当前阶段 round，全局超限不修改任何 round。`no` 分支两者行为一致，均由 `StageStateMiddleware._update_after_hil` 写入 `done`（checkpointer 启用后存在复跑场景，状态文件必须自洽）。

### 2.5 HIL 确认语义

`confirm_continue` 的 `yes/no` 语义**因触发场景不同而有差异**，不存在跨场景的统一语义：

| 触发场景 | `yes` 的含义 | `no` 的含义 |
|----------|-------------|-------------|
| **阶段超限**（reviewer1/reviewer2 达上限） | 继续当前阶段，清零当前阶段 round，给予续跑配额 | 结束任务，输出当前版本；**middleware** 写入 `current_stage="done"` |
| **全局兜底**（`max_iterations` 达上限） | 仅允许继续执行，**不修改**阶段 round | 结束任务，输出当前版本；**middleware** 写入 `current_stage="done"`，Orchestrator 随后结束任务 |

因此：

- 在 `reviewer1` 超限时，`yes` 表示继续 `reviewer1`，**不是**切换到 `reviewer2`
- 在 `reviewer2` 超限时，`yes` 表示继续 `reviewer2`
- 只有当前 Reviewer 明确 `ACCEPT`，才会进入下一阶段或最终输出
- 任意场景下 `no` 均导致 `current_stage="done"`，均由 `StageStateMiddleware._update_after_hil` 负责写入；两者的差异仅在于 `yes` 的后续动作（阶段超限清零 round，全局超限不修改 round）

> **范围约束**：本次仅实现"超限后是否继续"这一类 HIL 交互。`confirm_continue` 不扩展为通用仲裁入口，不承担 Writer 与 Reviewer 争议裁决等其他 HIL 场景。
>
> **运行模式约束**：本架构当前仅支持**交互式运行**。当 reviewer 阶段超限或触发全局 `max_iterations` 兜底时，必须通过 HIL 由用户显式回答 `confirm_continue`。无人值守/批处理/静默执行模式不在本次设计范围内，不提供"自动按 yes/no 默认决策"的开关。

### 2.7 状态守卫设计（agent + guardrails）

#### 设计原则

- **agent 决策**："写什么内容、怎么审核、怎么提修改意见"由 agent 自主决定
- **middleware 记账**：`current_stage`、`reviewer1_round`、`reviewer2_round`、`awaiting_confirm_for` 由 `StageStateMiddleware` 独占写入，Orchestrator **只读不写**
- **state file 持久化**：`/drafts/review-state.json` 是流程状态的唯一真相源，不是 agent 的工作草稿

Orchestrator 可以读取状态文件辅助决策，但不得直接修改其中任何字段。
让 Orchestrator 既决策又记账，等于"读 agent 自己写的账本"，不构成真正的 guardrail。

#### `/drafts/review-state.json`

```json
{
  "current_stage": "reviewer1",
  "reviewer1_round": 0,
  "reviewer2_round": 0,
  "reviewer2_enabled": true,
  "awaiting_confirm_for": null
}
```

| 字段 | 取值 | 说明 |
|------|------|------|
| `current_stage` | `"reviewer1"` / `"reviewer2"` / `"done"` | 当前所处阶段 |
| `reviewer1_round` | 整数，从 0 开始 | reviewer1 已成功完成的审核次数 |
| `reviewer2_round` | 整数，从 0 开始 | reviewer2 已成功完成的审核次数 |
| `reviewer2_enabled` | bool | 初始化时从配置写入，后续只读 |
| `awaiting_confirm_for` | `null` / `"reviewer1"` / `"reviewer2"` | 当前是否正在等待一次阶段超限 HIL 确认；非 null 时禁止再次委派对应 reviewer。全局 `max_iterations` 超限不写入此字段，由 Orchestrator 直接调用 `confirm_continue` |

初始文件由 `agent_factory.py` 在构建 Orchestrator 之前写入，后续仅由 `StageStateMiddleware` 更新。

#### 轮次计数语义

> **round 的定义是"reviewer 成功完成一次审核并产出 verdict"。**

- 委派 reviewer 但执行失败（异常、超时等）：**不计入 round**
- Writer 修订：**不计入 round**
- 委派时不预先加 1，避免失败调用产生脏计数
- `StageStateMiddleware` 在 reviewer task **成功返回后**读取对应 verdict 文件，然后递增 round

verdict 读取路径：
- reviewer1 → `/drafts/review-verdict.json`
- reviewer2 → `/drafts/review-verdict-stage2.json`

middleware 从 verdict 文件读取结论，**不解析 task 的自由文本返回值**，耦合点固定在 verdict 文件 schema。

#### 状态更新时机

| 时机 | 触发事件 | 状态变更 |
|------|----------|----------|
| 初始化 | `agent_factory` 创建文件 | 写入全部初始值；后续 `agent_factory` 不再参与更新 |
| reviewer1 成功返回 | `task(subagent_type="reviewer1")` 完成 | `reviewer1_round += 1`；若 verdict=ACCEPT 且 `reviewer2_enabled=false`，则 `current_stage="done"`；若 verdict=ACCEPT 且 `reviewer2_enabled=true`，则 `current_stage="reviewer2"` |
| reviewer2 成功返回 | `task(subagent_type="reviewer2")` 完成 | `reviewer2_round += 1`；若 verdict=ACCEPT，则 `current_stage="done"` |
| 阶段超限，等待 HIL | reviewer round 达到上限，middleware 拦截委派 | `awaiting_confirm_for="reviewer1"` 或 `"reviewer2"`（全局 `max_iterations` 超限不写此字段） |
| confirm_continue 返回 yes（阶段超限） | HIL 用户确认继续当前阶段 | middleware 清零当前阶段 round；`awaiting_confirm_for=null` |
| confirm_continue 返回 no（阶段超限） | HIL 用户确认结束任务 | middleware 写入 `current_stage="done"`；`awaiting_confirm_for=null` |
| confirm_continue 返回 yes（全局超限） | HIL 用户确认继续执行 | 不修改阶段 round；`review-state.json` 无变化；Orchestrator 继续执行 |
| confirm_continue 返回 no（全局超限） | HIL 用户确认结束任务 | middleware 写入 `current_stage="done"`；保持状态文件自洽 |

#### 三个代码守卫点

| # | 守卫点 | 类型 | 触发条件 | 处置方式 |
|---|--------|------|----------|----------|
| 0 | 任务终态拦截 | 非法状态 | `task` 目标为任意 reviewer，但 `current_stage == "done"` | **抛出异常**，阻止委派 |
| 1 | 回退拦截 | 非法状态 | `task` 目标为 `reviewer1`，但 `current_stage == "reviewer2"` | **抛出异常**，阻止委派 |
| 2 | 非法阶段切换 | 非法状态 | `task` 目标为 `reviewer2`，但 `current_stage != "reviewer2"` | **抛出异常**，阻止委派 |
| 3 | 超限兜底 | 正常业务事件 | `task` 目标为 reviewer，对应 round 已达上限且 `awaiting_confirm_for` 为 null | **不抛异常**，写入 `awaiting_confirm_for`，拦截 task 执行，返回控制消息引导 Orchestrator 调用 `confirm_continue` |

> **超限 vs 非法状态的处置边界**：
> - 超限（守卫 3）是预期内的正常业务分支，不是程序错误，**必须走 HIL 流程**。抛异常会导致用户看到"任务失败"而非"是否继续"的决策交互。
> - 回退/越级（守卫 1、2）是 agent 行为违反流程约束，属于程序异常，**应抛出异常**中止执行。

**守卫 3 的执行逻辑**：

当超限触发时，middleware **不调用 `handler(request)`**（即不实际派发 reviewer 任务），而是：
1. 写入 `awaiting_confirm_for = target` 到 `review-state.json`
2. 直接返回一条控制消息，告知 Orchestrator 需要调用 `confirm_continue`

当 `awaiting_confirm_for != null` 时，后续任何对该阶段 reviewer 的委派都会被守卫 3 再次拦截（因为超限条件仍然成立），直到 `confirm_continue` 返回并清空该字段。

**reviewer2 准入规则（唯一真相源）**：

1. reviewer2 的唯一准入条件是 `review-state.json` 中 `current_stage == "reviewer2"`
2. reviewer1 的 ACCEPT 仅作为状态推进事件，由 `StageStateMiddleware` 在 reviewer1 审核任务成功返回后读取 verdict 文件并更新 `current_stage`
3. Orchestrator 不得绕过 `review-state.json`，直接依据 `review-verdict.json` 决定是否委派 reviewer2

三个守卫点均在 `StageStateMiddleware.awrap_tool_call` 中实现，在委派前校验、在返回后更新状态。

#### 职责划分

| 关注点 | 由谁负责 |
|--------|----------|
| 文档写什么内容 | Writer agent（完全 agentic） |
| 审核意见与质量判断 | Reviewer agent（完全 agentic） |
| 阶段切换决策、修订反馈分发 | Orchestrator prompt（agentic，在守卫约束范围内） |
| 状态文件读取（辅助决策） | Orchestrator（只读） |
| 流程状态记录与推进 | `StageStateMiddleware`（独占写入） |
| 非法回退、越级切换、超限拦截 | `StageStateMiddleware`（代码强制） |

### 2.6 verdict 文件分离

```
/drafts/review-verdict.json          → reviewer1 的结构化审核结论
/drafts/review-verdict-stage2.json   → reviewer2 的结构化审核结论
```

两个文件使用相同的 schema（与现有格式保持一致）：

```json
{
  "verdict": "REVISE",
  "summary": "缺少异常处理设计和数据模型定义"
}
```

说明：

- `verdict`：必填，取值为 `ACCEPT` 或 `REVISE`

- 当 `reviewer2.enabled=false` 时，不产生 `/drafts/review-verdict-stage2.json`

> **`hil_confirm` 参数废弃**：双阶段审核架构中，`confirm_continue` 是必需能力而非可选功能。`[STAGE_LIMIT_REACHED]` 协议要求 Orchestrator 收到控制消息后唯一合法的下一步是调用 `confirm_continue`。若工具未注入，协议断裂，Orchestrator 陷入死局——这不是功能降级，而是协议失效。因此 `hil_confirm` 参数从 `build_orchestrator_prompt()` 签名中移除，`agent_factory.py` 改为无条件注入 `confirm_continue`。

提示词中明确：

1. `reviewer2.enabled=false` 时，`reviewer1 ACCEPT` 后直接输出
2. 只有 `ACCEPT` 才允许阶段切换（由 `StageStateMiddleware` 强制，prompt 无需重复计数逻辑）
3. 进入 `reviewer2` 后不允许回退到 `reviewer1`（由 `StageStateMiddleware` 强制）
4. 进入第二阶段时，直接让 `reviewer2` 审核当前草稿，不需要 Writer 重新写
5. Orchestrator 每次决策前读取 `/drafts/review-state.json`，以当前状态为准，**不自行维护计数**
6. **`[STAGE_LIMIT_REACHED]` 控制消息协议**：当 reviewer task 返回内容以 `[STAGE_LIMIT_REACHED]` 开头时，这是系统守卫发出的控制信号，不是 reviewer 的审核意见。此时唯一允许的下一步是调用 `confirm_continue`，将该消息中的说明作为 `status` 参数传入。**不得**将此消息解读为 REVISE，**不得**再次委派 reviewer，**不得**委派 Writer。
7. **`confirm_continue` 的 `status` 参数内容要求**：`status` 不能只传超限数字告警，必须同时包含：① 哪个 reviewer、已完成几轮；② 该 reviewer 在本阶段持续坚持要求修改的**核心痛点摘要**（从最后一轮 reviewer 返回文本中提取，不超过 3 条）。这是用户做出"继续/结束"决策的核心依据——单纯显示"已达 N 轮上限"无法支撑有价值的判断。
   - 提取摘要时，必须优先抓取 reviewer 原始反馈中带有“必须修改”“严重问题”“关键遗漏”“阻断项”等高优先级语义的内容。
   - 若 reviewer 返回文本存在明确分节、编号或项目符号，优先摘取这些高优先级条目；汇总后保留不超过 3 条，每条一句，避免泛化成空洞总结。
   - **不得**优先摘取“建议优化”“可选改进”“非强制建议”类内容，除非反馈中不存在更高优先级问题。
8. **`[VERDICT_PARSE_ERROR]` 控制消息协议**：当 `StageStateMiddleware` 无法读取或解析 reviewer 的 verdict 文件时，必须将该轮按**系统级强制 REVISE**处理，并向 Orchestrator 返回以 `[VERDICT_PARSE_ERROR]` 开头的控制消息。该消息不是 reviewer 的审核意见。Orchestrator **不得**将其直接转交给 Writer，必须按"reviewer 输出异常"分支处理（例如重试同一 reviewer 一次，或中止并报告系统警告）。

#### `src/tools/hil.py`

`confirm_continue` 的返回文本开头加机器可读前缀，中间件通过前缀可靠判断用户决策，Orchestrator 仍读到完整人类可读文本：

```python
@tool
def confirm_continue(status: str) -> str:
    decision = interrupt({"status": status})
    normalized = str(decision).strip().lower()
    if normalized in ("yes", "y", "继续", "是"):
        return (
            "USER_DECISION: YES\n"
            "用户选择继续迭代。请从第 1 轮重新开始计数（进入续跑阶段），"
            "再给约一轮完整配额，Reviewer 提前 ACCEPT 则提前退出。"
        )
    return (
        "USER_DECISION: NO\n"
        "用户选择结束迭代，以当前版本作为最终输出，请进入步骤 6。"
    )
```

> **为什么需要前缀**：`StageStateMiddleware._update_after_hil` 接收的是工具返回的完整字符串，而不是用户的原始输入。若直接匹配 `"yes"` 等关键词，因实际返回文本是完整句子，永远不会命中，所有 yes 都会被误处理为 `current_stage="done"`。加前缀后，中间件只需检查 `"USER_DECISION: YES" in result_str` 即可可靠判断。

#### `src/middleware/stage_state.py`（新增）

`StageStateMiddleware` 挂载在 **Orchestrator** 上，是 `review-state.json` 的**唯一写入者**。
在 `awrap_tool_call` 中：委派前校验（三个守卫点），task 成功返回后更新状态。

```python
class StageStateMiddleware(AgentMiddleware):
    """Orchestrator 专属中间件：独占维护 review-state.json，强制执行阶段切换约束。

    职责：
    - 委派前：校验三个守卫点（回退拦截、非法切换、超限兜底）
    - 返回后：递增 round、推进 current_stage、管理 awaiting_confirm_for
    """

    def __init__(self, state_path: str, reviewer1_max: int, reviewer2_max: int):
        self._state_path = state_path
        self._reviewer1_max = reviewer1_max
        self._reviewer2_max = reviewer2_max

    async def awrap_tool_call(self, request, handler):
        tool_call = request.tool_call if hasattr(request, "tool_call") else {}
        tool_name = tool_call.get("name", "") if isinstance(tool_call, dict) else ""
        target = (tool_call.get("args") or {}).get("subagent_type", "") if isinstance(tool_call, dict) else ""

        if tool_name == "task" and target in ("reviewer1", "reviewer2"):
            # 返回 None 表示"需要拦截"（超限分支），否则正常执行
            intercept_msg = self._check_before_dispatch(target)
            if intercept_msg is not None:
                return intercept_msg   # 不调用 handler，直接返回控制消息

        result = await handler(request)

        if tool_name == "task" and target in ("reviewer1", "reviewer2"):
            self._update_after_reviewer(target)     # 递增 round，推进 current_stage
        elif tool_name == "confirm_continue":
            self._update_after_hil(result)          # 清零 round 或设置 done

        return result

    def _check_before_dispatch(self, target: str) -> str | None:
        """委派前校验。

        返回值：
        - None：校验通过，正常派发
        - str：需要拦截，返回控制消息给 Orchestrator（超限分支，不抛异常）
        非法状态（守卫 1、2）直接抛出异常，因为这属于 agent 行为异常，不是正常业务分支。
        """
        state = self._read_state()
        current = state["current_stage"]
        awaiting = state.get("awaiting_confirm_for")

        # 前置守卫：任务已结束，不允许再委派任何 reviewer
        if current == "done":
            raise RuntimeError("StageState: 任务已结束（current_stage=done），不允许再委派 reviewer")

        # 守卫 1：非法状态——禁止从 reviewer2 回退到 reviewer1
        if target == "reviewer1" and current == "reviewer2":
            raise RuntimeError("StageState: 禁止回退——current_stage 已为 reviewer2")

        # 守卫 2：非法状态——当前阶段尚未推进到 reviewer2，不允许委派
        # 唯一判断依据是 current_stage，不读取 verdict 文件
        if target == "reviewer2" and current != "reviewer2":
            raise RuntimeError("StageState: 非法切换——current_stage 尚未推进到 reviewer2")

        # 守卫 3：正常业务事件——超限，进入 HIL 分支，不抛异常
        max_val = self._reviewer1_max if target == "reviewer1" else self._reviewer2_max
        round_val = state.get(f"{target}_round", 0)
        if round_val >= max_val:
            if awaiting is None:
                # 首次触发超限：写入等待状态
                self._write_state({**state, "awaiting_confirm_for": target})
            # 返回控制消息，不执行 reviewer task，引导 Orchestrator 调用 confirm_continue
            return (
                f"[STAGE_LIMIT_REACHED] {target} 已完成 {round_val} 轮审核，达到上限（{max_val} 轮）。"
                f"请调用 confirm_continue 决定是否继续（yes = 给予续跑配额，no = 结束任务输出当前版本）。"
            )

        return None   # 校验通过

    def _update_after_reviewer(self, target: str) -> None:
        """reviewer task 成功返回后：递增 round，读取 verdict 推进阶段。"""
        state = self._read_state()
        verdict = self._read_verdict(target)          # 从 verdict 文件读取，不解析自由文本
        round_key = f"{target}_round"
        state[round_key] = state.get(round_key, 0) + 1

        if verdict == "ACCEPT":
            if target == "reviewer1" and state.get("reviewer2_enabled"):
                state["current_stage"] = "reviewer2"
            else:
                state["current_stage"] = "done"

        self._write_state(state)

    def _update_after_hil(self, result: Any) -> None:
        """confirm_continue 返回后更新 review-state.json。

        处理两类触发场景：
        - 阶段超限（awaiting_confirm_for in {"reviewer1", "reviewer2"}）：
            yes → 清零当前阶段 round，awaiting_confirm_for=null；
            no  → current_stage="done"，awaiting_confirm_for=null。
        - 全局超限（awaiting_confirm_for 为 null 或其他值）：
            yes → 不修改状态文件；
            no  → 写入 current_stage="done"，保持状态文件自洽（checkpointer 启用后存在复跑场景）。

        注意：result 是 confirm_continue 工具返回的完整字符串（如"USER_DECISION: YES\n用户选择继续迭代……"），
        不是用户的原始输入。必须通过前缀 USER_DECISION: YES/NO 来判断，
        不能直接匹配 "yes"/"no" 等关键词。
        """
        state = self._read_state()
        awaiting = state.get("awaiting_confirm_for")
        result_str = str(result)

        if awaiting not in ("reviewer1", "reviewer2"):
            # 全局超限触发的 HIL：yes 时不修改状态文件；
            # no 时同样写入 current_stage="done"，保持 review-state.json 自洽
            # （checkpointer 启用后存在复跑/恢复场景，"陈旧值无害"不成立）
            if "USER_DECISION: NO" in result_str:
                state["current_stage"] = "done"
                self._write_state(state)
            return

        if "USER_DECISION: YES" in result_str:
            state[f"{awaiting}_round"] = 0   # 清零当前阶段 round，给予续跑配额
            state["awaiting_confirm_for"] = None
        else:
            state["current_stage"] = "done"
            state["awaiting_confirm_for"] = None

        self._write_state(state)

    def _read_state(self) -> dict: ...
    def _write_state(self, state: dict) -> None: ...
    def _read_verdict(self, target: str) -> str:
        """读取 verdict 文件，返回 verdict 字段值（"ACCEPT" 或 "REVISE"）。

        容错约定：
        1. 文件不存在或无法读取 → 视为 REVISE，记录 WARNING
        2. JSON 解析失败（LLM 在 JSON 外附加了说明文字等） → 视为 REVISE，记录 WARNING
        3. verdict 字段缺失或值不为 ACCEPT/REVISE → 视为 REVISE，记录 WARNING

        以上情况均不重试（文件内容是 LLM 已写入的最终产物，重读不会改变结果）。
        但这些 REVISE 属于“系统级强制 REVISE”，不是 reviewer 的正常审核结论。
        调用方（_update_after_reviewer）除按 REVISE 处理、禁止 ACCEPT/阶段推进外，
        还必须向 Orchestrator 暴露一条 `[VERDICT_PARSE_ERROR]` 控制消息，
        告知这是 reviewer 输出异常，而非普通修改意见。
        """
        # reviewer1 → /drafts/review-verdict.json
        # reviewer2 → /drafts/review-verdict-stage2.json
        ...
```

## 5. reviewer2 提示词设计

### 5.1 `stage=1` 与 `stage=2` 的差异

| 维度 | `stage=1`（reviewer1） | `stage=2`（reviewer2） |
|------|------------------------|------------------------|
| 审核目标 | 从零开始全面审核 | 从独立视角全面审核（不跟踪 reviewer1 的问题） |
| 反馈来源 | 需求原文与参考文件 | 需求原文与参考文件（与 reviewer1 相同，不读 reviewer1 的 verdict） |
| verdict 输出路径 | `/drafts/review-verdict.json` | `/drafts/review-verdict-stage2.json` |
| 与 reviewer1 的关系 | 无 | 完全独立，互不负责 |

> **Skill 共用说明**：reviewer1 和 reviewer2 共用同一个 `skills/reviewer/tech-doc-review/SKILL.md`。
> skill 内部通过 agent 身份（`reviewer1` / `reviewer2`）分支决定 verdict 文件路径，
> 无需为 reviewer2 单独创建 skill 文件。
> 对于 verdict 文件这类仅包含 `verdict` / `summary` 两个字段的完整 JSON 状态文件，skill 应优先使用 `write_file` **全量覆盖写入**，不再要求 `read_file + edit_file` 替换整块。双阶段架构下 reviewer1 / reviewer2 已使用不同路径，原先为规避路径冲突而禁止 `write_file` 的约束不再适用。

### 5.2 `stage=2` 系统提示词补充内容

【第二阶段审核特别说明】

你是 reviewer2，当前文档已经通过了 reviewer1 的审核。你的职责是从自己的独立视角对文档进行终审。

## 6. Orchestrator 提示词扩展

### 6.1 新增工作流规则

```text
多阶段审核流程：

【第1阶段：reviewer1 审核循环，最多 {reviewer1_max} 轮】
- 每轮：Writer 写 → reviewer1 审
- reviewer1 ACCEPT：若 reviewer2.enabled=true 则进入第2阶段（直接让 reviewer2 审核，不需要 Writer 重新写）；否则直接输出
- reviewer1 REVISE：继续停留在第1阶段
- reviewer1 达上限（{reviewer1_max} 轮）：调用 confirm_continue
  - yes：继续 reviewer1，reviewer1 阶段计数清零重新获得完整配额
  - no：结束任务，输出当前版本（不进入 reviewer2）

【第2阶段：reviewer2 审核循环，最多 {reviewer2_max} 轮】（仅当 reviewer2.enabled=true 时存在）
- 仅在 reviewer1 ACCEPT 后进入，reviewer2 阶段计数从1重新开始
- 每轮：reviewer2 审 → REVISE → Writer 改 → reviewer2 再审
- reviewer2 ACCEPT：进入最终输出
- reviewer2 REVISE：继续停留在第2阶段
- reviewer2 达上限（{reviewer2_max} 轮）：调用 confirm_continue
  - yes：继续 reviewer2，reviewer2 阶段计数清零重新获得完整配额
  - no：结束任务，输出当前版本

【重要约束】
- 阶段切换只能由 ACCEPT 触发，不允许因达到上限而自动切换
- 一旦进入 reviewer2，不允许回退到 reviewer1
- reviewer1 和 reviewer2 的阶段计数完全独立，互不影响
- 全局 max_iterations 作为系统兜底上限，触发时同样调用 confirm_continue；no 时与阶段超限相同（middleware 写入 current_stage="done"，任务结束）；yes 时唯一差异是不清零任何阶段 round
- Writer 修订时，当前阶段 Reviewer 的最新反馈具有最高优先级；若与历史 Reviewer 意见冲突，必须以当前阶段反馈为准，不得折中
```

### 6.2 HIL 触发场景

| 场景 | 提示文案示例（`status` 参数） | yes 的含义 | no 的含义 |
|------|-------------------------------|------------|-----------|
| `reviewer1` 达上限 | "reviewer1 已完成 3 轮审核，仍返回 REVISE。核心问题：① §3.2 缺少错误处理流程；② API 契约描述存在歧义；③ 数据流图缺失关键节点。是否继续 reviewer1 审核？" | 继续 `reviewer1` | 结束任务并输出当前版本 |
| `reviewer2` 达上限 | "reviewer2 已完成 2 轮审核，仍返回 REVISE。核心问题：① 部署方案未覆盖回滚场景；② 监控指标定义不完整。是否继续 reviewer2 审核？" | 继续 `reviewer2` | 结束任务并输出当前版本 |
| 全局兜底上限 | "已达到最大总迭代轮次（当前处于 reviewer1 阶段，第 2 轮）。是否继续当前审核阶段？" | 继续当前阶段 | 结束任务并输出当前版本 |

### 6.3 verdict 的两种用途与读取职责

verdict 信息在系统中有两种完全不同的用途，由不同角色负责：

| 用途 | 读取方 | 数据来源 | 说明 |
|------|--------|----------|------|
| **流程决策**（切阶段、超限、结束） | `StageStateMiddleware` | verdict 文件（`review-verdict.json` / `review-verdict-stage2.json`） | reviewer 返回后，middleware 读取 verdict 字段更新 `review-state.json` |
| **HIL 状态提示 / 日志摘要** | Orchestrator / 日志系统 | verdict 文件 `summary` 字段 | 仅用于 `confirm_continue` 的 `status` 参数展示，或日志摘要，**不传递给 Writer** |
| **Writer 修订输入** | Orchestrator | reviewer **原始任务返回文本** | REVISE 时，Orchestrator 将 reviewer 的完整原始返回文本传给 Writer；**不得从 verdict 文件获取** |

**Orchestrator 不得基于 verdict 文件做流程分支决策**，所有流程状态（阶段、轮次、超限）均以 `review-state.json` 为准。

verdict 文件路径：

```text
reviewer1 → /drafts/review-verdict.json
reviewer2 → /drafts/review-verdict-stage2.json
```

**verdict 文件的职责边界**：

- `verdict` 字段：供 `StageStateMiddleware` 读取，用于递增轮次、推进阶段、判断是否结束
- `summary` 字段：仅用于 HIL 状态提示（`confirm_continue` 的 `status` 参数）或日志摘要展示

**Writer 修订的唯一完整反馈源是 reviewer 的原始任务返回文本，不是 verdict 文件。**

当 `review-state.json` 显示当前阶段未推进（reviewer 返回 REVISE）时，Orchestrator 必须将 reviewer 原始返回文本完整传递给 Writer（保留审核详情、必须修改项、非强制建议；可去除无关寒暄，但不得只传 `summary`）。verdict 文件不得作为 Writer 修订输入的唯一来源。

当 Writer 执行修订时，若当前阶段 Reviewer 的反馈与历史 Reviewer 的意见存在冲突，Orchestrator 必须明确告知 Writer：**以当前阶段 Reviewer 的最新反馈为唯一优先依据**。这条规则在 `current_stage == "reviewer2"` 时尤其重要；`reviewer1` 的 ACCEPT 不构成后续修订约束。

当 `current_stage == "reviewer2"` 且 reviewer2 返回 `REVISE` 时，Orchestrator 在委派 Writer 修订任务时应附加如下语义等价的明确说明：

```text
注意：你正在接受第二阶段终审。请以 reviewer2 的本轮反馈为唯一优先依据。
若其与 reviewer1 在第一阶段提出或接受的方案存在冲突，请忽略这些历史冲突意见，
不要折中，不要回退到 reviewer1 已接受的旧方案，直接按 reviewer2 的最新要求修订。
```

## 7. 风险评估

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| Orchestrator 误把"超限"理解为"切换阶段" | 中 | 提示词明确写死"阶段切换只能由 ACCEPT 触发" |
| `reviewer2` 开关关闭时，Orchestrator 仍错误委派 `reviewer2` | 中 | Prompt 与 AgentFactory 两处同时受 `reviewer2.enabled` 控制 |
| `reviewer2` 过早接手未成熟草稿 | 低 | 只允许 `reviewer1 ACCEPT` 后进入 `reviewer2` |
| 两个 Reviewer 来回拉扯，流程震荡 | 低 | 明确"进入 `reviewer2` 后不得回退" |
| 双阶段循环与全局 `max_iterations` 概念混淆 | 中 | 将全局 `max_iterations` 定位为系统兜底，而非主流程控制 |
| 配置中误将两个 Reviewer 设为相同模型 | 低 | 启动时校验 `(provider, model)` 组合不能完全相同 |
| `agents.reviewer` 重命名破坏现有配置 | 中 | 启动时检测旧命名并给出明确错误提示，引导用户手动迁移 |
| `reviewer2` 读取 `reviewer1` 的 verdict 文件（独立性被污染） | 低 | **已知软边界**：两者共享 `/drafts` 目录，无文件系统级隔离。缓解：system_prompt 明确禁止 `reviewer2` 读取 `review-verdict.json`；不引入虚拟文件系统过滤（开发成本远超风险）。若未来需要硬隔离，可考虑为各 reviewer 配置独立的 backend root |
