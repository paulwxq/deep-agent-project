# 通用软件工程文档生成系统演进设计

## 1. 演进目标
将现有的“SAS 数据血缘分析设计工具”重构为一个通用的、基于领域驱动的软件工程文档自动生成平台。系统应能根据用户选择的文档类型（需求规格说明书/系统概要设计说明书/系统详细设计说明书/测试计划）和项目类型（通用软件/AI系统/数仓ETL），自动选择对应的文档规格与模板，并复用现有的 Writer + 双 Reviewer 流程。

## 2. 核心架构变更

### 2.1 交互层：动态任务注入
- **Rich Prompt 交互**：在启动时增加交互菜单，由用户指定：
  - **文档类型**：需求规格说明书、系统概要设计说明书、系统详细设计说明书、测试计划。
  - **项目类型**：通用软件、AI系统、数据仓库/ETL。
  - **需求文件名**：需求文件默认位于 `./input/`。
- **当前版本约束**：
  - 删除 Manager 自动推断文档类型和项目类型的能力。
  - 启动时必须明确提供：文档类型、项目类型、需求文件名。
  - `./input/` 目录中，除需求文件外的其他文件和子目录，都视为参考材料。
- **任务上下文构建**：Manager 仅将用户选择、需求文件名和输入目录注入任务上下文，不负责解释文档细则，也不指定具体 spec/template 文件。

### 2.2 逻辑层：文档规格注册表 (Document Spec Registry)
- **位置**：
  - `skills/specs/`
  - `skills/templates/`
- **实现方式**：
  - 每个 spec 文件使用 YAML 定义文档规则。
  - 每个 template 文件使用 Markdown 定义文档骨架。
- **职责边界**：
  - `spec`：定义文档规则，决定“写什么、审什么”。
  - `template`：定义 Markdown 物理结构，决定“长什么样”。
  - `skill`：定义角色的方法论，决定“怎么写、怎么审”。
- **选择原则**：
  - Manager 不指定 spec/template。
  - Writer 和 Reviewer 在各自 skill 内，根据任务上下文和显式映射表，自主读取相同的 spec/template。
  - 第一版不由代码框架接管选择过程，保持 agentic 风格；若后续稳定性不足，再逐步收窄为代码固定逻辑。
- **映射原则**：
  - Writer 和 Reviewer 的 skill 中都必须写入同一份显式映射表。
  - 映射表以“文档类型 + 项目类型 -> spec/template 路径”的方式维护。
  - 第一版不额外抽取独立映射文件，以减少 agent 多层查找的复杂度。

### 2.3 代理层：专家化分工

#### Manager
- **定位**：流程调度者。
- **职责**：
  - 收集用户选择的文档类型、项目类型和输入路径。
  - 组织 Writer / Reviewer1 / Reviewer2 的执行顺序。
  - 管理阶段状态、HIL 和结束条件。
  - 在修订轮次中，将 Reviewer 原始反馈尽量原样传递给 Writer。
- **禁止事项**：
  - 不解释 spec 细节。
  - 不指定具体 spec/template 文件。
  - 不改写 Reviewer 的具体修改意见，只允许附加最小流程说明。

#### Writer (文档写手)
- **定位**：技术执行引擎。
- **改进**：不再硬编码章节，而是根据任务上下文自动匹配 spec/template。
- **核心逻辑**：
  - 根据 `文档类型 + 项目类型` 按 skill 中的显式映射表定位 spec/template。
  - 按 spec 约束和 template 骨架生成初稿。
  - 在初稿成功写入正式草稿文件后，写入 `drafts/writer-task-context.json`，记录本轮实际采用的任务上下文与 spec/template 选择结果。
  - 在修订阶段尽量保留 Reviewer 原始反馈中的细节，不擅自弱化问题。

#### Reviewer 1 (首席架构师 / Domain & Logic Architect)
- **定位**：架构与逻辑主审。
- **重点**：
  - 正确性。
  - 覆盖性。
  - 合理性。
  - 深度。
- **配置**：**Thinking 模式开启**。
- **执行原则**：
  - 根据任务上下文读取与 Writer 相同的 spec/template。
  - 重点判断“内容是否正确、完整、合理”。

#### Reviewer 2 (工程交付专家 / Engineering & Standards Lead)
- **定位**：交付与工程可用性终审。
- **重点**：
  - 可实施性。
  - 边界清晰度。
  - 规范性。
  - 交付质量。
- **配置**：**Thinking 模式关闭**（追求极致的条框校验）。
- **执行原则**：
  - 根据任务上下文读取与 Writer 相同的 spec/template。
  - 重点判断“文档是否足够用于实施、评审、测试和交付”。

## 3. 详细设计：文档规格注册表结构

建议的文件组织形式：
```text
skills/
├── specs/
│   ├── general_software_requirements_specification.yaml
│   ├── general_software_system_overview_design.yaml
│   ├── data_warehouse_system_detailed_design.yaml
│   └── ai_system_overview_design.yaml
├── templates/
│   ├── standard_requirements_specification.md
│   ├── standard_system_overview_design.md
│   ├── data_warehouse_system_detailed_design.md
│   └── ai_system_overview_design.md
├── writer/
│   ├── tech-doc-writing/
│   │   └── SKILL.md
│   └── tech-doc-revision/
│       └── SKILL.md
└── reviewer/
    └── tech-doc-review/
        └── SKILL.md
```

显式映射表示例：

- `通用软件 + 系统概要设计说明书`
  - spec: `skills/specs/general_software_system_overview_design.yaml`
  - template: `skills/templates/standard_system_overview_design.md`
- `数据仓库/ETL + 系统详细设计说明书`
  - spec: `skills/specs/data_warehouse_system_detailed_design.yaml`
  - template: `skills/templates/data_warehouse_system_detailed_design.md`

约束：
- Writer 和 Reviewer 必须维护内容相同的显式映射表，禁止各自维护不同版本。
- spec 内部通过 `template_path` 指向模板文件。
- Manager 只传递任务上下文，不参与 spec/template 的选择细节。
- 第一版保持 agentic：由 Writer 和 Reviewer 在 skill 中自行读取共享 spec/template；仅当后续出现一致性或稳定性问题时，再考虑将部分解析逻辑收敛到代码层。

## 4. 多 Agent 整体处理流程

### 4.1 主流程

1. **Manager 启动任务**
   - 通过 HIL / Rich 交互要求用户明确提供：
     - 文档类型
     - 项目类型
     - 需求文件名
   - 默认输入文件所在的目录为 `./input/`
   - `./input/` 中除需求文件外的其他文件和子目录均视为参考材料
   - 本步结束后，系统初始化或重置：
     - `drafts/review-state.json`

2. **Manager 下发 Writer 任务**
   - 只向 Writer 传递：
     - 文档类型
     - 项目类型
     - 需求文件路径
     - 输入文件所在的目录
   - 不指定 spec/template 文件

3. **Writer 自主选择规则资产**
   - 根据 skill 中的显式映射表选择唯一的 spec/template
   - 读取：
     - spec
     - template
     - 需求文件
     - `./input/` 下的参考材料
   - 生成初稿并写入正式草稿文件
   - 本步结束后，Writer 写入或覆盖：
     - `drafts/design.md`

4. **Writer 写入任务上下文黑板**
   - 在初稿成功落盘后，Writer 创建或覆盖：
     - `drafts/writer-task-context.json`
   - 该文件用于记录：
     - 文档类型
     - 项目类型
     - 需求文件
     - 输入目录
     - 参考材料范围
     - Writer 实际采用的 skill
     - 本轮实际采用的 spec/template
     - 最终审核通过后输出到 `./output/` 时使用的文件名
   - 本步结束后，Writer 写入或覆盖：
     - `drafts/writer-task-context.json`

5. **Manager 发起 Reviewer1 审核**
   - 只向 Reviewer1 传递：
     - 文档类型
     - 项目类型
     - 需求文件路径
     - 输入文件所在的目录
     - 草稿路径
   - Reviewer1 根据自己的 skill 和显式映射表独立选择 spec/template
   - 然后读取 `drafts/writer-task-context.json` 做一致性比对
   - 若发现与 Writer 记录的 spec/template 不一致，必须先执行一次自检复核
   - 只有在复核后仍确认自己选择正确、而 Writer 选择错误时，才返回 `spec_template_mismatch`
   - 若复核后确认 Writer 选择无误，则继续执行“架构与逻辑主审”
   - 本步结束后，Reviewer1 写入：
     - `drafts/review-verdict.json`
   - 同时系统中间件更新：
     - `drafts/review-state.json`

6. **Manager 根据 Reviewer1 结果处理**
   - 若 Reviewer1 返回 `REVISE`：
     - Manager 尽量原样将 Reviewer1 原始反馈转交给 Writer
     - Writer 重新选择并确认规则资产，进行重写或修订
   - 若 Reviewer1 返回 `ACCEPT`：
     - Manager 推进到 Reviewer2

7. **Manager 发起 Reviewer2 审核**
   - Manager 继续只传任务上下文和草稿路径
   - Reviewer2 根据自己的 skill 和显式映射表独立选择 spec/template
   - 然后读取 `drafts/writer-task-context.json` 做一致性比对
   - 若发现与 Writer 记录的 spec/template 不一致，必须先执行一次自检复核
   - 只有在复核后仍确认自己选择正确、而 Writer 选择错误时，才返回 `spec_template_mismatch`
   - 若复核后确认 Writer 选择无误，则继续执行“交付与工程可用性终审”
   - 本步结束后，Reviewer2 写入：
     - `drafts/review-verdict-stage2.json`
   - 同时系统中间件更新：
     - `drafts/review-state.json`

8. **Manager 根据 Reviewer2 结果处理**
   - 若 Reviewer2 返回 `REVISE`：
     - Manager 尽量原样将 Reviewer2 原始反馈转交给 Writer
     - Writer 进行重写或修订
   - 若 Reviewer2 返回 `ACCEPT`：
     - Manager 读取 `drafts/review-state.json`
     - 当 `current_stage == "done"` 时，判定审核流程结束
     - Manager 读取 `drafts/writer-task-context.json` 中的 `output_filename`
     - 将 `drafts/design.md` 复制到 `./output/{output_filename}`
     - 流程结束，输出最终文档
   - 本步不再向 `./drafts/` 新增输出文件；最终产物写入：
     - `./output/{output_filename}`

### 4.2 `writer-task-context.json`

定位：
- 由 Writer 独占写入
- Reviewer 只读，不更新
- 用于记录 Writer 本轮实际采用的任务上下文与规则资产选择结果

建议字段：

```json
{
  "doc_type": "系统概要设计说明书",
  "project_type": "AI系统",
  "requirement_file": "./input/example.md",
  "input_directory": "./input",
  "reference_scope": "除需求文件外，input 目录下其他文件和子目录均为参考材料",
  "selected_skill": "skills/writer/tech-doc-writing/SKILL.md",
  "selected_spec": "skills/specs/ai_system_overview_design.yaml",
  "selected_template": "skills/templates/ai_system_overview_design.md",
  "output_filename": "SAS数据血缘分析模块技术设计文档.md"
}
```

写入时机：
- Writer 完成初稿并成功写入正式草稿文件后再写入
- 若初稿未成功写入，则不创建或不更新该文件
- 当前架构改造后，不再生成 `drafts/output-filename.txt`
- 最终输出阶段直接从 `drafts/writer-task-context.json` 中读取 `output_filename`

与最终输出的关系：
- Manager 不直接从草稿文件名推导最终输出文件名
- Manager 以 `drafts/review-state.json` 中 `current_stage == "done"` 作为审核完成信号
- 审核完成后，Manager 从 `drafts/writer-task-context.json` 读取 `output_filename`
- 然后将 `drafts/design.md` 复制到 `./output/{output_filename}`

### 4.3 Reviewer 的两类 `REVISE`

Reviewer 返回 `REVISE` 时，需要区分两类情况，但不引入全新的 verdict 结构。

#### 普通内容问题

适用场景：
- 内容覆盖不足
- 逻辑错误
- 边界不清
- 结构不完整
- 工程可实施性不足

处理方式：
- Reviewer 正常给出正文级修改建议
- Manager 尽量原样把反馈转给 Writer
- Writer 进行常规修订

建议 verdict JSON：

```json
{
  "verdict": "REVISE",
  "summary": "文档仍存在若干内容问题，需要继续修订。",
  "revise_reason": "content_issue",
  "selected_skill": "skills/reviewer/tech-doc-review/SKILL.md",
  "selected_spec": "skills/specs/general_software_system_overview_design.yaml",
  "selected_template": "skills/templates/standard_system_overview_design.md"
}
```

#### `spec/template` 选择错误

适用场景：
- Reviewer 根据自己的 skill 显式映射表选择出的 spec/template
- 与 `writer-task-context.json` 中 Writer 记录的 `selected_spec/selected_template`
- 不一致

处理原则：
- 这是系统级阻断问题，不是普通正文问题
- Reviewer 发现不一致后，不得立即返回阻断结论
- Reviewer 必须先执行一次自检复核，确认自己没有误读任务上下文、误用映射表或误判文档类型/项目类型
- 只有在复核后仍确认 Writer 选择错误时，才停止正文审核并返回阻断结论
- 若复核后确认 Writer 选择无误，则继续正常正文审核
- 不再继续给出正文级修改建议

Reviewer 返回内容必须包含：
- 当前任务上下文
- Writer 实际采用的 spec/template
- Reviewer 判断的正确 spec/template
- 明确要求 Writer 重新按正确规则资产重写或重修

Manager 的处理方式：
- 不改写该反馈
- 原样转交给 Writer
- 由 Writer 重新选择规则资产，并重新生成或修订文档

建议 verdict JSON：

```json
{
  "verdict": "REVISE",
  "summary": "Writer 采用的 spec/template 与当前任务上下文不一致，需先纠正规则资产后再重新生成文档。",
  "revise_reason": "spec_template_mismatch",
  "selected_skill": "skills/reviewer/tech-doc-review/SKILL.md",
  "selected_spec": "skills/specs/general_software_system_overview_design.yaml",
  "selected_template": "skills/templates/standard_system_overview_design.md"
}
```

#### Reviewer verdict JSON 扩展原则

两个 Reviewer 都应在各自的 verdict JSON 中记录自己本轮实际采用的规则资产：

- reviewer1 -> `drafts/review-verdict.json`
- reviewer2 -> `drafts/review-verdict-stage2.json`

建议字段：
- `verdict`
- `summary`
- `revise_reason`（仅在 `REVISE` 时需要，可选）
- `selected_skill`
- `selected_spec`
- `selected_template`

说明：
- Reviewer 的 verdict JSON 只记录 Reviewer 自己实际采用的 skill/spec/template
- 不在 reviewer verdict JSON 中重复记录 Writer 的选择结果
- Writer 的选择结果统一记录在 `drafts/writer-task-context.json`

## 5. Spec Schema V1

`spec` 文件用于定义某一类文档的规则，不承载运行框架配置。V1 先收敛最小可用字段，保证 Writer 与 Reviewer 能基于同一份规格工作。

### 5.1 必需字段

- `id`
  - 稳定标识，用于日志、调试和测试引用。
  - 示例：`general_software_system_overview_design`
- `name`
  - 人类可读名称。
  - 示例：`系统概要设计说明书`
- `doc_type`
  - 文档类型。
  - 枚举值应与交互层保持一致，如：`需求规格说明书`、`系统概要设计说明书`、`系统详细设计说明书`、`测试计划`
- `project_type`
  - 项目类型。
  - 枚举值应与交互层保持一致，如：`通用软件`、`AI系统`、`数据仓库/ETL`
- `description`
  - 对该类文档用途的简要说明，供 Writer 和 Reviewer 建立共同上下文。
- `target_audience`
  - 目标读者列表。
  - 示例：架构师、后端研发、测试负责人
- `template_path`
  - 对应模板路径，指向 `skills/templates/` 下的 Markdown 模板文件。
- `required_sections`
  - 必备章节列表。
  - 由 Writer 用于组织正文结构，也作为 Reviewer 判断完整性的依据。
- `writing_rules`
  - 写作规则列表。
  - 用于约束 Writer 的写法和展开深度。
- `review_focus`
  - Reviewer 关注点定义。
  - 必须包含 `reviewer1` 和 `reviewer2` 两组列表，分别描述两位 Reviewer 的检查重点。

### 5.2 推荐字段

以下字段不作为 V1 阻断项，但建议为后续质量提升预留：

- `optional_sections`
  - 可选章节列表。
- `focus_points`
  - 当前文档类型最需要展开说明的重点内容。
- `forbidden_content`
  - 不应出现的内容，如空话、无结论表述、未落地的技术口号。
- `input_expectations`
  - 期望输入材料说明，如需求清单、现有系统背景、接口资料、数据口径说明。
- `output_conventions`
  - 输出规范，如标题风格、图表要求、表格要求、文件命名建议。
- `clarification_policy`
  - 信息不足时的默认处理策略，如“明确标记待确认”或“要求先澄清”。
- `quality_bar`
  - 文档达到何种标准才可视为可交付。

### 5.3 不属于 Spec 的内容

以下内容不应写入 spec，而应留在运行框架、agent 配置或状态机逻辑中：

- 模型/provider 配置
- thinking 开关
- Reviewer 轮次上限
- HIL / 阶段状态机规则
- Manager 调度策略
- 长篇 prompt 正文

### 5.4 Spec 示例

```yaml
id: general_software_system_overview_design
name: 系统概要设计说明书
doc_type: 系统概要设计说明书
project_type: 通用软件
description: 面向架构评审与开发实施的系统级概要设计文档。

target_audience:
  - 架构师
  - 后端研发
  - 测试负责人

template_path: skills/templates/standard_system_overview_design.md

required_sections:
  - 项目概述
  - 总体架构设计
  - 核心功能设计
  - 数据架构设计
  - 系统接口设计
  - 非功能性设计

writing_rules:
  - 文档必须能指导后续详细设计与开发实现
  - 不得只写原则，关键模块必须写清职责与边界
  - 必须说明核心技术选型及理由
  - 必须覆盖异常、边界和非功能性约束

review_focus:
  reviewer1:
    - 需求覆盖是否完整
    - 架构是否合理
    - 模块边界是否清晰
    - 是否存在逻辑冲突或关键遗漏
  reviewer2:
    - 是否足以指导实施与评审
    - 接口、异常、约束是否清楚
    - 文档结构与术语是否规范统一
    - 是否达到交付质量
```

## 6. Writer Skill 关键章节示例
