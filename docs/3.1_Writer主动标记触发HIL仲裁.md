# Writer 主动标记触发 HIL 仲裁

> **版本**: v1.0  
> **日期**: 2026-03-24  
> **状态**: 待审核  
> **父文档**: `docs/3_需求澄清与人机协作扩展设计.md`  

---

## 1. 需求背景

### 1.1 现有 HIL 触发场景

当前系统支持两类 HIL 中断：

| 触发时机 | 触发条件 | HIL 工具 |
|----------|----------|----------|
| 需求澄清 | Orchestrator 发现需求存在关键歧义（冲突/边界不清/约束缺失） | `ask_user` |
| 超限确认 | Writer-Reviewer 迭代循环达到 `max_iterations` 上限 | `confirm_continue` |

### 1.2 新需求场景

在迭代过程中，可能出现以下情况：

- **Writer 与 Reviewer 在某技术选型上持续拉锯**
- Reviewer 标记为 REVISE，给出了具体反馈
- Writer 认为自己的方案更合理，在修订记录中说明"未采纳 → 理由"
- Reviewer 下一轮看到理由后，继续坚持或接受
- 双方在此问题上反复，但未达到循环上限

此时，用户比 LLM 更适合做仲裁——因为这是技术判断分歧，不是文档质量问题。

### 1.3 原方案的局限性

最初考虑让 Orchestrator 的 LLM 主观判断"分歧是否大到需要仲裁"，但存在以下问题：

| 问题 | 分析 |
|------|------|
| 无法量化触发条件 | "分歧大"是纯主观判断，不同模型/不同调用结果可能不同 |
| 增加提示词复杂度 | Orchestrator 提示词已包含大量工作流逻辑，再叠主观判断会臃肿 |
| 误触发风险 | 可能频繁打断用户，体验变差 |

---

## 2. 解决方案：Writer 主动标记

### 2.1 核心思路

**不依赖 LLM 主观判断，而是让 Writer 自己决定是否需要用户介入。**

当 Writer 认为 Reviewer 的某条反馈不合理、且无法通过对话说服 Reviewer 时，可以在修订记录中明确包含 **`[提请用户决策]`** 强特征标记。Orchestrator 在解析 Writer 的 `content` 或读取 `design.md` 时，检测到此标记后调用 `confirm_continue`，由用户做最终仲裁。

### 2.2 触发条件

| 条件 | 说明 |
|------|------|
| 触发者 | Writer（主动标记） |
| 触发方式 | 在修订记录的"未采纳"条目中写明 `[提请用户决策]` |
| 触发时机 | Orchestrator 委派下一轮 Reviewer 之前 |
| 无需额外配置 | 复用现有的 `confirm_continue` 工具 |

### 2.3 用户决策选项（闭环逻辑）

`confirm_continue` 的现有 yes/no 二选一可以覆盖技术仲裁场景，但 Orchestrator 需根据结果执行不同的后续指令：

| 用户选择 | 对应语义 | Orchestrator 对 Writer 的下一轮指令 (闭环驱动) |
|----------|----------|------------------------------------------------|
| **yes** / 继续 | 支持 Writer | "用户已仲裁：支持你的方案。请忽略 Reviewer 关于 XXX 的意见，将其标记为最终决定。" |
| **no** / 结束 | 支持 Reviewer | "**用户已仲裁：驳回你的理由。**请务必按照 Reviewer 的原始意见进行修订，不得再次提出争议。" |

> **注：** 为了实现这一逻辑，`src/tools/hil.py` 中的 `confirm_continue` 返回值应修改为更通用的状态码（如 `USER_DECISION: YES/NO`），由 Orchestrator 根据上下文解读其具体含义。

## 3. 实现方案

### 3.1 提示词与工具改动总览

---

### 3.2 Writer 提示词修改

**文件**：`src/prompts/writer_prompt.py`

**在修订规则部分增加以下内容**（放在现有的"修订记录格式"之后）：

```
提请用户决策：
如果 Reviewer 的某条反馈你认为不合理、且经过评估后认为自己的方案更优，但担心 Reviewer 在下一轮仍会坚持原意见导致双方持续拉锯，可以在修订记录的"未采纳"条目中附加 [提请用户决策] 标记。

示例：
  ## 修订记录
  ### 第 2 轮修订
  - 未采纳：Reviewer 认为应使用 Redis 缓存 → 理由：[理由描述]。[提请用户决策]

注意：
- "[提请用户决策]"意味着你暂时搁置此争议，期望用户给出最终指示
- Orchestrator 检测到此标记后，会暂停并请用户做最终仲裁
- 在用户做出决定前，不要自行决定最终方案
```

---

### 3.3 Orchestrator 提示词修改

**文件**：`src/prompts/orchestrator_prompt.py`

**在"判定 Reviewer 结论的方法"段落之后，增加以下段落**：

```
技术分歧仲裁（Writer 主动标记检测）：

在委派下一轮 Reviewer 之前，请先检查 Writer 的回复文本或 /drafts/design.md 的"修订记录"。
如果发现任何"未采纳"条目中包含 [提请用户决策] 标记：

1. 汇总争议：提取争议点、Writer 理由及 Reviewer 原始意见。
2. 调用 confirm_continue 工具，status 参数包含上述汇总信息。
3. 根据工具返回的决策结果（YES/NO）采取行动：
   - 结果为 YES (支持 Writer)：在下一轮委派中告知 Writer：“用户已仲裁支持你的方案，请将该点标记为最终决定并继续。”
   - 结果为 NO (支持 Reviewer)：在下一轮委派中告知 Writer：“用户已仲裁驳回你的理由。你必须采纳 Reviewer 关于 XXX 的意见进行修订，不得再次提出争议。”

注意：只有 Writer 明确标记 [提请用户决策] 的条目才触发此机制，
普通的"未采纳 → 理由"不触发 HIL，仅需在下一轮告知 Reviewer 你的理由。
```

---

### 3.4 流程示意

```
Writer 修订 → 发现 Reviewer 某条反馈不合理
    ↓
Writer 在修订记录中写："未采纳 → 理由 → 提请用户决策"
    ↓
Orchestrator 检测到"提请用户决策"标记
    ↓
调用 confirm_continue，告知用户争议内容和双方理由
    ↓
用户决策（yes/no）
    ↓
yes → Writer 方案作为最终决定，继续
no  → Writer 按 Reviewer 意见修订
```

## 4. 与现有 HIL 的关系

| HIL 触发场景 | 触发条件 | 现有/新增 |
|-------------|----------|-----------|
| 需求澄清 | Orchestrator 发现需求歧义 | 现有 |
| 超限确认 | 循环达上限 | 现有 |
| **Writer 主动标记仲裁** | Writer 在修订记录中写"提请用户决策" | **新增** |

新增的触发场景**复用**现有的 `confirm_continue` 工具和 `_run_with_hil()` 机制，无需新增 HIL 工具。

---

## 5. 风险评估

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| Writer 滥用"提请用户决策"标记，频繁打断用户 | 低 | 提示词中明确说明"只有真正无法调和的分歧才使用" |
| Orchestrator 漏检标记 | 低 | 检测逻辑简单（字符串匹配），且通过 read_file 直接读取文件，非文本解析 |
| 用户对技术争议无法做出判断 | 低 | 用户可以输入"我需要更多信息"拒绝决策，或选择 no 让 Writer 遵从 Reviewer |

## 7. 后续扩展方向

如果未来需要更细粒度的用户仲裁，可以扩展 `confirm_continue` 的选项：

```
confirm_continue 的 status 参数可包含：
- 争议技术选点摘要
- Writer 方案
- Reviewer 方案
- 用户可选：采纳 Writer / 采纳 Reviewer / 其他方案（自由文本输入）
```

但这是后话，当前先用 yes/no 二选一落地。
