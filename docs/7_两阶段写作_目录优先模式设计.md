# 7. 两阶段写作：目录优先模式设计

## 1. 背景与目标

### 1.1 问题

当前 Writer 在初稿阶段一次性生成完整文档，存在以下问题：

1. **结构性返工成本高**：若 Reviewer 对文档的章节组织有意见（如"缺少某个模块的设计章节"），Writer 需要对全文进行较大范围的改动
2. **首次生成时间长**：Writer 需要一次性输出大量 token，单次 LLM 调用可能耗时 20+ 分钟
3. **结构与内容混合审核**：Reviewer 第1轮同时审核"章节结构是否合理"和"内容是否详细"，反馈比较分散

### 1.2 方案目标

引入"目录优先"的两阶段写作模式：

- **阶段一**：Writer 仅生成文档目录大纲（骨架），经 Reviewer 审核批准后再进入阶段二
- **阶段二**：Writer 基于已批准的目录结构填充完整内容，Reviewer 专注审核内容质量

**预期收益**：
- 结构性问题在阶段一暴露，避免全文写完后推倒重来
- Reviewer 审核职责分离，阶段一看结构、阶段二看内容
- 整体迭代时间预计缩短 20-30%（主要来自减少返工）

## 2. 整体架构变化

### 2.1 新工作流程

```
【现有流程】
Writer 写全文 → Reviewer 审全文 → [修订循环] → 完成

【新流程】
阶段一（目录）：
  Writer 写目录 → Reviewer 审目录 → [修订循环] → 目录 ACCEPT

阶段二（全文）：
  Writer 基于目录写全文 → Reviewer 审全文 → [修订循环] → 全文 ACCEPT → 完成
```

### 2.2 文件职责

| 文件 | 阶段一 | 阶段二 |
|------|--------|--------|
| `/drafts/outline.md` | Writer 写入目录大纲 | 只读（作为结构基线） |
| `/drafts/design.md` | 不存在 | Writer 写入完整文档 |
| `/drafts/review-verdict.json` | Reviewer 写入目录审核结论 | Reviewer 写入全文审核结论 |

# 技术设计文档目录大纲撰写指南

## 什么是目录大纲

目录大纲是完整文档的"骨架"，**不包含正文内容**，仅包含：

- 所有一级章节（`##`）的标题
- 所有二级小节（`###`）的标题
- 每个小节 2-3 句话的**内容摘要**（说明"这一节会写什么"，供 Reviewer 判断覆盖情况）

不包含：正文段落、代码片段、伪代码、数据模型定义、接口签名等详细内容。

## 标准输出格式

```markdown
# [文档标题]

## 1. [章节名]
[1句话说明本章核心目的]

### 1.1 [小节名]
**将覆盖**：[2-3句话，说明这一节的内容边界和关键设计点]

### 1.2 [小节名]
**将覆盖**：...

## 2. [章节名]
...
```

## 章节粒度建议

- 一级章节：5-10 个，对应需求的主要模块划分
- 二级小节：每章 2-5 个，不宜超过 6 个
- 三级标题（`####`）：大纲阶段不需要，留给全文撰写时决定
- 每个小节的内容摘要：不超过 3 句话，聚焦"写什么"而不是"怎么写"

## 撰写原则

1. **需求驱动**：每个章节都应能对应至少一条需求；有需求的内容必须有对应章节承载
2. **完整性优先**：宁可多一个章节也不要遗漏需求，结构问题比内容缺失更难返工
3. **面向 Reviewer**：摘要要让 Reviewer 能判断"这个结构够不够，需求点有没有漏"
4. **面向后续 Writer**：章节划分要清晰，让填写阶段的 Writer 能按章节独立工作

## 核心约束

- **输出文件**：唯一输出文件是 `/drafts/outline.md`，**禁止写入** `/drafts/design.md`
- **不写正文**：不要在大纲阶段输出任何正文段落，节约 token 留给全文阶段
- **完成标志**：调用 `write_file` 将内容写入 `/drafts/outline.md` 后即视为完成
- **目录访问**：必须先阅读需求文件和 `/input/` 下的参考文件，确保大纲不遗漏需求
```

# 技术设计文档目录大纲审核检查清单

## 审核准备

1. 读取需求文件，明确所有需求点
2. 读取 `/drafts/outline.md`，获取待审核的目录结构
3. 无需读取 `/input/` 下的完整参考文件（除非需要核实某个具体需求细节）

## 审核维度（大纲专用，不同于全文审核）

### 1. 需求覆盖性（最高优先级）
- 所有需求点是否都有对应的章节承载？
- 是否有遗漏的重要模块（如异常处理、配置部署、性能考虑等）？
- 每个小节的摘要描述，是否能判断出该节会处理对应需求？

### 2. 结构合理性
- 章节组织是否符合逻辑顺序（如：概述 → 架构 → 模块 → 流程 → 部署）？
- 是否存在职责重叠的章节（内容范围描述有大量交叉）？
- 章节粒度是否合适（不过粗也不过细）？

### 3. 扩展完整性
- 章节摘要是否清晰说明了"这节将写什么"？
- 后续 Writer 能否根据这个大纲独立完成每个章节的填充？

## 不在本轮审核范围内

以下内容在大纲阶段**不要求、不审核、不因此 REVISE**：

- 具体的技术实现细节、算法描述
- 代码片段、API 签名、伪代码
- 数据模型的完整字段定义
- 性能数字、配置参数的具体值
- 内容"不够详细"（大纲本就是骨架，详细内容留给全文阶段）

## REVISE 判断标准

**必须 REVISE 的情况**：
- 某条需求完全没有章节承载（需求覆盖漏项）
- 章节结构严重不合逻辑，导致全文写完后必然需要大规模重组
- 关键模块缺失（如：没有核心算法章节、没有部署配置章节等）

**不应 REVISE 的情况**：
- 章节内容摘要"写得不够详细"
- 某个设计细节在大纲中没体现（细节在全文阶段填充）
- 章节顺序有轻微调整空间但不影响覆盖完整性

## ACCEPT 标准

大纲结构能够完整承载所有需求，Writer 能够据此写出完整文档，且无明显的结构性遗漏。
```

### 4.3 修改：`src/prompts/orchestrator_prompt.py`

**改动说明**：将工作流从单循环改为两阶段循环，增加阶段一（目录审核）和阶段二（全文审核）的控制逻辑。

**完整修改后的 `build_orchestrator_prompt` 函数**：

```python
def build_orchestrator_prompt(
    max_iterations: int = 3,
    requirement_filename: str = "requirement.txt",
    hil_confirm: bool = False,
) -> str:
    req_path = f"/input/{requirement_filename}"

    step5_confirm = (
        f"5. 每个阶段最多迭代 {max_iterations} 轮。如果某阶段达到最大轮次仍未通过：\n"
        f"   - 如果工具列表中存在 confirm_continue，使用该工具告知用户当前迭代情况\n"
        f"     （例如：\"目录审核已完成 {max_iterations} 轮，Reviewer 最后一轮返回 REVISE，\n"
        f"     主要问题：XXX。是否重置计数、再给约一轮完整配额继续迭代？\"）\n"
        f"     - 用户回答 yes：重置该阶段计数，再给约一轮完整配额\n"
        f"     - 用户回答 no：如果在阶段一，直接跳过目录审核进入阶段二；如果在阶段二，输出当前版本 + 最后一轮审核意见后退出\n"
        f"   - 如果工具列表中不存在 confirm_continue（非交互模式）：\n"
        f"     - 阶段一达到上限：跳过目录审核，直接进入阶段二（以当前 outline.md 为结构基线）\n"
        f"     - 阶段二达到上限：输出当前版本 + 最后一轮审核意见后退出"
    ) if hil_confirm else (
        f"5. 每个阶段最多迭代 {max_iterations} 轮。\n"
        f"   - 阶段一达到上限仍未通过：跳过目录审核，直接进入阶段二（以当前 outline.md 为结构基线）\n"
        f"   - 阶段二达到上限仍未通过：输出当前版本 + 最后一轮审核意见后退出"
    )

    return f"""\
你是一个技术设计文档生成系统的编排器（Orchestrator）。你的职责是协调 Writer 和 Reviewer 两个子代理，通过两阶段迭代循环生成高质量的技术设计文档。

关键目录与文件：
- /input/ — 用户输入目录（只读）
  - {requirement_filename} — 主需求文件
  - 其他文件 — 需求相关的参考文件（如代码、数据定义、配置等），供 Agent 按需阅读
- /drafts/ — Agent 工作目录（可写）
  - outline.md — 阶段一：Writer 输出的文档目录大纲（每轮更新）
  - design.md — 阶段二：Writer 输出的完整设计文档（每轮更新）
  - review-verdict.json — Reviewer 的结构化结论（每轮更新）
  - qa-supplement.md — Writer 向用户澄清需求后生成的问答记录（若存在，权威性高于原始需求）

工作流程：

【阶段一：目录大纲审核】
1. 委派 Writer 子代理生成文档目录大纲，告知：
   - 需求文件路径 {req_path}
   - /input/ 下可能有参考文件需要阅读
   - 目录大纲保存到 /drafts/outline.md（不是 design.md）
   - 明确说明：本次任务是生成目录大纲而非完整文档
2. Writer 完成后，用 read_file 读取 /drafts/outline.md 验证文件存在且内容非空：
   - 若文件存在且有内容：继续下一步
   - 若文件不存在或为空：视为 Writer 未完成，重新委派并明确说明上次未写入 /drafts/outline.md
3. 委派 Reviewer 审核目录大纲，告知：
   - 审核对象是 /drafts/outline.md（目录大纲，不是完整设计文档）
   - 需求文件路径 {req_path}
   - 审核标准：需求覆盖性和结构合理性，不要求内容细节
4. 如果 Reviewer 返回 REVISE，将反馈完整传递给 Writer 修订目录大纲（修订后仍保存到 /drafts/outline.md）
5. 如果 Reviewer 返回 ACCEPT，进入阶段二

【阶段二：完整文档生成与审核】
6. 委派 Writer 子代理基于已批准的目录大纲撰写完整设计文档，告知：
   - 需求文件路径 {req_path}
   - 必须先读取 /drafts/outline.md 作为文档结构基线，严格按照已批准的章节结构填充内容
   - 完整文档保存到 /drafts/design.md
   - 明确说明：本次任务是基于已批准目录填充完整内容
7. Writer 完成后，用 read_file 读取 /drafts/design.md 验证文件存在且内容非空：
   - 若文件存在且有内容：继续下一步
   - 若文件不存在或为空：视为 Writer 未完成，重新委派并明确说明
8. 委派 Reviewer 审核完整文档，告知：
   - 审核对象是 /drafts/design.md（完整设计文档）
   - 需求文件路径 {req_path}
9. 如果 Reviewer 返回 REVISE，将反馈完整传递给 Writer 修订 /drafts/design.md
10. 如果 Reviewer 返回 ACCEPT，结束循环
{step5_confirm}
11. 迭代结束后（无论 ACCEPT 还是达到最大轮次），先根据需求内容为最终文档建议一个简洁的中文文件名，使用 write_file 写入 /drafts/output-filename.txt。文件内容仅包含文件名本身，不含路径、不含解释文字、不含引号。必须在返回最终回复之前完成此步骤
12. 最后返回迭代摘要（两个阶段各经过几轮、最终结论、关键改进点），不需要返回文档全文

判定 Reviewer 结论的方法（按优先级）：
1. 在 Reviewer 的返回文本中查找 "VERDICT: ACCEPT" 或 "VERDICT: REVISE"
2. 如果文本解析失败，用 read_file 读取 /drafts/review-verdict.json 中的 verdict 字段
3. 如果都失败，视为 REVISE（安全兜底，多审一轮的代价小于误判通过）

重要规则：
- 不要自己编写或修改设计文档内容，始终通过委派 Writer 完成
- 不要自己审核文档，始终通过委派 Reviewer 完成
- 向 Writer 传递反馈时，完整保留 Reviewer 的具体意见，不要过度概括
- **后续轮次委派 Reviewer 时**（即各阶段第 2 轮及以后），必须在任务描述的开头明确写上"这是第N轮审核（后续轮次）"，以便 Reviewer 采用精简准备流程（不重新读取 /input/ 下的参考文件）。同时列出上一轮 REVISE 中的必须修改项摘要，以便 Reviewer 优先验证
- **阶段一修订轮**委派 Writer 时，明确要求：先读取 /drafts/outline.md，只修改受影响章节，禁止写入 /drafts/design.md
- **阶段二修订轮**委派 Writer 时，明确要求：先读取 /drafts/design.md，优先使用 edit_file，只修改受影响章节，默认不要重新全量扫描 /input/
- 阶段二的 Writer 必须以 /drafts/outline.md 为结构基线，不得随意增删章节；若确有结构调整需要，须在任务返回中说明
- **唯一正式草稿文件**：outline.md（阶段一）和 design.md（阶段二）。忽略其他变体文件
- 不要让 Writer 或 Reviewer 浏览 /drafts/_backups 或 /output
- 最终输出时，返回迭代摘要（两个阶段各经过几轮迭代、最终审核结论、关键改进点）。不需要在响应中包含文档全文——文档的唯一正式源是 /drafts/design.md，由 main.py 负责复制到 output/ 目录\
"""
```
