# 7.1 reviewer2受限能力终审模式设计

> 暂时不考虑开发。本文档仅保留为后续可选思路，不进入当前版本实施范围。

## 1. 背景

当前双阶段审核架构中，`reviewer1` 与 `reviewer2` 默认共享相近的能力边界：

- 都可以读取工作区文件
- 都可以使用已注入的工具
- 都可以访问 MCP / Context7 等扩展能力

这种设计能够快速复用现有实现，但也带来两个问题：

1. 两个 Reviewer 的工作方式过于接近，第二阶段容易退化为“再跑一遍 reviewer1”。
2. `reviewer2` 的独立性目前主要依赖提示词约束，属于软边界，仍可能出现过度勘查工作区、读取过多上下文或间接受上一阶段影响的情况。

因此，有必要为 `reviewer2` 增加“受限能力终审模式”，将其定位为：

- 不负责大范围事实勘查
- 不负责外部资料检索
- 重点负责需求一致性、逻辑完整性、架构合理性和最终可落地性判断


## 2. 设计目标

本次设计希望实现以下目标：

1. 保留双阶段审核架构不变。
2. 保留 `reviewer1` 的全能力审核模式。
3. 将 `reviewer2` 调整为“受限能力终审者”。
4. 强化 `reviewer2` 的独立性，减少其主动浏览共享目录和外部资源的机会。
5. 降低第二阶段审核的 token 成本、工具调用成本和失败面。


## 3. 核心思路

### 3.1 reviewer1 与 reviewer2 的职责分工

建议将两个 Reviewer 明确拆分为两种角色：

#### reviewer1：证据型 Reviewer

特点：

- 可以读取需求文件、设计文档和必要参考材料
- 可以使用外部工具与 MCP
- 可以做更充分的事实核查和技术核对
- 更适合发现“需求遗漏、样例未覆盖、外部 API/框架用法不当”等问题

#### reviewer2：裁判型 Reviewer

特点：

- 不做大范围勘查
- 不访问外部搜索和 MCP
- 只基于最小审核输入做终审
- 更适合判断“当前文档是否已经满足需求、逻辑是否自洽、是否达到可开发标准”


### 3.2 reviewer2 不是“完全无工具”

`reviewer2` 不应被设计为完全无工具。

原因是：当前项目的 Reviewer 工作方式本身依赖文件工具来读取审核对象。如果彻底移除工具，`reviewer2` 将无法稳定读取：

- 主需求文件
- `/drafts/design.md`
- 必要的补充说明文件

因此，更合理的方向不是“零工具”，而是“最小工具集 + 最小可见范围”。


## 4. reviewer2 需要新增的配置项

结论：**是的，需要为 `reviewer2` 增加专门的能力边界配置。**

建议不要只用一个布尔开关粗暴控制，而是拆成几类配置。

### 4.1 能力模式配置

建议新增一个 reviewer 能力模式字段，例如：

```yaml
agents:
  reviewer2:
    capability_mode: "restricted"
```

建议语义：

- `full`：全能力模式，与当前 reviewer1 类似
- `restricted`：受限能力模式，仅保留最小审核能力

这样未来如果需要回滚、A/B 对比或临时放开能力，不需要改代码结构，只需要调整配置。


### 4.2 MCP 能力开关

建议新增：

```yaml
agents:
  reviewer2:
    enable_mcp: false
```

语义：

- `false`：不为 reviewer2 注入 Context7 等 MCP 工具
- `true`：允许 reviewer2 使用 MCP

默认建议：

- `reviewer1.enable_mcp = true`
- `reviewer2.enable_mcp = false`


### 4.3 外部工具开关

建议新增：

```yaml
agents:
  reviewer2:
    enable_external_tools: false
```

这里的“外部工具”主要指：

- Tavily / Web Search
- 其他会引导 Reviewer 去检索外部世界的工具

默认建议：

- `reviewer1.enable_external_tools = true`
- `reviewer2.enable_external_tools = false`


### 4.4 文件系统能力模式

建议新增：

```yaml
agents:
  reviewer2:
    filesystem_mode: "minimal_readonly"
```

建议语义：

- `full`：与当前 FilesystemBackend 类似，可广泛浏览工作区
- `minimal_readonly`：只能读取白名单文件，不允许广泛浏览

如果当前框架不方便一步到位实现真正的只读白名单 backend，也建议先在设计层明确这个目标，再分阶段落地。


### 4.5 允许读取的文件白名单

建议新增：

```yaml
agents:
  reviewer2:
    allowed_read_paths:
      - "/input/{requirement_filename}"
      - "/drafts/design.md"
      - "/drafts/qa-supplement.md"
      - "/drafts/review-materials.md"
```

说明：

- `reviewer2` 应仅允许读取终审必需材料
- 不建议默认允许其浏览整个 `/drafts/`
- 不建议允许其读取 `review-verdict.json`
- 不建议允许其进入 `_backups/`、`output/` 等目录

其中 `/drafts/review-materials.md` 可作为未来扩展项，用于承接 Orchestrator 为终审阶段整理的有限中间结果。


## 5. reviewer2 的最小输入建议

在受限能力模式下，建议 `reviewer2` 默认只接收以下输入：

1. 主需求文件
2. 当前版本的 `/drafts/design.md`
3. 若存在，则读取 `/drafts/qa-supplement.md`
4. 若未来需要，可以额外读取一个由 Orchestrator 整理的终审材料文件

不建议默认允许其读取：

- `/drafts/review-verdict.json`
- reviewer1 的完整原始反馈
- 大量 `/input/` 参考代码
- MCP 返回的外部资料


## 6. 对当前系统行为的影响

### 6.1 优点

1. `reviewer2` 更容易保持独立性。
2. 双 Reviewer 的角色差异更清晰。
3. 第二阶段审核成本更低。
4. 工具调用失败、MCP 调用失败、搜索不稳定等问题对 reviewer2 的影响显著降低。
5. reviewer2 更像“终审裁判”，而不是“第二个 reviewer1”。


### 6.2 风险

1. 如果输入材料准备不足，`reviewer2` 可能缺少必要事实依据。
2. 终审质量会更依赖 Orchestrator 对审核材料的组织质量。
3. 如果限制过严，`reviewer2` 可能无法发现“设计与样例不一致”的问题。

因此，这个方案的关键不是“把 reviewer2 关死”，而是：

- 只给它终审真正必需的上下文
- 去掉那些会诱导它重新勘查世界的能力


## 7. 推荐方案

我建议采用以下默认策略：

### 7.1 reviewer1

- `capability_mode = full`
- `enable_mcp = true`
- `enable_external_tools = true`
- `filesystem_mode = full`

### 7.2 reviewer2

- `capability_mode = restricted`
- `enable_mcp = false`
- `enable_external_tools = false`
- `filesystem_mode = minimal_readonly`
- `allowed_read_paths` 仅包含：
  - 主需求文件
  - `/drafts/design.md`
  - `/drafts/qa-supplement.md`（若存在）
  - 未来可选的 `/drafts/review-materials.md`


## 8. 分阶段落地建议

建议分两步实施，而不是一步做到最严格隔离。

### 第一阶段：软限制 + 配置建模

先完成：

1. 增加 reviewer2 专属配置项
2. reviewer2 不注入 MCP
3. reviewer2 不注入外部搜索工具
4. reviewer2 提示词中明确限制读取范围

这一阶段改造成本低，能先拿到大部分收益。


### 第二阶段：硬限制

若后续测试发现 reviewer2 仍然存在明显“乱翻目录”或读取越界行为，再继续做：

1. reviewer2 专属 FilesystemBackend
2. 路径白名单
3. 目录遍历限制
4. 只读虚拟工作区


## 9. 是否值得做

结论：**值得做，而且方向合理。**

但最佳落点不是：

- reviewer1 全能力
- reviewer2 完全无工具

而是：

- reviewer1 全能力
- reviewer2 最小能力、最小读取范围、无 MCP、无外部搜索

这更符合当前项目的双阶段审核目标，也更容易在工程上稳定落地。

如果接受本设计，建议后续单独产出实施文档，至少覆盖：

2. `src/config_loader.py` 扩展相应配置结构
3. `src/agent_factory.py` 按 reviewer2 配置决定其注入的 tools / MCP / backend

5. 增加相关测试，验证 reviewer2 在 restricted 模式下不会获得不应有的能力
