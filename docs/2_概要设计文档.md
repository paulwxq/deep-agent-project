# 审核员模块（Writer-Reviewer Agent）概要设计文档

> **版本**: v1.0  
> **日期**: 2026-03-21  
> **基于**: deepagents SDK v0.4.12 / langchain >=1.2.20 / langgraph >=1.1.3  
> **参考**: [Deep Agents 官方文档](https://docs.langchain.com/oss/python/deepagents/overview) · [GitHub 仓库](https://github.com/langchain-ai/deepagents) · [SDK 调研报告](./references/deepagents_sdk_research_cn.md)

## 1. 系统概述

### 1.1 目标

构建一个基于 `deepagents` SDK 的多 Agent 协作系统，实现"需求→技术设计文档"的自动化生成与质量审核。系统包含三个核心角色：

| 角色 | 职责 |
|------|------|
| **Orchestrator**（编排器/父代理） | 接收用户需求，协调 Writer 和 Reviewer 的交互循环，控制迭代轮数，输出最终文档 |
| **Writer**（子代理） | 根据需求编写可落地的技术设计文档（Markdown），接受反馈后修订 |
| **Reviewer**（子代理） | 审核设计文档的完整性、无歧义性和可落地性，给出 ACCEPT / REVISE 结论及具体反馈 |

### 1.2 核心工作流

```
用户输入需求（CLI文本 / md文件路径）
        │
        ▼
  ┌─────────────┐
  │ Orchestrator │  ← 读取需求，创建写作任务
  └──────┬──────┘
         │  task(description="需求 + 反馈", subagent_type="writer")
         ▼
  ┌─────────────┐
  │   Writer     │  ← 撰写/修订设计文档，保存到 /drafts/design.md
  └──────┬──────┘
         │  返回"草稿已完成"
         ▼
  ┌─────────────┐
  │ Orchestrator │  ← 委派审核任务
  └──────┬──────┘
         │  task(description="审核 /drafts/design.md，需求原文在 /requirements/input.md", subagent_type="reviewer")
         ▼
  ┌─────────────┐
  │  Reviewer    │  ← 审核文档，返回 ACCEPT/REVISE + 反馈
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │ Orchestrator │  ── REVISE且轮次<max → 重新委派Writer（附带反馈）
  │              │  ── ACCEPT或轮次>=max → 输出最终文档
  └─────────────┘
```

### 1.3 技术选型依据

采用 **方案 A：编排器管理的委派循环**（Orchestrator-managed delegation loop），原因如下：

1. **符合 SDK 设计哲学**：`create_deep_agent()` + `SubAgent` 字典式配置是 SDK 的一等公民用法，Orchestrator 通过 `task` 工具自然地委派子代理
2. **共享文件系统**：Writer 和 Reviewer 通过 `FilesystemBackend` 的项目文件系统（`/drafts/design.md`）交换草稿，Orchestrator 无需在上下文中传递大量文档内容
3. **灵活的迭代控制**：通过 Orchestrator 的 system_prompt 指定最大轮次（可通过配置修改），同时 Reviewer 的 ACCEPT/REVISE 判定提供自然的终止条件
4. **上下文隔离**：每次 task 调用生成独立上下文窗口，Writer 的中间思考和 Reviewer 的详细分析不会污染 Orchestrator 的上下文

> **备选方案 B（LangGraph 自定义工作流 + CompiledSubAgent）** 提供确定性更强的循环控制，但实现复杂度更高。当前需求的迭代逻辑较为直观，方案 A 足以满足。如果后续需要更精细的状态管理（如并行审核、多文档处理），可升级为方案 B。

### 2.3 数据流

| 阶段 | 发起方 | 接收方 | 传递内容 | 传递媒介 |

| 0. 需求固化 | main.py | 项目文件系统 | 用户需求原文 | Python 本地文件 IO（`Path.write_text()`）— Agent 启动前写入 |
| 1. 下发写作任务 | Orchestrator | Writer | 需求文件路径 `/requirements/input.md` + 之前的审核反馈（如有） | `task` 工具参数 |
| 2. 草稿交付 | Writer | 项目文件系统 | 设计文档 Markdown 内容 | `write_file("/drafts/design.md")` |
| 3. 下发审核任务 | Orchestrator | Reviewer | 审核指令 + 需求路径 `/requirements/input.md` + 草稿路径 `/drafts/design.md` | `task` 工具参数 |
| 4. 审核结果 | Reviewer | Orchestrator | ACCEPT/REVISE + 反馈详情 | `task` 返回值 |
| 5. 最终输出 | main.py | 用户 | 最终设计文档（直接复制 `/drafts/design.md` → `output/`） | 文件系统复制（非 Agent 响应） |

## 3. Agent 详细设计

### 3.1 Orchestrator（编排器）

**身份**：父代理（Parent Agent），由 `create_deep_agent()` 直接创建。

**职责**：
- 解析用户输入的需求文本
- 按顺序委派 Writer 和 Reviewer 子代理
- 管理迭代循环：跟踪当前轮次，判断终止条件
- 汇总并输出最终文档

**系统提示词要点**：

```
你是一个技术文档生产流程的编排器，负责协调 Writer 和 Reviewer 完成高质量技术设计文档。

关键文件路径：
- /requirements/input.md — 用户需求原文（只读，由系统在启动前写入）
- /drafts/design.md — Writer 输出的设计文档草稿（每轮更新）
- /drafts/review-verdict.json — Reviewer 的结构化结论（每轮更新）

工作流程：
1. 收到用户需求后，委派 Writer 子代理撰写设计文档，告知需求文件路径 /requirements/input.md
2. Writer 完成后，委派 Reviewer 子代理审核文档，告知需求路径 /requirements/input.md 和草稿路径 /drafts/design.md
3. 解析 Reviewer 的审核结论（详见下方解析规则）
4. 如果结论为 REVISE，将审核反馈连同需求文件路径一起发给 Writer 进行修订
5. 如果结论为 ACCEPT，或已达到最大轮次({max_iterations}轮)，结束循环
6. 返回迭代摘要（经过几轮、最终结论、关键改进点），不需要返回文档全文——最终文档由 main.py 直接从 /drafts/design.md 复制到 output/ 目录

Reviewer 结论解析规则（重要）：
Reviewer 的结论通过两个渠道传递，按优先级依次尝试：
1. **文本解析（首选）**：在 Reviewer 返回的文本中查找 "VERDICT: ACCEPT" 或 "VERDICT: REVISE"（通常在最后一行）
2. **文件兜底**：如果文本中未找到明确的 VERDICT 行，读取 /drafts/review-verdict.json 文件，解析其中的 "verdict" 字段
3. **最终兜底**：如果以上两种方式都无法确定结论，将该轮视为 REVISE，并在委派 Writer 时说明"上一轮审核结论不明确，请 Writer 基于 Reviewer 反馈内容自行判断需要改进的部分"

重要规则：
- 你自己不编写或修改设计文档，只负责编排
- 每次委派 Writer 时，必须包含所有历史审核反馈，并提醒 Writer 需求原文在 /requirements/input.md
- 每次委派 Reviewer 时，必须同时告知需求文件路径和草稿路径，确保 Reviewer 能基于需求审核文档
- 最终输出时，返回迭代摘要（经过几轮迭代、最终审核结论、关键改进点）。不需要在响应中包含文档全文——文档的唯一正式源是 /drafts/design.md，由 main.py 负责复制到 output/ 目录

任务规划（write_todos）：
在开始工作前和每个关键节点，使用 write_todos 工具维护任务列表。格式示例：
- [ ] 第1轮：委派 Writer 撰写初稿
- [ ] 第1轮：委派 Reviewer 审核初稿
- [ ] 第1轮：评估审核结论，决定是否进入下一轮
- [ ] 第2轮：委派 Writer 根据反馈修订（如需要）
- [ ] 第2轮：委派 Reviewer 审核修订稿（如需要）
- [ ] 输出最终文档

每完成一个步骤后，立即使用 write_todos 更新对应条目为已完成，并标注结果摘要（如"Reviewer 结论: REVISE，3项待改进"）。这能帮助你在多轮迭代中始终保持对全局进度的掌控。
```

**配置参数**：

| 参数 | 值 | 说明 |
|------|-----|------|
| `model` | 从 `agents.yaml` 读取 | 推荐使用推理能力强的模型，负责编排决策 |
| `tools` | `[]` | 不需要额外工具，内置 `task`/`write_todos`/文件操作已足够 |
| `subagents` | `[writer_subagent, reviewer_subagent]` | 两个专用子代理 |
| `backend` | `FilesystemBackend(root_dir=".", virtual_mode=True)` | 项目文件系统，子代理间共享 |
| `middleware` | `[LoggingMiddleware()]` | 自定义日志中间件（捕获 task 委派的完整对话内容；Writer/Reviewer 各自也需独立挂载） |
| `name` | `"orchestrator"` | 用于流式输出和日志标识 |

### 3.2 Writer（撰写者子代理）

**身份**：SubAgent（字典式配置），由 Orchestrator 通过 `task` 工具调用。

**职责**：
- 根据需求撰写完整的、可落地的技术设计文档
- 接受 Reviewer 反馈后修订文档
- 对设计方案中的技术选型给出明确的最佳建议，不留选择题给 Coder
- 将文档保存到项目文件系统 `/drafts/design.md`

**系统提示词要点**：

```
你是一位资深的技术架构师和技术文档撰写者。你的目标是根据业务需求编写能够直接指导代码开发的技术设计文档。

需求来源：
业务需求原文存放在 /requirements/input.md。开始工作前，先用 read_file 读取该文件，充分理解需求后再动笔。修订阶段如需回顾需求细节，也随时重新读取此文件。

输出要求：
1. 文档使用 Markdown 格式
2. 必须包含：背景与目标、技术方案、模块设计、数据结构、接口定义、错误处理、部署方案等章节
3. 所有技术选型必须给出明确建议和理由，不要给出"方案A或方案B供选择"的形式
4. 接口定义要包含入参、出参、异常情况的完整描述
5. 数据结构需要给出完整的字段定义和约束
6. 所有模块间的交互必须清晰描述，不能有歧义

搜索工具使用规范：
你配备了网络搜索工具，用途为**研究型搜索**：
- ✅ 搜索相关技术的官方文档、API 参考、版本信息，确保设计文档中的技术细节准确
- ✅ 查找行业最佳实践来支撑你的技术选型
- ✅ 确认第三方库/框架的能力边界和兼容性
- ❌ 不要过度搜索导致偏离写作主线，搜索应服务于具体的设计决策

修订要求：
- 收到审核反馈后，认真阅读每一条反馈
- 对于合理的建议，在文档中采纳并改进
- 你有权不采纳 Reviewer 的某项建议——如果你认为自己的方案更合理，可以保留。但你**必须**在 /drafts/design.md 末尾的"修订记录"章节中记录你的决定和理由，格式如下：

  ## 修订记录
  ### 第N轮修订
  - ✅ 已采纳：[反馈摘要] → [修改说明]
  - ❌ 未采纳：[反馈摘要] → [保留理由]

  这份记录让 Reviewer 在下一轮审核时能看到你的思考过程，避免双方在同一设计点上反复拉锯。如果你给出了充分的理由，Reviewer 应当尊重你的选择。
- 修订后将完整文档（含修订记录）重新写入 /drafts/design.md

任务规划（write_todos）：
开始撰写前，先用 write_todos 将文档章节拆分为子任务，例如：
- [ ] 分析需求，提取关键功能点
- [ ] 撰写"概述"章节（背景、目标、术语）
- [ ] 撰写"技术方案"章节（架构、选型、流程）
- [ ] 撰写"模块设计"章节（职责、交互、类/函数）
- [ ] 撰写"数据设计"章节（模型、流转）
- [ ] 撰写"接口设计"章节（签名、异常、示例）
- [ ] 撰写"非功能性设计"章节（日志、性能、安全）
- [ ] 撰写"部署与运维"章节
- [ ] 全文自检，确保无遗漏和歧义
逐步完成并更新状态。收到修订反馈时，将每条反馈项作为新的 todo 条目追踪处理进度。

将最终文档使用 write_file 工具保存到 /drafts/design.md。
```

**配置**：

```python

    "name": "writer",

    "system_prompt": WRITER_SYSTEM_PROMPT,  # 从 prompts/writer_prompt.py 加载
```

### 3.3 Reviewer（审核者子代理）

**身份**：SubAgent（字典式配置），由 Orchestrator 通过 `task` 工具调用。

**职责**：
- 从项目文件系统读取 `/requirements/input.md`（需求原文）和 `/drafts/design.md`（设计草稿），基于需求审核设计文档
- 从**需求覆盖性**、**可落地性**、**无歧义性**、**完整性**四个维度评估
- 给出 ACCEPT（通过）或 REVISE（需修订）结论
- REVISE 时提供具体、可操作的反馈清单
- 对于技术选型，可以参与讨论并给出建议，但不强制 Writer 接受
- 不直接修改文档

**系统提示词要点**：

```
你是一位严谨的技术设计文档审核专家。你的任务是基于业务需求审核技术设计文档，判断其是否足以指导代码落地。

审核前准备（必须执行）：
1. 先用 read_file 读取 /requirements/input.md，理解业务需求原文
2. 再用 read_file 读取 /drafts/design.md，获取当前设计文档
3. 对照需求逐条审核设计文档

审核维度：
1. 需求覆盖性：对照 /requirements/input.md 中的每一条需求，设计文档是否都有对应的技术方案？是否有遗漏的需求点？
2. 可落地性：Coder 能否仅凭此文档完成开发？是否有遗漏的实现细节？
3. 无歧义性：是否存在含糊的描述？是否有"可以选择A或B"这样的未决策项？
4. 完整性：接口、数据结构、错误处理是否齐全？
5. 合理性：技术方案是否合理？架构是否合适？是否有过度设计或设计不足？

审核规则：
- 需求覆盖性是最高优先级：如果有需求点未被设计覆盖，必须标记为 REVISE，并明确指出遗漏的需求条目
- 对于明确的问题（缺失内容、歧义描述、接口不完整），必须标记为 REVISE 并给出具体修改建议
- 对于技术选型的讨论，你可以给出自己的建议，但如果 Writer 的方案本身合理可行，你应该尊重 Writer 的选择
- Writer 不需要接受你的所有建议，只要最终文档满足"需求全覆盖、可落地、无歧义、完整"的标准
- 如果文档整体已达到可指导代码落地的标准，即使有小的改进空间，也应该判定为 ACCEPT

处理 Writer 的修订记录：
- 文档末尾可能包含"修订记录"章节，记录了 Writer 对你上一轮反馈的处理结果（已采纳/未采纳及理由）
- 对于 Writer 标记为"未采纳"并给出充分理由的条目，如果其方案确实合理可行，你应当接受 Writer 的决定，**不要**在本轮重复提出相同的建议
- 只有当 Writer 的理由明显不成立（如事实错误、逻辑矛盾），才应再次标记为 REVISE 并说明为什么 Writer 的理由不充分

搜索工具使用规范：
你配备了网络搜索工具，但用途严格限定为**验证型搜索**：
- ✅ 验证文档中提到的库/框架是否存在、版本号是否正确、API 是否如文档所述
- ✅ 核实特定技术方案的可行性（如"Redis 是否支持某特性"）
- ❌ 不要用搜索去寻找替代方案来推翻 Writer 的选型
- ❌ 不要做开放式的技术调研
搜索结果应作为审核反馈的事实依据，例如："文档中声称库X支持特性Y，经搜索验证，该特性在v3.0后已废弃，建议Writer核实。"

输出格式（严格遵守）：

你的最终输出**必须**包含以下结构。结论行的格式是 Orchestrator 判定流程是否继续的唯一依据，必须精确匹配。

**结论行格式（必须出现且精确匹配）**：

VERDICT: ACCEPT

或

VERDICT: REVISE

⚠️ 结论行必须独占一行，以 "VERDICT: " 开头（注意冒号后有一个空格），后跟 ACCEPT 或 REVISE（全大写）。不要使用任何其他变体。

正确示例：
- VERDICT: ACCEPT
- VERDICT: REVISE

错误示例（不要这样写）：
- ## 审核结论：ACCEPT       ← 不要用中文冒号或 Markdown 标题
- I accept this document    ← 不要用自然语言表述
- Verdict: Accept           ← 不要用小写
- VERDICT:ACCEPT            ← 冒号后必须有空格

**完整输出模板**：

## 审核详情

### 需求覆盖性：[全覆盖/有遗漏]
- 逐条核对 /requirements/input.md，列出覆盖情况
- 如有遗漏：需求第X条"[需求摘要]" → 设计文档中未找到对应方案

### 可落地性：[通过/需改进]
- 具体问题和建议...

### 无歧义性：[通过/需改进]
- 具体问题和建议...

### 完整性：[通过/需改进]
- 具体问题和建议...

### 合理性：[通过/需改进]
- 具体问题和建议...

## 建议（非强制）
- 可以改进但不影响通过的建议...

VERDICT: ACCEPT

注意：VERDICT 行放在输出的**最后一行**，方便 Orchestrator 可靠提取。

结论文件输出（必须执行）：
审核完成后，在输出文本结论的同时，**必须**使用 write_file 将结构化结论写入 /drafts/review-verdict.json，格式如下：
{"verdict": "ACCEPT", "summary": "文档整体达标，可指导代码落地"}
或
{"verdict": "REVISE", "summary": "3项问题需修订：...", "issues": ["问题1", "问题2", "问题3"]}
此文件作为 Orchestrator 判定结论的兜底依据，确保即使文本解析失败也能可靠获取结论。

任务规划（write_todos）：
审核开始前，用 write_todos 创建审核检查清单，例如：
- [ ] 读取 /requirements/input.md（需求原文）
- [ ] 读取 /drafts/design.md（设计文档）
- [ ] 检查需求覆盖性（逐条核对需求，确认每条都有对应设计）
- [ ] 检查可落地性（实现细节是否充分）
- [ ] 检查无歧义性（是否有未决策项）
- [ ] 检查完整性（接口、数据结构、错误处理）
- [ ] 检查合理性（架构适配度、选型依据）
- [ ] 汇总结论，输出 VERDICT 行并写入 review-verdict.json
逐项完成并更新状态，确保审核过程系统化、不遗漏。

首先使用 read_file 工具分别读取 /requirements/input.md 和 /drafts/design.md，然后对照审核。
不要修改设计文档本身。
```

**配置**：

```python
reviewer_subagent = {
```

### 3.4 TodoListMiddleware 任务规划策略

`TodoListMiddleware` 是 deepagents SDK 四大核心支柱之一（Planning），灵感来自 Claude Code 的 `TodoWrite` 上下文工程技术。它通过强制 LLM 创建和维护结构化任务列表，使 Agent 在多轮工具调用中保持方向感，不会遗漏步骤或偏离目标。

**SDK 内置行为**：`TodoListMiddleware` 由 SDK **自动注入**到所有 Agent（包括主代理和每个子代理）的中间件栈中，排在第一位。无需在 `middleware` 参数中显式声明。它向每个 Agent 提供两个内置工具：

| 工具 | 功能 |
|------|------|
| `write_todos` | 创建或更新结构化任务列表，每个条目包含描述和完成状态 |
| `read_todos` | 读取当前任务列表，用于回顾进度和决策下一步 |

**各 Agent 的 TodoList 使用场景**：

| Agent | 使用时机 | 典型 Todo 内容 |
|-------|----------|---------------|
| **Orchestrator** | 收到需求后立即创建全局迭代计划；每个关键节点（Writer完成、Reviewer返回结论）后更新状态 | `第1轮：委派Writer → 委派Reviewer → 评估结论`；`第2轮：…` |
| **Writer** | 开始撰写前将文档章节拆分为子任务；收到修订反馈时将每条反馈作为新 todo 条目 | `撰写"概述"章节`；`撰写"接口设计"章节`；`处理反馈：补充错误码定义` |
| **Reviewer** | 开始审核前创建审核检查清单；逐项检查后更新状态 | `检查可落地性`；`检查无歧义性`；`检查完整性`；`汇总结论` |

**上下文工程价值**：在本项目的多轮 Writer-Reviewer 迭代场景中，`TodoListMiddleware` 的价值尤为突出：

1. **Orchestrator 的全局进度追踪**：Orchestrator 需要在最多 N 轮迭代中保持清晰的状态感知——当前是第几轮、Writer 是否完成、Reviewer 的结论是什么。`write_todos` 将这些信息结构化地固化在上下文中，避免 LLM 在长对话中"忘记"当前进度。
2. **Writer 的章节化写作**：复杂需求可能涉及多个章节的设计文档。Writer 通过 `write_todos` 将写作任务拆分为独立章节，确保每个章节都被覆盖，特别是在处理修订反馈时，能逐条追踪每个反馈项的处理状态。
3. **Reviewer 的系统化审核**：Reviewer 需要从多个维度审核文档。通过 `write_todos` 创建检查清单，确保审核过程不遗漏任何维度，并为最终结论提供结构化依据。
4. **跨轮次的连续性**：虽然每次 `task` 调用会创建独立的子代理上下文，但 Orchestrator 的 todo list 在整个运行期间持续存在，提供跨轮次的状态连续性。

**注意事项**：每个 Agent 的 todo list 是**上下文隔离**的——子代理（Writer/Reviewer）的 todo list 对 Orchestrator 不可见，反之亦然。Orchestrator 只能通过 `task` 工具返回值获取子代理的执行结果，而非直接读取子代理的 todo list。

### 3.5 迭代控制策略

| 参数 | 默认值 | 配置位置 | 说明 |
|------|--------|----------|------|
| `max_iterations` | 3 | `agents.yaml` → `global.max_iterations` | 最大 Writer-Reviewer 交互轮次 |
| 终止条件 | — | Orchestrator system_prompt | Reviewer 返回 ACCEPT **或** 达到 max_iterations |
| 超限处理 | — | Orchestrator system_prompt | 达到上限后，输出当前版本并附带最后一轮审核意见 |

Orchestrator 的 `recursion_limit` 由 SDK 默认设置为 1000，足以支撑 3 轮迭代（每轮约涉及十余次工具调用）。

### 3.6 Reviewer 结论传递协议

**问题背景**：Reviewer 的审核结论通过 `task` 工具返回给 Orchestrator。虽然 deepagents SDK 的子代理**支持** `response_format`（Structured Output）来约束最终响应格式，但当前设计选择不使用该功能（原因见下方分析），而是通过提示词约束 + 文件兜底来确保结论的可靠传递。这存在 LLM 吐词不稳定导致解析失败的风险（例如写成 "I accept this" 而非标准的 `VERDICT: ACCEPT`），因此需要多层降级保障。

**双通道 + 三级降级策略**：

```
Reviewer 输出
    ├── 通道 1：文本中的 VERDICT 行（最后一行）
    │       ↓ 解析失败
    ├── 通道 2：/drafts/review-verdict.json 文件
    │       ↓ 文件不存在或解析失败
    └── 通道 3：视为 REVISE（安全兜底）
```

| 层级 | 策略 | 实施位置 | 说明 |
|------|------|----------|------|
| **L1：格式约束** | Reviewer 提示词中用正反例严格定义 `VERDICT: ACCEPT/REVISE` 格式 | 3.3 节 Reviewer 系统提示词 | 防止 LLM 用自然语言或非标准变体表述结论 |
| **L2：文件兜底** | Reviewer 同时将结论写入 `/drafts/review-verdict.json`（结构化 JSON） | 3.3 节 Reviewer 系统提示词 | 文本解析失败时的备用数据源 |
| **L3：弹性解析** | Orchestrator 按优先级尝试文本解析 → 文件读取 → 默认 REVISE | 3.1 节 Orchestrator 系统提示词 | 确保任何情况下都能得出结论，不会阻塞流程 |

**`/drafts/review-verdict.json` 文件格式**：

```json
{
  "verdict": "REVISE",
  "summary": "3项问题需修订",
  "issues": [
    "缺少错误处理的详细设计",
    "接口定义中参数类型未明确",
    "部署方案未覆盖回滚策略"
  ]
}
```

**为什么当前不使用 Structured Output（`response_format`）？**

deepagents SDK 的子代理**支持** `response_format` 参数，可以约束子代理的**最终响应**为结构化 JSON（中间的工具调用流程不受影响）。但当前设计选择不使用，基于以下工程权衡：

1. **Provider 兼容性风险**：本项目支持 DashScope、MiniMaxi、BigModel、OpenRouter 四个 Provider，各家对 `response_format` 的支持程度不一（尤其是 API 兼容模式下的 MiniMaxi 和 BigModel），引入 Structured Output 可能导致 Provider 切换时出现不可预期的行为差异
2. **反馈表达灵活性**：Reviewer 的审核反馈包含丰富的上下文（引用文档具体段落、对比需求条目、给出修改建议），自由文本比固定 JSON schema 更适合承载这类非结构化的反馈内容
3. **当前方案已足够健壮**：三级降级策略（提示词约束 + 文件兜底 + 默认 REVISE）提供了可比的可靠性保障，且不依赖任何 Provider 特性
4. **实现复杂度**：Structured Output 需要定义 Pydantic schema 并修改 Orchestrator 的结果解析逻辑，增加了代码复杂度

> **未来优化方向**：如果实际运行中发现文本解析的失败率过高（特别是某些模型不遵循提示词中的 VERDICT 格式约束），应认真评估 Structured Output 方案。届时可为 Reviewer 子代理配置 `response_format` 使其最终响应返回 `{"verdict": "ACCEPT"|"REVISE", "feedback": "..."}` 结构，这是 SDK 原生支持的能力，可以从根本上消除解析不确定性。

**安全兜底的设计哲学**：L3 层将不明确的结论默认视为 REVISE 而非 ACCEPT，因为"多审一轮"的代价远小于"误判通过导致产出低质量文档"。

## 4. LLM 配置方案

### 4.1 设计原则

- **国内优先**：运行环境在中国大陆，优先使用可直连的国内模型提供商（DashScope / MiniMaxi / BigModel），OpenRouter 作为海外模型补充通道
- **YAML 驱动**：所有 Provider 和 Agent 的模型配置集中在 `config/agents.yaml` 文件中
- **按角色优化**：不同 Agent 可以使用不同提供商和模型，根据角色特点选择最合适的组合
- **统一抽象**：通过模型工厂函数屏蔽各提供商接入差异（原生支持 vs API 兼容模式），对 Agent 层透明

### 4.2 Provider 接入方案

项目支持四个模型提供商，接入方式各不相同：

#### a) DashScope（阿里云百炼）—— LangChain 原生支持

LangChain 通过 `langchain-qwq` 包提供了对 DashScope 的**原生集成**，完整支持工具调用（tool calling）、结构化输出、流式输出和异步调用。

> **关于包名**：`langchain-qwq` 的名称源于最初对 QwQ 推理模型的支持，但该包目前提供了两个类：`ChatQwQ`（专用于 QwQ/QvQ 推理模型）和 `ChatQwen`（用于所有 Qwen 系列模型）。LangChain 官方文档（python.langchain.com/docs/integrations/chat/qwen/）将其作为 Qwen/DashScope 的**推荐集成方案**。本项目使用 `ChatQwen` 类。目前不存在独立的 `langchain-qwen` 包。社区包中的 `ChatTongyi` 可作为备选，但功能更新不如 `langchain-qwq` 活跃。

| 项目 | 值 |
|------|-----|
| LangChain 类 | `ChatQwen`（来自 `langchain-qwq` 包） |
| 可用模型 | `qwen3-max`、`qwen-plus`、`qwen-turbo`、`qwen-flash` 等 |
| 环境变量 | `DASHSCOPE_API_KEY`（必需）、`DASHSCOPE_API_BASE`（国内用户必需） |
| pip 包 | `langchain-qwq` (v0.3.4+) |

```python
from langchain_qwq import ChatQwen

model = ChatQwen(
    model="qwen3-max",
    temperature=0.3,
    max_tokens=8192,
    max_retries=6,
)
```

> **⚠️ 国内端点配置**：`langchain-qwq` 默认使用国际端点（`https://dashscope-intl.aliyuncs.com/compatible-mode/v1`）。中国大陆环境**必须**设置 `DASHSCOPE_API_BASE` 环境变量指向国内端点，否则会出现网络连接问题：
> ```bash
> DASHSCOPE_API_BASE=https://dashscope.aliyuncs.com/compatible-mode/v1
> ```

> **推荐作为主力 Provider**：DashScope 在国内直连、有原生 LangChain 集成、Qwen 系列模型支持完整的 tool calling，最适合作为 Orchestrator/Writer/Reviewer 的默认模型。

#### b) MiniMaxi —— Anthropic API 兼容模式

LangChain 社区包中的 `MiniMaxChat` 仅支持旧版模型（abab5 系列），不支持 MiniMax-M2.5/M2.7。MiniMax 官方推荐使用 **Anthropic API 兼容模式**，通过 `ChatAnthropic` + 自定义 `base_url` 接入。

| 项目 | 值 |
|------|-----|
| LangChain 类 | `ChatAnthropic`（来自 `langchain-anthropic` 包） |
| 接入模式 | Anthropic API 兼容（自定义 `base_url`） |
| 可用模型 | `MiniMax-M2.7`、`MiniMax-M2.5` |
| 环境变量 | `MINIMAX_API_KEY`（设置到 `ANTHROPIC_API_KEY`） |
| pip 包 | `langchain-anthropic` |
| API 端点 | `https://api.minimaxi.com/anthropic` |

```python
from langchain_anthropic import ChatAnthropic

model = ChatAnthropic(
    model="MiniMax-M2.7",
    base_url="https://api.minimaxi.com/anthropic",
    api_key=os.environ["MINIMAX_API_KEY"],
    temperature=0.3,
    max_tokens=8192,
)
```

> **注意**：使用 Anthropic 兼容模式时，`api_key` 需要显式传入 MiniMax 的密钥（而非 Anthropic 的），避免与真正的 `ANTHROPIC_API_KEY` 环境变量冲突。模型工厂函数中会处理这一细节。

#### c) BigModel（智谱 AI）—— OpenAI API 兼容模式

智谱 AI 同时支持 OpenAI 和 Anthropic 兼容模式。选择 **OpenAI API 兼容模式**，因为 `ChatOpenAI` 的 `base_url` 支持更成熟稳定。

| 项目 | 值 |
|------|-----|
| LangChain 类 | `ChatOpenAI`（来自 `langchain-openai` 包） |
| 接入模式 | OpenAI API 兼容（自定义 `base_url`） |
| 可用模型 | `glm-5`、`glm-4.7` |
| 环境变量 | `ZHIPUAI_API_KEY`（设置到 `OPENAI_API_KEY`） |
| pip 包 | `langchain-openai` |
| API 端点 | `https://open.bigmodel.cn/api/paas/v4/` |

```python
from langchain_openai import ChatOpenAI

model = ChatOpenAI(
    model="glm-5",
    base_url="https://open.bigmodel.cn/api/paas/v4/",
    api_key=os.environ["ZHIPUAI_API_KEY"],
    temperature=0.3,
    max_tokens=8192,
)
```

> **注意**：同 MiniMax，`api_key` 需要显式传入智谱的密钥，避免与真正的 `OPENAI_API_KEY` 环境变量冲突。

#### d) OpenRouter —— 海外模型代理通道

通过 OpenRouter 访问 OpenAI、Claude 等海外模型。LangChain 支持 `init_chat_model("openrouter:model")` 原生格式。

| 项目 | 值 |
|------|-----|
| LangChain 类 | 通过 `init_chat_model()` 自动解析，底层为 `ChatOpenAI` |
| 接入模式 | LangChain 原生 `"openrouter:"` 前缀 |
| 可用模型 | `openai/gpt-5.2`、`anthropic/claude-sonnet-4-6` 等 |
| 环境变量 | `OPENROUTER_API_KEY` |
| pip 包 | `langchain-openai` |

```python
from langchain.chat_models import init_chat_model

model = init_chat_model(
    "openrouter:openai/gpt-5.2",
    temperature=0.3,
    max_tokens=8192,
)
```

### 4.3 Provider 对照总结

| Provider | 接入方式 | LangChain 类 | pip 包 | API 端点 | 环境变量 |
|----------|----------|-------------|--------|----------|----------|
| **DashScope** | 原生集成 | `ChatQwen` | `langchain-qwq` | DashScope 官方（需配置国内端点） | `DASHSCOPE_API_KEY` + `DASHSCOPE_API_BASE` |
| **MiniMaxi** | Anthropic 兼容 | `ChatAnthropic` + `base_url` | `langchain-anthropic` | `https://api.minimaxi.com/anthropic` | `MINIMAX_API_KEY` |
| **BigModel** | OpenAI 兼容 | `ChatOpenAI` + `base_url` | `langchain-openai` | `https://open.bigmodel.cn/api/paas/v4/` | `ZHIPUAI_API_KEY` |
| **OpenRouter** | 原生前缀 | `init_chat_model("openrouter:...")` | `langchain-openai` | OpenRouter 官方 | `OPENROUTER_API_KEY` |

### 4.4 YAML 配置文件结构

文件路径：`config/agents.yaml`

```yaml
# 全局配置
global:
  max_iterations: 3
  output_dir: "./output"
  log_level: "DEBUG"

# ========================================
# Provider 定义：声明可用的模型提供商
# ========================================
providers:
  dashscope:
    type: "dashscope"                  # 使用 ChatQwen 原生集成（langchain-qwq 包）
    api_key_env: "DASHSCOPE_API_KEY"
    base_url_env: "DASHSCOPE_API_BASE" # 国内端点，从环境变量读取

  minimax:
    type: "anthropic_compatible"       # 使用 ChatAnthropic + 自定义 base_url
    base_url: "https://api.minimaxi.com/anthropic"
```

```python
# src/model_factory.py 伪代码

import os
from langchain_core.language_models import BaseChatModel

def create_model(provider_config: ProviderConfig, agent_config: AgentModelConfig) -> BaseChatModel:
    """根据 Provider 类型和 Agent 配置创建 LangChain 模型实例。
    
    通过 Provider type 字段路由到不同的模型创建策略，
    对上层调用者屏蔽原生集成 vs API 兼容模式的差异。
    """
    api_key = os.environ[provider_config.api_key_env]
    params = agent_config.params

    match provider_config.type:
        case "dashscope":
            from langchain_qwq import ChatQwen
            base_url = os.environ.get(provider_config.base_url_env) if provider_config.base_url_env else None
            return ChatQwen(
                model=agent_config.model,
                api_key=api_key,
                **({"base_url": base_url} if base_url else {}),
                **params,
            )

        case "anthropic_compatible":
            from langchain_anthropic import ChatAnthropic
            return ChatAnthropic(
                model=agent_config.model,
                base_url=provider_config.base_url,
                api_key=api_key,    # 使用 Provider 自己的 key，非 ANTHROPIC_API_KEY
                **params,
            )

        case "openai_compatible":
            from langchain_openai import ChatOpenAI
            return ChatOpenAI(
                model=agent_config.model,
                base_url=provider_config.base_url,
                api_key=api_key,    # 使用 Provider 自己的 key，非 OPENAI_API_KEY
                **params,
            )

        case "openrouter":
            from langchain.chat_models import init_chat_model
            return init_chat_model(
                model=f"openrouter:{agent_config.model}",
                **params,
            )

        case _:
            raise ValueError(f"不支持的 provider type: {provider_config.type}")
```

### 4.6 关键设计说明

**为什么不统一使用 `init_chat_model("provider:model")`？**

`init_chat_model()` 的 `"provider:model"` 格式虽然简洁，但存在以下局限性：

1. **不支持自定义 `base_url`**：MiniMax 和 BigModel 的兼容模式需要指定自定义 API 端点，`init_chat_model()` 不直接支持这种透传
2. **API Key 冲突**：MiniMax 走 Anthropic 兼容、BigModel 走 OpenAI 兼容，它们各自的 API Key 会与真正的 `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` 环境变量冲突。必须通过构造函数的 `api_key` 参数显式传入
3. **DashScope 的特殊性**：`ChatQwen` 来自 `langchain-qwq` 包（尽管包名看似仅针对 QwQ 推理模型，实则提供通用的 `ChatQwen` 类），不在 `init_chat_model()` 的标准 provider 映射中。且该包默认使用国际端点，国内用户需额外配置 `DASHSCOPE_API_BASE`

因此采用**模型工厂模式**：通过 YAML 中的 `providers.type` 字段路由到对应的模型创建策略，在工厂函数内部处理各 Provider 的接入差异，对 Agent 层完全透明。

**API Key 隔离策略**：

| Provider | 实际使用的 LangChain 类 | API Key 来源 | 如何避免冲突 |
|----------|------------------------|-------------|-------------|
| DashScope | `ChatQwen` | `DASHSCOPE_API_KEY` | 独立环境变量，无冲突 |
| MiniMaxi | `ChatAnthropic` | `MINIMAX_API_KEY` | 通过构造函数 `api_key=` 显式传入，不读取 `ANTHROPIC_API_KEY` |
| BigModel | `ChatOpenAI` | `ZHIPUAI_API_KEY` | 通过构造函数 `api_key=` 显式传入，不读取 `OPENAI_API_KEY` |
| OpenRouter | `init_chat_model()` | `OPENROUTER_API_KEY` | `init_chat_model` 原生支持 OpenRouter 路由 |

### 5.4 Agent 对话日志中间件

通过完整的 `AgentMiddleware` 类同时拦截**模型调用**和**工具调用**两个层面，既记录 LLM 的可见输出内容，也记录工具调用和 Agent 间通信。

> **重要**：`middleware` 在 deepagents SDK 中**不会被子代理继承**（参见 SDK 继承规则）。因此必须为 Orchestrator、Writer、Reviewer **各自显式挂载** `LoggingMiddleware` 实例，否则子代理的内部操作（read_file、write_file、搜索、LLM 推理）不会被捕获。

```python
# src/middleware/logging_middleware.py 伪代码

from deepagents.middleware import AgentMiddleware
import logging

logger = logging.getLogger("deep_agent_project")

MAX_CONTENT_PREVIEW = 800
MAX_TASK_MESSAGE_LOG = 2000
MAX_TASK_RESULT_LOG = 2000

class LoggingMiddleware(AgentMiddleware):
    """记录 Agent 的模型输出、工具调用和子代理通信。

    每个 Agent（Orchestrator / Writer / Reviewer）都需要独立挂载本中间件。

    使用 SDK 标准的四个钩子中的三个：
    - wrap_model_call：环绕式钩子，在 handler 调用后记录模型的可见输出
    - before_tool_call：工具执行前记录工具名称及 task 委派内容
    - after_tool_call：工具执行后记录 task 子代理返回结果
    """

    def wrap_model_call(self, request, handler):
        """环绕式钩子：执行模型调用，并在调用后记录可见输出。

        注意：捕获的是模型的可见输出文本，不是内部思维链。
        纯工具调用回合可能没有文本内容。
        """
        response = handler(request)

        agent_name = getattr(request, "agent_name", None) or "unknown"
        last_msg = response.messages[-1] if hasattr(response, "messages") and response.messages else None

        if last_msg and hasattr(last_msg, "content") and last_msg.content:
            content = last_msg.content
            if len(content) <= MAX_CONTENT_PREVIEW:
                logger.debug(f"📝 模型输出: {content}", extra={"agent_name": agent_name})
            else:
                logger.debug(
                    f"📝 模型输出（前{MAX_CONTENT_PREVIEW}字符）: {content[:MAX_CONTENT_PREVIEW]}",
                    extra={"agent_name": agent_name},
                )
                logger.debug(
                    f"📝 模型输出（完整，{len(content)}字符）: {content}",
                    extra={"agent_name": agent_name},
                )

        if last_msg and hasattr(last_msg, "tool_calls") and last_msg.tool_calls:
            tool_names = [tc["name"] for tc in last_msg.tool_calls]
            logger.debug(
                f"🔧 工具调用意图: {', '.join(tool_names)}",
                extra={"agent_name": agent_name},
            )

        return response

    def before_tool_call(self, tool_call, runtime):
        """工具执行前：记录工具名称，task 工具额外记录完整委派内容"""
        agent_name = runtime.agent_name or "unknown"
        tool_name = tool_call.get("name", "unknown")

        if tool_name != "task":
            logger.debug(f"工具执行: {tool_name}", extra={"agent_name": agent_name})
            return

        # task 工具：记录 Agent 间对话（委派内容）
        # SDK 源码中 task 工具的参数名为 subagent_type（子代理名）和 description（任务内容）
        args = tool_call.get("args", {})
        target = args.get("subagent_type", "unknown")
        task_message = args.get("description", "")
        logger.info(
            f"📤 [{agent_name}→{target}] 委派任务",
            extra={"agent_name": agent_name},
        )
        logger.info(
            f"📤 [{agent_name}→{target}] 任务内容: {task_message[:MAX_TASK_MESSAGE_LOG]}",
            extra={"agent_name": agent_name},
        )

    def after_tool_call(self, tool_call, result, runtime):
        """工具执行后：task 工具记录子代理返回的完整结果"""
        tool_name = tool_call.get("name", "unknown")
        if tool_name != "task":
            return

        agent_name = runtime.agent_name or "unknown"
        args = tool_call.get("args", {})
        target = args.get("subagent_type", "unknown")
        result_text = str(result)
        logger.info(
            f"📥 [{target}→{agent_name}] 返回结果: {result_text[:MAX_TASK_RESULT_LOG]}",
            extra={"agent_name": agent_name},
        )
        if len(result_text) > MAX_TASK_RESULT_LOG:
            logger.debug(
                f"📥 [{target}→{agent_name}] 返回结果（完整，{len(result_text)}字符）: {result_text}",
                extra={"agent_name": agent_name},
            )
```

**日志记录的四个层面**：

| 层面 | 钩子 | 日志级别 | 记录内容 | 覆盖范围 | 调试价值 |
|------|------|---------|----------|----------|----------|
| **模型输出** | `wrap_model_call`（handler 调用后） | DEBUG | LLM 每次返回的可见文本内容（短输出完整记录，长输出截断 + 完整两条） | 全部 Agent（各自挂载） | 辅助理解 Agent 的决策方向，但不等同于内部思维链 |
| **工具调用** | `before_tool_call` | DEBUG | 工具名称（read_file、write_file、web_search 等） | 全部 Agent（各自挂载） | 追踪文件读写、搜索调用等操作细节 |
| **Agent 间对话** | `before_tool_call` + `after_tool_call`（`task` 工具） | INFO | 委派的完整任务内容（需求 + 反馈文本）和子代理返回的完整结果 | Orchestrator | 还原 Agent 间的真实对话，满足"看到 Agent 之间交流"的需求 |
| **子代理内部** | `wrap_model_call` + `before_tool_call` | DEBUG/INFO | Writer/Reviewer 自身的模型输出、文件操作、搜索过程 | Writer、Reviewer（各自挂载） | 追踪子代理的完整执行链路，排查"Writer 为什么没有采纳某条反馈" |

> **middleware 不继承**：deepagents SDK 中子代理的 `middleware` 不从父代理继承，必须在每个子代理的定义中显式挂载 `LoggingMiddleware` 实例。这是覆盖"子代理内部"日志层面的前提。
>
> **设计说明**：模型输出日志使用 DEBUG 级别，生产运行时可通过 `log_level: INFO` 关闭，仅在需要调试提示词效果时开启。内容截取前 800 字符以避免日志文件膨胀。Agent 间对话使用 INFO 级别，始终记录——这是需求明确要求的"多个 agent/subagent 之间的交流"。
>
> **关于 `wrap_model_call` 的能力边界**：该钩子在 `handler(request)` 返回后捕获的是 LLM 每次返回的**可见输出文本**，不是模型的内部思维链（reasoning trace）。在纯工具调用回合中，模型可能只返回工具调用指令而没有文本内容，此时该日志不会产出。因此这一层日志的价值是"辅助理解 Agent 的决策方向"，而非"完整还原推理过程"。

### 5.5 日志模块初始化

```python
# src/logger.py 伪代码

import logging
from logging.handlers import RotatingFileHandler
from pathlib import Path

def setup_logger(log_level: str = "DEBUG", log_dir: str = "logs"):
    """初始化日志系统"""
    Path(log_dir).mkdir(parents=True, exist_ok=True)

    logger = logging.getLogger("deep_agent_project")
    logger.setLevel(logging.DEBUG)

    file_handler = RotatingFileHandler(
        filename=f"{log_dir}/app.log",
        maxBytes=10 * 1024 * 1024,  # 10MB
        backupCount=5,
        encoding="utf-8",
    )
    file_handler.setLevel(logging.DEBUG)

    console_handler = logging.StreamHandler()
    console_handler.setLevel(getattr(logging, log_level.upper()))

    formatter = logging.Formatter(
        fmt="[%(asctime)s.%(msecs)03d] [%(levelname)s] [%(agent_name)s] %(message)s",
        datefmt="%Y-%m-%d %H:%M:%S",
        defaults={"agent_name": "system"},
    )
    file_handler.setFormatter(formatter)
    console_handler.setFormatter(formatter)

    logger.addHandler(file_handler)
    logger.addHandler(console_handler)

    return logger
```

```markdown
# 技术设计文档审核检查清单

## 审核准备

审核前**必须**先读取需求原文 /requirements/input.md，再读取设计文档 /drafts/design.md，对照审核。

## 审核维度

### 1. 需求覆盖性检查（对照 /requirements/input.md）
- [ ] 逐条核对：需求文档中的每一条功能需求，在设计文档中是否有对应的技术方案？
- [ ] 非功能需求（日志、配置、部署、多环境支持等）是否都有设计覆盖？
- [ ] 对于长需求文档，使用 grep 工具在 /drafts/design.md 中搜索需求关键词，辅助确认覆盖情况（避免仅凭 read_file 后的记忆遗漏中间段落）
- [ ] 如发现需求遗漏，明确列出"需求第X条：[需求摘要] → 设计文档中未找到对应方案"

### 2. 可落地性检查
- [ ] Coder 能否仅凭此文档完成开发？
- [ ] 所有模块的入口函数/类是否已定义？
- [ ] 关键算法/流程是否有伪代码或步骤描述？
- [ ] 第三方依赖是否已明确版本？
- [ ] 配置项是否列出了完整的 key-value 结构？

### 3. 无歧义性检查
- [ ] 是否存在"方案A或方案B"式的未决策项？
- [ ] 接口参数类型和约束是否明确？
- [ ] 模块边界和职责是否清晰？
- [ ] 异常处理策略是否确定？

### 4. 完整性检查
- [ ] 接口设计是否包含入参、出参、异常？
- [ ] 数据模型是否包含字段、类型、约束？
- [ ] 模块间交互是否有完整的数据流描述？

### 5. 合理性检查
- [ ] 架构复杂度是否与需求匹配（不过度设计）？
- [ ] 技术选型是否有合理依据？
- [ ] 模块划分是否职责单一？

## 审核原则

1. **尊重 Writer 的技术判断**：如果 Writer 的方案合理可行，即使不是你的首选，也应放行
2. **区分"必须改"和"建议改"**：必须改的问题归入 REVISE 理由，建议改的放入非强制建议
3. **给出具体建议**：不要只说"不够详细"，要说清楚缺什么、怎么改
4. **标准是"能指导代码落地"**：不追求完美，达到标准就放行
```

### 6.4 技能加载与隔离机制

本项目使用 `FilesystemBackend`，Agent 可直接从磁盘读取技能文件，**无需手动注入**。

**加载方式**：

由于 `FilesystemBackend(root_dir=".", virtual_mode=True)` 将项目根目录映射为文件系统根路径 `/`，Agent 可直接通过路径访问磁盘上的 `./skills/` 目录。只需在 `create_deep_agent()` 或子代理定义中通过 `skills` 参数指定技能目录即可：

> **前提**：`root_dir="."` 是相对于进程的 cwd 解析的。`main.py` 在启动时通过 `os.chdir(project_root)` 将 cwd 固定为项目根目录（见 8.3 节），确保 Agent 视角下的 `/skills/tech-doc-writing/` 始终对应磁盘上的 `{项目根}/skills/tech-doc-writing/`。

```python
# 子代理定义中直接指定技能路径，SDK + FilesystemBackend 自动从磁盘加载
writer_subagent = {
    "name": "writer",
    "skills": ["/skills/tech-doc-writing/"],  # 映射到 ./skills/tech-doc-writing/
    ...
}

reviewer_subagent = {
    "name": "reviewer",
    "skills": ["/skills/tech-doc-review/"],   # 映射到 ./skills/tech-doc-review/
    ...
}
```

与 `StateBackend` 方案相比，**不需要** `load_skills_files()` 函数、`create_file_data()` 转换、或 `invoke(files={...})` 注入。技能文件修改后直接生效，无需重新构建。

**隔离机制**：

SDK 在构建每个子代理时，根据其 `skills` 参数创建 **独立的 `SkillsMiddleware` 实例**，只扫描对应目录：

```python
# SDK 源码（graph.py）中的关键逻辑：
subagent_skills = spec.get("skills")
if subagent_skills:
    subagent_middleware.append(SkillsMiddleware(backend=backend, sources=subagent_skills))
```

效果如下：

| Agent | `skills` 参数 | `SkillsMiddleware` 扫描范围 | 系统提示词中注入的技能元数据 |
|-------|--------------|---------------------------|--------------------------|
| Orchestrator | 未设置 | 不挂载，无技能 | 无 |
| Writer | `["/skills/tech-doc-writing/"]` | 仅 `/skills/tech-doc-writing/` | 仅 `tech-doc-writing` 的 frontmatter |
| Reviewer | `["/skills/tech-doc-review/"]` | 仅 `/skills/tech-doc-review/` | 仅 `tech-doc-review` 的 frontmatter |

此外，SDK 在子代理状态传递时通过 `_EXCLUDED_STATE_KEYS` 显式排除 `skills_metadata`，防止父代理的技能状态泄露到子代理（反之亦然）。

**总结**：`FilesystemBackend` 让技能加载零样板代码——技能文件存放在磁盘上的 `./skills/` 目录，每个 Agent 的 `SkillsMiddleware` 只会扫描自己 `skills` 参数指定路径下的技能，发现、加载和状态在 Agent 间完全隔离。

## 7. 工具集成

### 7.1 内置工具（无需额外配置）

deepagents SDK 自动提供以下工具，无需在 `tools` 参数中声明：

| 工具 | 用途 | 使用方 |
|------|------|--------|
| `write_todos` / `read_todos` | 任务规划与进度追踪（SDK 自动注入 `TodoListMiddleware` 提供，详见 3.4 节） | Orchestrator, Writer, Reviewer（全部 Agent 自动可用） |
| `read_file` | 读取项目文件系统中的文件 | Writer（读需求）、Reviewer（读需求 + 读草稿） |
| `write_file` | 写入项目文件系统中的文件 | Writer（保存草稿）、Reviewer（写结论 JSON） |
| `edit_file` | 编辑已有文件 | Writer（修订草稿） |
| `ls` / `glob` / `grep` | 文件浏览与搜索 | 全部 |
| `task` | 委派子代理 | Orchestrator |

### 7.2 可选工具：Tavily Web Search

当 `agents.yaml` 中 `tools.tavily.enabled` 为 `true` 时启用：

```python
# src/tools/web_search.py

import os
from typing import Literal
from tavily import TavilyClient

def create_web_search_tool():
    """创建 Tavily 网络搜索工具"""
    client = TavilyClient(api_key=os.environ["TAVILY_API_KEY"])

    def internet_search(
        query: str,
        max_results: int = 5,
        topic: Literal["general", "news", "finance"] = "general",
    ) -> str:
        """搜索互联网获取最新信息。用于查找技术文档、最佳实践、API参考等。"""
        return client.search(query, max_results=max_results, topic=topic)

    return internet_search
```

Tavily 工具同时配置给 Writer 和 Reviewer 两个子代理，但用途不同：

| Agent | 搜索用途 | 典型搜索示例 |
|-------|----------|-------------|
| **Writer** | **研究型搜索**：主动搜集技术信息来撰写设计方案 | "FastAPI async middleware best practices"、"Redis 7.0 新特性" |
| **Reviewer** | **验证型搜索**：核查文档中特定技术声明的准确性 | "pydantic v2 是否支持 recursive models"、"langchain init_chat_model openrouter" |

两者的差异通过各自的系统提示词约束（详见 3.2 节和 3.3 节），而非工具配置来实现。

## 8. 命令行接口设计

### 8.1 入口与参数

程序入口为 `main.py`，使用 Python 标准库 `argparse` 解析命令行参数：

```
# 方式一：直接传入需求文本
uv run main.py --requirement "开发一个用户认证模块，支持JWT和OAuth2"

# 方式二：指定需求文件（默认在项目根目录下查找）
uv run main.py --file requirements.md

# 方式三：指定需求文件的完整路径
uv run main.py --file ./docs/my_requirement.md

# 方式四：同时指定自定义配置文件
uv run main.py --file requirements.md --config ./config/agents.yaml

# 也支持 python 直接运行
python main.py --requirement "设计一个订单管理系统"
```

### 8.2 参数定义

| 参数 | 短参数 | 必需 | 默认值 | 说明 |
|------|--------|------|--------|------|
| `--requirement` | `-r` | 二选一 | — | 直接传入需求文本字符串 |
| `--file` | `-f` | 二选一 | — | 需求文档的文件名或路径，默认在项目根目录下查找 |
| `--config` | `-c` | 否 | `config/agents.yaml` | 配置文件路径 |
| `--output` | `-o` | 否 | 从配置文件读取 | 输出目录路径 |
| `--max-iterations` | `-m` | 否 | 从配置文件读取 | 覆盖最大迭代轮次 |
| `--log-level` | `-l` | 否 | 从配置文件读取 | 覆盖日志级别 |

### 8.3 入口逻辑

```python
# main.py 伪代码

import argparse
from pathlib import Path
from src.config_loader import load_config
from src.agent_factory import create_orchestrator_agent
from src.logger import setup_logger

def main():
    parser = argparse.ArgumentParser(description="Writer-Reviewer Agent 系统")
    group = parser.add_mutually_exclusive_group(required=True)
    group.add_argument("-r", "--requirement", type=str, help="需求文本")
    group.add_argument("-f", "--file", type=str, help="需求文档文件路径")
    parser.add_argument("-c", "--config", default="config/agents.yaml")
    parser.add_argument("-o", "--output", default=None)
    parser.add_argument("-m", "--max-iterations", type=int, default=None)
    parser.add_argument("-l", "--log-level", default=None)
    args = parser.parse_args()

    # 1. 确保 cwd 为项目根目录（FilesystemBackend(root_dir=".") 依赖此前提）
    import os
    project_root = Path(__file__).resolve().parent
    os.chdir(project_root)

    # 2. 加载配置
    config = load_config(args.config)
    if args.max_iterations:
        config.max_iterations = args.max_iterations
    if args.log_level:
        config.log_level = args.log_level

    # 3. 初始化日志
    logger = setup_logger(config.log_level)

    # 4. 读取需求
    if args.requirement:
        requirement_text = args.requirement
    else:
        file_path = Path(args.file)
        if not file_path.is_absolute() and not file_path.exists():
            file_path = Path.cwd() / args.file  # 默认在项目根目录下查找
        requirement_text = file_path.read_text(encoding="utf-8")

    # 5. 清理上一次运行的残留状态（防止残留的 review-verdict.json 干扰新一轮判定）
    drafts_dir = Path("drafts")
    if drafts_dir.exists():
        import shutil
        shutil.rmtree(drafts_dir)
        logger.info("已清理 drafts/ 目录")
    drafts_dir.mkdir(parents=True, exist_ok=True)

    # 6. 将需求写入固定路径，供 Writer/Reviewer 通过 read_file 读取
    req_dir = Path("requirements")
    req_dir.mkdir(parents=True, exist_ok=True)
    (req_dir / "input.md").write_text(requirement_text, encoding="utf-8")
    logger.info(f"需求已写入 requirements/input.md ({len(requirement_text)} 字符)")

    # 7. 创建并运行 Agent
    agent = create_orchestrator_agent(config)

    result = agent.invoke(
        {
            "messages": [
                {"role": "user", "content": f"请根据需求编写技术设计文档。需求原文已保存在 /requirements/input.md，请从该文件读取。"}
            ],
        },
        config={"configurable": {"thread_id": "session-001"}},
    )

    # 8. 复制最终产物到 output 目录
    # /drafts/design.md 是唯一正式产物，直接从文件系统复制，不从 Agent 响应解析
    drafts_path = Path("drafts/design.md")
    output_dir = Path(args.output or config.output_dir)
    output_dir.mkdir(parents=True, exist_ok=True)

    if drafts_path.exists():
        import shutil
        output_path = output_dir / "design.md"
        shutil.copy2(drafts_path, output_path)
        logger.info(f"最终文档已复制到 {output_path}")
    else:
        logger.error("Agent 运行完成但未找到 drafts/design.md，请检查 Writer 是否正常执行")

    # 9. 输出 Agent 的迭代摘要（经过几轮、最终结论等）
    final_message = result["messages"][-1].content if result.get("messages") else ""
    print(final_message)

if __name__ == "__main__":
    main()
```

## 9. Agent 工厂模块设计

### 9.1 核心函数：create_orchestrator_agent

```python
# src/agent_factory.py 伪代码

from deepagents import create_deep_agent
from deepagents.backends import FilesystemBackend
from langgraph.checkpoint.memory import MemorySaver
from src.model_factory import create_model

def create_orchestrator_agent(config: AppConfig) -> CompiledStateGraph:
    """根据配置创建完整的 Orchestrator Agent"""

    # 1. 通过模型工厂创建各 Agent 的模型实例（屏蔽 Provider 差异）
    orchestrator_model = create_model(
        config.providers[config.agents["orchestrator"].provider],
        config.agents["orchestrator"],
    )
    writer_model = create_model(
        config.providers[config.agents["writer"].provider],
        config.agents["writer"],
    )
    reviewer_model = create_model(
        config.providers[config.agents["reviewer"].provider],
        config.agents["reviewer"],
    )

    # 2. 构建可选工具列表
    tools = []
    if config.tools.get("tavily", {}).get("enabled"):
        from src.tools.web_search import create_web_search_tool
        tools.append(create_web_search_tool())

    # 3. 定义子代理
    # 注意：middleware 不继承自父代理，必须为每个子代理显式挂载
    writer_subagent = {
        "name": "writer",
        "description": "根据业务需求撰写可落地的技术设计文档。需求原文在 /requirements/input.md，草稿保存到 /drafts/design.md。接受反馈后修订文档。",
        "system_prompt": build_writer_prompt(),
        "tools": tools,
        "model": writer_model,
        "skills": ["/skills/tech-doc-writing/"],
        "middleware": [LoggingMiddleware()],
    }

    reviewer_subagent = {
        "name": "reviewer",
        "description": "基于 /requirements/input.md 中的业务需求审核 /drafts/design.md 中的技术设计文档，从需求覆盖性、可落地性、无歧义性、完整性、合理性评估，返回 ACCEPT 或 REVISE 结论及详细反馈。",
        "system_prompt": build_reviewer_prompt(),
        "tools": tools,
        "model": reviewer_model,
        "skills": ["/skills/tech-doc-review/"],
        "middleware": [LoggingMiddleware()],
    }

    # 4. 组装 Orchestrator（使用 FilesystemBackend 直接访问项目文件系统）
    # Orchestrator 也需要自己的 LoggingMiddleware 来捕获 task 委派时的完整对话内容
    agent = create_deep_agent(
        model=orchestrator_model,
        tools=[],
        system_prompt=build_orchestrator_prompt(config.max_iterations),
        subagents=[writer_subagent, reviewer_subagent],
        middleware=[LoggingMiddleware()],
        backend=FilesystemBackend(root_dir=".", virtual_mode=True),
        name="orchestrator",
        checkpointer=MemorySaver(),
    )

    return agent
```

## 11. 错误处理策略

| 错误场景 | 处理方式 |
|----------|----------|
| YAML 配置文件缺失或格式错误 | 启动时校验，抛出明确的错误信息并退出 |
| LLM API 调用失败（网络/限流） | LangChain 内置重试机制（`max_retries`），日志记录每次重试 |
| 工具调用格式异常或被截断 | `PatchToolCallsMiddleware` 自动修复损坏/中断的工具调用 JSON |
| 需求文件不存在 | 启动时检查，提示用户文件路径 |
| 子代理超时 | 通过模型配置的 `timeout` 参数控制，超时后由 SDK 抛出异常 |
| 达到最大迭代次数但文档仍未通过 | Orchestrator 输出当前版本 + 最后一轮审核意见，供用户参考 |
| Reviewer 结论解析失败 | 三级降级：文本 VERDICT 行 → review-verdict.json 文件 → 默认视为 REVISE（详见 3.6 节） |
| 上下文窗口溢出 | SDK 内置 `SummarizationMiddleware` 自动压缩上下文 |

## 12. 关键设计决策记录

### 决策 1：选择方案 A（编排器管理的循环）而非方案 B（LangGraph 自定义工作流）

- **理由**：当前需求是简单的 Writer→Reviewer 线性循环，方案 A 的实现更简洁且完全利用 SDK 原生能力。方案 B 的优势（确定性条件边、显式状态管理）在当前场景中收益不大。
- **风险**：LLM 可能不严格遵循提示词中的"最多N轮"指令，但通过 `recursion_limit` 可兜底。
- **升级路径**：如果后续需要更精细的控制，可将工作流重构为 `CompiledSubAgent`。

### 决策 2：使用 FilesystemBackend 而非 StateBackend

- **理由**：本项目是**本地 CLI 工具**，正好是官方文档推荐 `FilesystemBackend` 的场景（"Local development CLIs, coding assistants, development tools"）。使用 `FilesystemBackend` 的核心优势在于：
  1. **Skills 零样板加载**：Agent 直接从磁盘读取 `./skills/` 目录下的技能文件，无需 `load_skills_files()` 函数、`create_file_data()` 转换和 `invoke(files={...})` 注入
  2. **中间产物可调试**：Writer 生成的草稿直接写入 `./drafts/design.md`，方便人工查看中间稿和调试
  3. **技能热更新**：修改 `SKILL.md` 文件后直接生效，无需重新构建文件字典
- **安全措施**：通过 `virtual_mode=True` 限制 Agent 的文件访问范围在项目根目录内，防止路径穿越
- **权衡**：`StateBackend`（SDK 默认）提供完全的运行时隔离，更适合 Web 服务器或多租户环境。本项目是单用户本地 CLI，`FilesystemBackend` 的简洁性收益大于 `StateBackend` 的隔离性收益

### 决策 3：日志中间件为每个 Agent 独立挂载，并完整记录 Agent 间对话

- **理由**：deepagents SDK 的 `middleware` **不继承**给子代理——Orchestrator 上的中间件只能捕获 Orchestrator 自身的操作，Writer/Reviewer 的内部推理、文件读写、搜索调用全部不可见。需求明确要求"多个 agent/subagent 之间的交流要写入到日志中"。
- **方案**：(1) Orchestrator、Writer、Reviewer 各自独立挂载 `LoggingMiddleware` 实例；(2) `before_tool_call` + `after_tool_call` 中对 `task` 工具特殊处理——执行前记录委派的完整任务内容（需求 + 反馈文本），执行后记录子代理返回的完整结果文本，实现"Agent 间对话"的完整记录。
- **四个日志层面**：模型输出（全部 Agent 的 `wrap_model_call`）、工具调用（全部 Agent 的 `before_tool_call`）、Agent 间对话（Orchestrator 的 `task` 工具入参/返回）、子代理内部链路（Writer/Reviewer 各自的中间件）。
- **`wrap_model_call` 的能力边界**：该钩子在 `handler(request)` 返回后捕获的是模型的可见输出文本，不是内部思维链。纯工具调用回合可能无文本输出。价值定位是"辅助理解决策方向"，不宜过度依赖。
- **性能考量**：模型输出日志使用 DEBUG 级别，仅在需要时开启。Agent 间对话使用 INFO 级别，始终记录。内容分别设置截断阈值（模型输出 800 字符，对话 2000 字符），避免日志膨胀。

### 决策 4：采用模型工厂模式而非统一 `init_chat_model()` 前缀格式

- **理由**：运行环境在中国大陆，优先使用 DashScope/MiniMaxi/BigModel 三个国内提供商。其中 DashScope 使用 `ChatQwen` 原生集成（`langchain-qwq` 包），MiniMaxi 走 Anthropic API 兼容模式，BigModel 走 OpenAI API 兼容模式。三者的初始化方式各不相同，且兼容模式需要自定义 `base_url` 和隔离 API Key，无法通过 `init_chat_model("provider:model")` 统一处理。
- **方案**：引入 `model_factory.py`，通过 YAML 中的 `providers.type` 字段路由到不同的模型创建策略。对 Agent 层完全透明——Agent 代码只接收 `BaseChatModel` 实例，不关心底层是哪个 Provider。
- **API Key 隔离**：MiniMaxi 和 BigModel 分别借用 `ChatAnthropic` 和 `ChatOpenAI` 类，但通过构造函数 `api_key=` 显式传入各自的密钥（`MINIMAX_API_KEY` / `ZHIPUAI_API_KEY`），避免与真正的 `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` 环境变量冲突。
- **DashScope 国内端点**：`langchain-qwq` 默认使用国际端点（`dashscope-intl.aliyuncs.com`），国内用户必须通过 `DASHSCOPE_API_BASE` 环境变量覆盖为国内端点。模型工厂通过 YAML 中的 `base_url_env` 字段读取此配置。
- **关于 `langchain-qwq` 包名**：尽管名称暗示是 QwQ 推理模型专用，但该包实际提供 `ChatQwen`（通用 Qwen 模型）和 `ChatQwQ`（QwQ/QvQ 推理模型）两个类，是 LangChain 官方文档推荐的 DashScope 集成方案。不存在独立的 `langchain-qwen` 包；社区包中的 `ChatTongyi` 功能更新较慢，不推荐。

### 决策 5：Skills 通过 `FilesystemBackend` 直接从磁盘加载

- **理由**：项目使用 `FilesystemBackend`，技能文件是静态的本地文件，Agent 通过 `skills` 参数指向磁盘路径即可自动加载，这是 SDK 官方文档 Skills + FilesystemBackend 示例的标准用法。
- **隔离保证**：每个子代理的 `SkillsMiddleware` 只扫描自己 `skills` 参数指定的目录，技能的发现和加载完全隔离（详见 6.4 节）。
- **对比 StateBackend 方案**：`StateBackend` 需要 `load_skills_files()` → `create_file_data()` → `invoke(files={...})` 三步手动注入，代码更复杂且每次 `invoke` 都要重复。当前方案零样板代码。

### 决策 6：Reviewer 结论采用"提示词约束 + 文件兜底"而非 Structured Output

- **澄清**：deepagents SDK 的子代理**支持** `response_format`（Structured Output），可约束子代理最终响应的格式，中间工具调用不受影响。此处是基于工程权衡的**主动选择**，而非 SDK 限制。
- **选择理由**：(1) 多 Provider 兼容性——DashScope/MiniMaxi/BigModel/OpenRouter 各家对 `response_format` 支持程度不一，API 兼容模式下尤其不确定；(2) Reviewer 反馈内容丰富且非结构化，自由文本更适合承载；(3) 三级降级策略已提供足够的可靠性保障。
- **方案**：采用三级降级策略（详见 3.6 节）：(1) Reviewer 提示词中强约束 `VERDICT: ACCEPT/REVISE` 格式并给出正反例；(2) Reviewer 同时将结论写入 `/drafts/review-verdict.json` 作为文件兜底；(3) Orchestrator 按优先级尝试文本解析 → 文件读取 → 默认 REVISE。
- **未来优化**：如果文本解析失败率过高，应评估为 Reviewer 配置 `response_format` 返回 `{"verdict": "...", "feedback": "..."}` 结构，这是 SDK 原生支持的升级路径。
- **安全哲学**：最终兜底视为 REVISE 而非 ACCEPT，因为"多审一轮"的代价远小于"误判通过导致产出低质量文档"。

## 附录 A：deepagents SDK 关键 API 速查

| API | 说明 |
|-----|------|
| `create_deep_agent(model, tools, system_prompt, subagents, ...)` | SDK 唯一入口，返回 `CompiledStateGraph` |
| `task(description, subagent_type)` | 内置子代理委派工具。`description`: 任务详细描述（含上下文和期望输出格式）；`subagent_type`: 目标子代理名称（如 `"writer"` / `"reviewer"`） |
| `agent.invoke({"messages": [...]})` | 同步调用（`FilesystemBackend` 下无需 `files` 参数） |
| `agent.stream({"messages": [...]})` | 流式调用 |
| `agent.ainvoke(...)` / `agent.astream(...)` | 异步版本 |
| `init_chat_model("provider:model", **params)` | 供应商无关的模型初始化（本项目仅用于 OpenRouter，其他 Provider 由模型工厂直接构造） |
| `CompiledSubAgent(name, description, runnable)` | 将 LangGraph 图包装为子代理 |
| `MemorySaver()` | 内存检查点器，用于持久化 |
| `FilesystemBackend(root_dir, virtual_mode)` | 本地磁盘文件系统后端（本项目采用） |
| `StateBackend` | 默认虚拟文件系统后端（本项目未使用） |
| `create_file_data(content)` | 将字符串转为 StateBackend 虚拟文件格式（本项目未使用） |
